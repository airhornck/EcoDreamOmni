# EcoDreamOmni 专家联合评审报告
## 生产级修复方案：让工作流真正跑起来

> **文档版本**: v1.0  
> **评审日期**: 2026-05-25  
> **评审专家**: 架构专家组 × 业务专家组  
> **对齐真源**: `文档2_最终可行性完整产品方案_素人号矩阵AI平台.md` (v5.0，唯一产品真源)  
> **问题来源**: TaskHub 任务创建后在 task-hub / review-publish-center / trend-scout / publisher 均不可见，工作流无法运转

## 执行摘要

经双专家组联合评审，当前问题**不是前端刷新 Bug，而是系统架构层面的"集成缺失"**。代码中已经构建了 TaskHub API、Workflow Engine、Pipeline Service、Publisher 等独立模块，但**缺少使 TaskHub 成为真正中央编排器的集成层**。各模块处于物理隔离状态，任务创建后无法自动进入工作流执行，导致端到端业务链路完全断裂。

**核心结论**：必须分 4 个 Phase 进行生产级修复，优先级为 **P0 数据持久化 → P1 引擎贯通 → P2 链路打通 → P3 异步与观测**。

### 1.3 任务状态机真源

```
DRAFT → CONFIGURING → QUEUED → RUNNING → [PAUSED / HUMAN_WAIT]
                                           ↓
                         ┌─────────────────┼─────────────────┐
                         ▼                 ▼                 ▼
                    COMPLETED          FAILED           CANCELLED
                         ▲                 │
                         └──── RETRY ──────┘
```

**状态转换规则（真源）**：

| 转换 | 触发条件 | 自动/手动 |
|------|---------|----------|
| DRAFT → CONFIGURING | 任务创建，变量绑定完成 | 自动 |
| CONFIGURING → QUEUED | 参数校验通过 | 自动 |
| QUEUED → RUNNING | Workflow Engine 认领任务 | 自动 |
| RUNNING → HUMAN_WAIT | 执行到 `human_approval` 节点 | 自动 |
| RUNNING → COMPLETED | 全部节点执行成功 | 自动 |
| RUNNING → FAILED | Agent 错误 / FAIL_FAST | 自动 |
| HUMAN_WAIT → RUNNING | 运营点击【通过】| 手动 |
| HUMAN_WAIT → FAILED | 运营点击【驳回】| 手动 |
| HUMAN_WAIT → CONFIGURING | 运营点击【打回修改】| 手动 |
| ANY → CANCELLED | 运营点击【取消】| 手动 |

> **注意**：`APPROVED_WAITING_PUBLISH` 不是真源状态机中的正式状态，当前代码中此状态是 pragmatic 添加，但造成了状态机污染。建议在修复中将其归并为 `RUNNING` 的一个子阶段（通过 `current_node_index` 或扩展状态元数据表达），而非独立状态。

### 1.4 各模块数据契约（真源）

| 模块 | 真源数据所有者 | 输出给下游 | 从上游接收 |
|------|--------------|-----------|-----------|
| **TrendScout** | TrendReport / Topic 表 | `adopted_topic` | — |

## 第二部分：架构专家诊断报告

### 2.1 技术债务总览

当前系统存在 **6 大架构断层**，按严重程度排序：

```
┌─────────────────────────────────────────────────────────────┐
│ SEV-1  数据无持久化        → 任务随进程重启消失              │
│ SEV-1  工作流引擎悬空      → 任务创建后永不执行              │
│ SEV-2  多系统任务模型割裂   → 5套独立的内存任务存储           │
│ SEV-2  异步执行缺失        → 无Celery，全靠BackgroundTasks   │
│ SEV-3  人工审核断环        → HUMAN_WAIT后无法恢复工作流       │
│ SEV-3  全链路观测缺失      → trace_id仅为占位，无OTel集成    │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 六大架构断层详解

#### 断层 1：数据无持久化（SEV-1）

**现状**：
```python
# apps/backend/src/services/task_hub.py
_task_db: Dict[str, Task] = {}
_node_exec_db: List[TaskNodeExecution] = []
```

TaskHub、WorkflowEngine、PipelineService、Publisher、HumanInLoop 全部使用内存字典/列表存储。

**后果**：
- 后端服务重启 = 全部任务数据清零
- 多 worker 部署时数据不共享
- 无法支持滚动更新、水平扩展

#### 断层 2：工作流引擎悬空（SEV-1）

**现状**：
- `workflow_engine.start_execution(task_id, template_id)` 已定义但**从未被调用**
- TaskHub 创建任务后状态永远为 `DRAFT`
- WorkflowEngine 拥有独立的 `_execution_db`，与 TaskHub `_task_db` 无任何关联

**后果**：
- 任务永远不会被工作流引擎认领
- 节点永远不会执行
- `current_node_index` 永远是 0
- 永远无法到达 `HUMAN_WAIT` 节点

#### 断层 3：多系统任务模型割裂（SEV-2）

**现状**：系统中存在 5 套独立的"任务"概念：

| 系统 | 存储变量 | 任务模型 | 与 TaskHub 关系 |
|------|---------|---------|---------------|
| TaskHub | `_task_db` | `Task` dataclass | 自称中心，但无集成 |
| PipelineService | `_task_db` | `PipelineTask` | 完全独立 |
| WorkflowEngine | `_execution_db` | `WorkflowExecution` | 接受 task_id 参数但无人调用 |
| Publisher | `_task_db` | `PublishTask` | 经 review-publish-center 确认后才创建 |
| Orchestrator | 无显式存储 | `GroupShard` | 不读写 TaskHub |

**后果**：
- 同一个业务任务在系统中分裂为多个互不相通的记录
- 运营在 TaskHub 看到的任务状态与 Publisher 看到的发布状态完全脱节
- 无法做全链路追踪（"这条内容现在到哪了？"）

#### 断层 4：异步执行缺失（SEV-2）

**现状**：
```python
# pipeline_service.py 中的注释
"MVP: Uses FastAPI BackgroundTasks for zero-dependency async execution.
 Production: Migrate to Celery + Redis"
```

**后果**：
- 异步任务随进程死亡而丢失
- 无重试机制（除简易的 DLQ 内存列表外）
- 无法支持定时发布、批量校准等耗时操作
- 无法水平扩展 worker

#### 断层 5：人工审核断环（SEV-3）

**现状**：
- `task_hub.submit_human_decision()` 可以修改 `review_decision` 和 `status`
- 但**没有代码**将决策结果反馈给 WorkflowEngine 以恢复节点执行
- `ReviewPublishCenterPage` 的后端 API (`review_publish.py`) 是并行路径，不经过 TaskHub 状态机

**后果**：
- 运营在审核中心点击【通过】后，任务状态变成 `APPROVED_WAITING_PUBLISH`
- 但工作流引擎不知道要恢复执行
- 发布节点永远不会被触发

#### 断层 6：全链路观测缺失（SEV-3）

**现状**：
- 数据模型中有 `trace_id` 字段
- 但无 OpenTelemetry SDK 集成
- 无跨模块的 span 传播

**后果**：
- 无法诊断"任务卡在哪一步"
- 无法做性能 profiling
- 生产环境故障排查困难

### 2.3 前端字段契约断裂

```
Frontend (TaskHubCreatePage)          Backend (CreateTaskRequest)
─────────────────────────────────────────────────────────────────
persona_story_id    ──────────────→   ❌ 不存在（静默丢弃）
node_id             ──────────────→   ❌ 不存在（静默丢弃）
content_series_id   ──────────────→   ❌ 不存在（静默丢弃）
new_series_name     ──────────────→   ❌ 不存在（静默丢弃）
variables           ──────────────→   ❌ 不存在（静默丢弃，前端发 variables，后端要 prompt_variables）
```

**注意**：前端 inline drawer (`TaskHubPage.tsx`) 发送 `status: 'queued'`，但后端强制覆盖为 `DRAFT`；且 inline drawer **缺少必填字段 `persona_id`**，会导致 422 错误。

## 第三部分：生产级修复方案

### 3.1 修复原则

1. **真源唯一**：`tasks` 表是任务全生命周期的唯一真源，其他系统的任务记录必须关联 `task_id`
2. **状态机自治**：TaskHub 拥有状态机，WorkflowEngine 拥有执行计划，两者通过标准接口协作，不强耦合
3. **渐进演进**：不推倒重来，在现有模块基础上补全集成层
4. **向后兼容**：修复期间现有 API 保持兼容，通过新增接口和数据库字段实现平滑过渡
5. **可观测优先**：每次修复都要补充对应的日志、指标或 trace

### 3.2 Phase 1：数据持久化与模型统一（Week 1）

**目标**：消灭内存存储，让任务数据可重启、可共享、可查询。

### 3.3 Phase 2：工作流引擎与 TaskHub 贯通（Week 2）

#### 3.3.1 工作流启动契约

```python
def create_task(..., auto_start: bool = False) -> Task:
    # ... 创建任务记录 ...
    if auto_start:

        transition_task(task.id, "CONFIGURING")
        transition_task(task.id, "QUEUED")
```

#### 3.4.3 HumanInLoop 决策后恢复工作流执行

修改 `task_hub.submit_human_decision()`：

```python
def submit_human_decision(task_id: str, decision: str, ...):
    task = get_task(task_id)
    task.review_decision = decision
    task.reviewed_at = datetime.utcnow()
    
    if decision == "APPROVE":
        task.status = "APPROVED_WAITING_PUBLISH"   # 或 RUNNING (发布子阶段)
        # 关键：通知 WorkflowEngine 恢复执行
        if task.execution_id:
            workflow_engine.resume_execution(task.execution_id)
    elif decision == "REJECT":
        task.status = "FAILED"
        if task.execution_id:
            workflow_engine.fail_execution(task.execution_id, reason="Human rejected")
    elif decision == "REVISE":
        task.status = "CONFIGURING"
        # 打回修改：WorkflowEngine 回滚到 ContentForge 节点
        if task.execution_id:
            workflow_engine.rollback_to_node(task.execution_id, node_type="content_generation")
    
    db.commit()
```

```typescript

  task_hub_task_id?: string        // 新增
```

### 3.5 Phase 4：异步执行与可观测性（Week 4）

#### 3.5.1 Celery 集成

引入 Celery + Redis 作为正式异步执行层：

```python
# celery_app.py
celery_app = Celery("ecodream", broker="redis://redis:6379/0", backend="redis://redis:6379/0")

# tasks.py
@celery_app.task(bind=True, max_retries=3)
def execute_workflow_node(self, execution_id: str, node_index: int):
    try:
        workflow_engine.execute_next_node(execution_id)
    except Exception as exc:
        raise self.retry(exc=exc, countdown=60)

@celery_app.task
def async_calibration_batch(report_id: str):
    data_analyst.run_calibration(report_id)

@celery_app.task
def scheduled_publish(publish_task_id: str):
    publisher.execute_publish(publish_task_id)
```

替换 `pipeline_service.py` 中的 `BackgroundTasks`：

```python
# 旧
background_tasks.add_task(_async_run_pipeline, task_id)

# 新
from src.celery_app import execute_workflow_node
execute_workflow_node.delay(execution_id, node_index)
```

#### 3.5.2 CronHub 定时发布

使用 Celery Beat 作为定时调度器：

```python
# 当任务设置 scheduled_at 时
celery_app.send_task(
    "scheduled_publish",
    args=[publish_task_id],
    eta=task.scheduled_at,
)
```

#### 3.5.3 OpenTelemetry 全链路追踪

引入 `opentelemetry-api` + `opentelemetry-sdk`：

```python
from opentelemetry import trace

tracer = trace.get_tracer("ecodream.workflow")

def execute_next_node(execution_id: str):
    execution = get_execution(execution_id)
    with tracer.start_as_current_span("workflow.execute_node") as span:
        span.set_attribute("task.id", execution.task_id)
        span.set_attribute("execution.id", execution_id)
        span.set_attribute("node.index", execution.current_node_index)
        # ... 执行节点 ...
```

确保 `trace_id` 在以下边界传播：
- Frontend → Backend (HTTP header: `X-Trace-Id`)
- FastAPI → Celery Worker (message header)
- TaskHub → WorkflowEngine → Agent (context 传递)

## 第五部分：风险与缓解

| 风险 | 严重度 | 缓解措施 |
|------|--------|---------|
| **数据迁移失败** | 高 | Alembic migration 必须可回滚；生产数据迁移前在 staging 全量验证；保留内存存储作为 fallback 一周 |
| **状态机变更导致现有任务异常** | 高 | 现有内存中的任务在升级前全部归档或清空；状态机变更通过新增字段实现，不改历史状态定义 |
| **WorkflowEngine 与 TaskHub 循环依赖** | 中 | 通过事件总线或接口抽象层解耦；禁止直接 import，改用 `src/events.py` 中的事件发布/订阅 |
| **Celery 引入增加部署复杂度** | 中 | Docker Compose 中已预留 Redis；Celery Worker 以独立 service 启动；提供 `docker-compose.dev.yml` 本地一键启动 |
| **前端字段变更导致兼容性破坏** | 低 | 后端新增字段均为 Optional，旧前端不传不报错；Response 富化字段通过 JOIN 查询，不存冗余 |
| **性能退化（DB 查询 vs 内存）** | 低 | 保留 `_task_db` 作为 L1 缓存，写穿策略；列表查询加分页（limit/offset）；热点数据加 Redis 缓存 |
