## 一、行业真源与最佳实践研究

### 1.1 学术真源：Agentic Personalisation in Marketing (2026)

> "The system is designed to scale through a **modular content pipeline**. Rather than writing thousands of individual message variants, 'human-in-the-loop' marketers provide the system with **atomic components**—such as distinct tones of voice, offer types, greetings, et cetera. The agentic infrastructure then **autonomously permutes these atoms** to generate and serve personalised variants."
> — *Sustained Impact of Agentic Personalisation in Marketing*, arXiv 2604.08621v1

**启示**：内容生产的正确范式是「原子组件 → 自动排列组合」，而非「完整模板 → 变量填充」。

### 1.2 行业趋势：The Death of the Manifesto (2026)

> "We are moving away from monolithic 'mega-prompts' towards **modular component architectures**... In 2026, professional engineers build libraries of 'Prompt Fragments'—small, reusable segments that can be assembled on the fly. This 'Composable AI' approach allows for **thousands of permutations without rewriting a single word**."
> — *AI Prompt Engineering Trends 2026: The Shift to Management*, FlashPrompt

**启示**：Prompt 工程已从「单篇巨著」进化为「乐高积木」。内容策略配置同理。

### 1.3 交互范式：PromptCanvas & Dynamic Widgets (2025)

> "PromptCanvas **fragments prompts into dynamic widgets**—first-class interface components (e.g., 'Tone,' 'Setting,' 'Plot Twist'), which users can **create, edit, arrange, or compose on an infinite workspace**. Each widget value is reflected live in LLM calls, allowing rapid, low-cognitive-load iteration."
> — *Dynamic Prompting: Adaptive Strategies*, Amin et al., 2025

**启示**：用户界面应将策略元素呈现为可拖拽、可编辑、可组合的「动态组件」，而非静态表单或单选模板。

### 1.4 工程实践：Composable Prompt Architecture (Unreal Engine AI Toolkit 2026)

> "System prompts are built from a hierarchy of **PromptElement objects**, each carrying a numeric priority and a get_token_count() method... The PromptManager assembles the final prompt by: (i) collecting all registered elements, (ii) sorting by descending priority, (iii) greedily selecting elements until the token budget is exhausted."
> — *Unreal Engine AI Toolkit*, arXiv 2604.25318v1

**启示**：Prompt 组装应是「元素注册 → 优先级排序 → Token 预算内贪心选择 → 渲染输出」的工业化流程。

### 1.5 运营真源：小红书内容策略公式

> "好的种草 = **好的内容 × 好的场景 × 好的趋势**"
> — *小红书商品笔记爆款打造指南*, 小红书电商学习中心

> "创作公式：**锚定场景 → 挖掘痛点 → 产品卖点 → 购买引导**"
> — 同上

> "AI 辅助创作：**场景词 → 人群词 → 标题 → 内容框架**"
> — 同上

**启示**：内容策略的维度是乘法关系（×），不是加法关系（+）。每个维度都是独立的策略元素，可自由替换和组合。

## 二、方案 C：策略元素化架构（Strategy Element Architecture）

### 2.1 核心范式

```
┌─────────────────────────────────────────────────────────────────────┐
│           Content Strategy = Composition of Strategy Elements       │
│                                                                     │
│   内容策略 ≠ 模板（完整不可分割）                                    │
│   内容策略 ≠ 字段表单（独立无关联）                                  │
│   内容策略 = 策略元素的有序组合 + 变量映射 + 自定义片段               │
│                                                                     │
│   ┌─────────────┐     ┌─────────────┐     ┌─────────────┐          │
│   │ 策略元素库   │  →  │  策略组合    │  →  │  Prompt组装  │          │
│   │ Element Pool │     │  Strategy    │     │  Composition │          │
│   └─────────────┘     └─────────────┘     └─────────────┘          │
│                                                                     │
│   爆款分析提取元素 →  用户组合元素   →   LLM Prompt生成              │
│   人工创建元素   →   保存为可复用策略  →   六层Prompt映射            │
│   系统预设元素   →   变量自动推导      →   Token预算优化              │
└─────────────────────────────────────────────────────────────────────┘
```

### 2.4 Prompt 组装引擎（PromptCompositionEngine）

```python
class PromptCompositionEngine:
    """Prompt 组装引擎 —— 将 ContentStrategy 渲染为完整 Prompt.
    
    设计灵感：Unreal Engine AI Toolkit 的 Composable Prompt Architecture
    + Claude Code 的六层 Context Injection
    """
    
    def __init__(self):
        self.renderers: Dict[str, ElementRenderer] = {}
        self._register_default_renderers()
    
    def compose(
        self,
        content_strategy: ContentStrategy,
        topic: str,
        platform: str,
        token_budget: int = 4000,
    ) -> ComposedPrompt:
        """将内容策略组合渲染为完整 Prompt（含六层分解）."""
        
        # Step 1: 收集所有 Prompt 元素（含优先级和 Token 估算）
        prompt_elements: List[PromptElement] = []
        
        for ref in content_strategy.elements:
            element = self.load_element(ref.element_id)
            renderer = self.get_renderer(element.element_type)
            
            # 渲染为 Prompt 片段
            fragment = renderer.render(
                element=element,
                variables={**content_strategy.variables, **(ref.override_variables or {})},
                topic=topic,
                platform=platform,
            )
            
            # 包装为 PromptElement（含优先级和 Token 数）
            prompt_elements.append(PromptElement(
                fragment=fragment,
                priority=ref.priority,
                token_count=estimate_tokens(fragment),
                target_layer=renderer.target_layer,  # 六层中的目标层
            ))
        
        # Step 2: 按优先级排序，Token 预算内贪心选择
        prompt_elements.sort(key=lambda e: -e.priority)
        
        selected_fragments = []
        total_tokens = 0
        for pe in prompt_elements:
            if total_tokens + pe.token_count <= token_budget:
                selected_fragments.append(pe)
                total_tokens += pe.token_count
            else:
                # Token 不足时，记录被裁剪的元素（用于调试和优化）
                logger.warning("Token budget exhausted, skipping element: %s", pe.fragment[:50])
                break
        
        # Step 3: 按六层 Prompt 组织
        six_layers = {
            "platform_format": self._render_layer1(platform),
            "structure_template": self._merge_layer2(selected_fragments),
            "brand_knowledge": self._render_layer3(content_strategy),
            "keyword_injection": self._merge_layer4(selected_fragments),
            "persona_layer": self._render_layer5(content_strategy),
            "style_layer": self._merge_layer6(selected_fragments),
        }
        
        # Step 4: 追加自定义片段
        for custom in content_strategy.custom_fragments:
            six_layers["style_layer"] += f"\n\n[自定义约束]\n{custom}"
        
        return ComposedPrompt(
            six_layers=six_layers,
            full_prompt=self._assemble_full_prompt(six_layers),
            used_elements=[e.element_id for e in selected_fragments],
            pruned_elements=[e.element_id for e in prompt_elements if e not in selected_fragments],
            total_tokens=total_tokens,
        )
```

class ViralAnalyzerService:

## 三、四维度专家评审

### 3.1 架构师评审（权重 30%）

**关注维度**: 数据一致性、扩展性、技术债务、范式先进性

| 维度 | 评分 | 理由 |
|------|------|------|
| 数据一致性 | ⭐⭐⭐⭐ | `StrategyElement` 自描述（`content` + `render_template` + `variables`）， Schema 由 `element_type` 定义。虽然不同类型有不同 Schema，但渲染流程统一（Jinja2），消费端无需感知具体类型。 |
| 扩展性 | ⭐⭐⭐⭐⭐ | 新增策略维度只需：① 定义 `element_type` ② 实现 `ElementRenderer` ③ 前端注册卡片组件。无需改数据库 Schema、无需改表单结构、无需改 Pipeline 核心逻辑。 |
| 技术债务 | ⭐⭐⭐⭐ | 需新建 `strategy_elements` 表和 `strategy_sets` 表，但现有 `content_templates` 可迁移为 `strategy_sets`。`ViralAnalyzer` 改造是增量扩展而非推翻重写。 |
| 范式先进性 | ⭐⭐⭐⭐⭐ | 直接对齐 2026 年行业趋势：模块化 Prompt 架构（LEGO Blocks）、原子化内容组件（Agentic Marketing）、可组合策略（PromptCanvas）。这是「正确」的架构方向。 |

**小计**: (4 + 5 + 4 + 5) / 4 = **4.5 / 5.0**

**架构师结论**：
> 方案 C 是当前技术条件下能实现的**最优架构**。它既不落入「模板驱动」的僵化陷阱，也不陷入「字段聚合」的扩展瓶颈。`StrategyElement` 的原子化设计 + `PromptCompositionEngine` 的工业级组装流程，使系统具备面向未来的弹性。
>
> **唯一风险**：`render_template` 使用 Jinja2 引入模板引擎依赖，需评估在 FastAPI 后端引入 Jinja2 的成本（实际上 FastAPI 已依赖 Jinja2 用于响应模板，成本为零）。

## 六、风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| Alembic 多 head 冲突 | 中 | 高 | 本次迁移前必须先合并双 head；新建表迁移与现有表无关，风险隔离 |
| PromptCompositionEngine 渲染性能 | 低 | 中 | Jinja2 渲染是微秒级操作；Token 估算使用缓存（相同文本只算一次） |
| 前端拖拽排序组件引入 bundle 体积 | 低 | 低 | `@dnd-kit/core` + `@dnd-kit/sortable` 约 15KB gzip，可接受 |
| 策略元素类型过多导致 UI 混乱 | 中 | 中 | 前端按「常用/推荐」分组展示，默认只显示 5-6 个核心类型，高级用户展开全部 |
| 现有 ContentTemplate 用户数据迁移不完整 | 低 | 高 | 迁移脚本需完整测试：解析 `prompt_template` → 提取 structure_framework + keyword_strategy 元素；迁移失败时保留原表作为 fallback |
| Token 预算控制不准确 | 中 | 中 | 使用 `tiktoken` 做中文优化估算；设置 10% 安全余量；超预算时优先裁剪低优先级元素而非报错 |

## 七、与实验室（Playground）的协同设计

### 7.1 爆款笔记分析 → 策略元素提取流程

```
用户输入爆款笔记（Playground）
  → ViralAnalyzerService.analyze_note() → AnalysisReport
    → 用户点击「提取策略元素」
      → ViralAnalyzerService.extract_strategy_elements(report)
        → 生成 6-7 个 StrategyElement
          → 用户选择「保存到元素库」或「直接应用到当前任务」
            → 若保存：入库 strategy_elements，未来可在 Step 2 浏览选择
            → 若应用：直接填充到 Step 2 的已选元素列表
              → 用户可继续增删调整元素
                → 满意后「保存为策略组合」→ 入库 strategy_sets
```

### 7.2 策略元素的持续优化闭环

```
策略元素被使用 → ContentForge 生成内容 → 发布后收集互动数据
  → engagement_collect Function 采集数据
    → 计算 effectiveness_score（基于 likes/comments/saves/shares）
      → 更新 strategy_elements.effectiveness_score
        → 推荐算法优先推荐高 effectiveness_score 的元素
          → 形成「提取 → 使用 → 反馈 → 优化」闭环
```
