### 1.2 候选方案定义

基于用户的额外思考（第 8 项），识别出两种互斥的架构范式：

---

#### 方案 A：模板驱动型「主题与策略」（All-in-Template）

**核心范式**：内容模板是 Step 2 的唯一核心，所有配置项（Persona、Story、Series、Timeline、Keywords、Methodology）都作为模板的「特殊变量类型」，用户选择模板后统一填写变量表单。

```
Step 2: 主题与策略
├── ① 内容模板选择（核心入口）
│   ├── 智能推荐：基于 platform + topic + methodology_stage 推荐 Top3
│   ├── 手动浏览：模板库卡片（支持 platform / structure_type / source 过滤）
│   └── 选择后展开「变量填写表单」
│
└── ② 模板变量动态表单（根据模板.variables定义渲染）
    ├── 结构变量（type=text）：{痛点词}, {数字}, {对比物} ...
    ├── 人设变量（type=reference_persona）：下拉选择 Persona
    ├── 故事变量（type=reference_story）：级联选择 PersonaStory → Node
    ├── 系列变量（type=reference_series）：下拉选择 ContentSeries
    ├── 时间变量（type=reference_timeline）：下拉选择 TimelineEvent
    ├── 关键词变量（type=multi_select_keywords）：多选 KeywordLibrary + 自由输入
    └── 方法论变量（type=enum_methodology）：单选 5A 阶段
```

**数据流向**：
```
ContentTemplate.variables → 前端动态表单渲染 → 用户填写值 →
  ├─ 引用类型变量值 → TaskORM 对应字段（persona_id, story_id, series_id...）
  └─ 文本类型变量值 → task.prompt_variables[variable_name]
```

**必需的基础设施变更**：
1. `ContentTemplateORM.variables` 扩展变量类型系统：新增 `reference_*`、`enum_*`、`multi_select_*` 类型
2. `MarketingMethodology` ORM 化（当前纯内存 dataclass，重启丢失）
3. `ContentSeries` ORM 化（当前纯内存 dataclass，重启丢失）
4. `TaskORM` 新增 `content_template_id`、`methodology_stage_id`、`timeline_event_id`
5. Pipeline Step 2/3/5 全面改造：从「自动决策」变为「感知向导选择」

---

#### 方案 B：字段聚合型「主题与策略」（Multi-Field Configuration）

**核心范式**：Step 2 是「多字段配置面板」，内容模板只是其中一个字段。模板有自己的结构变量（如 {痛点词}），Persona、Story、Series 等是独立字段，彼此独立选择。

```
Step 2: 主题与策略
├── ① 内容模板选择（字段之一）
│   ├── 智能推荐：基于 platform + topic + methodology 推荐 Top3
│   ├── 手动浏览
│   └── 选择后：填写模板结构变量（{痛点词} 等，type=text 即可）
├── ② Persona 选择（独立字段）
├── ③ PersonaStory + Node 级联选择（独立字段）
├── ④ ContentSeries 选择（独立字段，从 Step 3 移入）
├── ⑤ TimelineEvent 选择（独立字段）
├── ⑥ 关键词策略（独立字段：KeywordLibrary 多选 + 自由输入 TagInput）
└── ⑦ 方法论 5A 阶段选择（独立字段：下拉枚举）
```

**数据流向**：
```
各字段独立选择 → 直接映射到 TaskORM 字段：
  content_template_id, persona_id, persona_story_id, node_id,
  content_series_id, timeline_event_id, methodology_stage_id,
  prompt_variables.keywords, prompt_variables.template_vars
```

**必需的基础设施变更**：
1. `TaskORM` 新增 `content_template_id`、`methodology_stage_id`、`timeline_event_id`
2. `prompt_variables` 中新增 `keywords`、`template_vars` 存储结构变量值和关键词
3. `ContentTemplateORM` 无需变更（variables 保持现有 text 类型）
4. `MarketingMethodology` 无需 ORM 化（通过 API 暴露内存中数据供下拉选择）
5. `ContentSeries` 无需 ORM 化（但长期建议迁移）
6. Pipeline Step 2/3/5 需要感知向导选择，但改造范围小于方案 A

## 二、四维度专家评审

### 2.1 架构师评审（权重 30%）

**评审专家**: 系统架构师  
**关注维度**: 数据一致性、扩展性、技术债务、范式合理性

#### 对方案 A（模板驱动）的评审

| 维度 | 评分 | 理由 |
|------|------|------|
| 数据一致性 | ⭐⭐ | `ContentTemplate.variables` 类型系统从纯 text 扩展为引用/枚举/多选，需要在前端表单引擎、后端 Schema 验证、Pipeline 消费三层保持一致。任何新增变量类型都需同时改三处，一致性风险极高。 |
| 扩展性 | ⭐⭐⭐⭐⭐ | 未来新增注入项（如品牌知识、合规标签）只需定义新变量类型，无需改表单布局，扩展性极佳。 |
| 技术债务 | ⭐⭐ | 需要 ORM 化 MarketingMethodology 和 ContentSeries（当前内存中），且要废弃现有的独立字段选择模式。已有 `persona_id` 等字段在 TaskORM 中，若改为模板变量驱动，这些字段的存在意义被削弱，造成「双轨制」债务。 |
| 范式合理性 | ⭐⭐⭐ | 「一切配置皆模板变量」是优雅的抽象，但过度抽象会导致简单配置（如选择 Persona）必须经过模板间接访问，增加认知负担。且模板并非总是先行确定——用户可能先选人设再选模板。 |

**小计**: (2 + 5 + 2 + 3) / 4 = **3.0 / 5.0**

#### 对方案 B（字段聚合）的评审

| 维度 | 评分 | 理由 |
|------|------|------|
| 数据一致性 | ⭐⭐⭐⭐⭐ | 每个配置项独立映射到 TaskORM 字段，关系清晰，Schema 验证简单，Pipeline 消费直接。 |
| 扩展性 | ⭐⭐⭐ | 新增配置项需修改 Step 2 组件、TaskORM、CreateTaskRequest、Pipeline 消费，四处联动。但每处修改都是线性的，无类型系统扩展负担。 |
| 技术债务 | ⭐⭐⭐⭐ | 复用现有 TaskORM 字段（persona_id 等），只需新增 3 个字段。MarketingMethodology 和 ContentSeries 保持内存中（短期不 ORM 化），通过 API 暴露即可，债务可控。 |
| 范式合理性 | ⭐⭐⭐⭐ | 符合用户心智模型：「先选模板，再补其他配置」。模板变量只负责「结构填充」（{痛点词}），独立字段负责「策略选择」（Persona/Methodology）。两种抽象各司其职。 |

**小计**: (5 + 3 + 4 + 4) / 4 = **4.0 / 5.0**

#### 架构师结论

> **推荐方案 B（字段聚合型）作为实施路径**。
>
> 方案 A 的「模板驱动」在长期是正确方向，但当前基础设施（Methodology 和 Series 无 ORM、variables 类型系统简陋）不足以支撑。强行实施会产生严重的「抽象泄漏」：用户选择 Persona 时，表面上是填模板变量，实际上系统必须将其同步到 `task.persona_id` 以兼容 Pipeline 和现有审核/发布逻辑。这种「变量即字段」的双轨映射会在 6 个月内成为主要技术债务。
>
> **建议演进路线**：Phase 1 实施方案 B；Phase 2（v4.2+）当 Methodology 和 Series ORM 化、variables 类型系统成熟后，再考虑向模板驱动范式迁移。

### 2.2 前端工程师评审（权重 20%）

**评审专家**: 前端技术负责人  
**关注维度**: 组件复杂度、状态管理、用户体验、TypeScript 类型安全

#### 对方案 A（模板驱动）的评审

| 维度 | 评分 | 理由 |
|------|------|------|
| 组件复杂度 | ⭐⭐ | 需要实现「动态表单渲染引擎」：根据后端返回的 variables 数组，按 type 渲染不同组件（text input / persona select / story cascade / keyword multi-select / methodology enum）。相当于在 Step 2 内嵌一个微型 Form Builder，复杂度极高。 |
| 状态管理 | ⭐⭐⭐ | 所有配置收敛为 `templateVariables: Record<string, unknown>`，状态扁平。但需要维护变量值到独立字段的「副作用同步」（如 persona 变量改变 → 同步 formData.personaId），增加状态机复杂度。 |

| TS 类型安全 | ⭐⭐ | `variables` 是 `List[Dict]`，运行时才能确定字段类型。前端需要大量 `as` 断言或运行时类型守卫，TypeScript 静态检查能力被严重削弱。 |

**小计**: (2 + 3 + 4 + 2) / 4 = **2.75 / 5.0**

#### 对方案 B（字段聚合）的评审

| 维度 | 评分 | 理由 |
|------|------|------|
| 组件复杂度 | ⭐⭐⭐⭐ | Step 2 变为「配置面板」，每个字段是独立组件（已大部分存在：PersonaSelect、StoryCascade 等）。只需新增 TemplateSelect（带变量表单）、KeywordTagInput、MethodologySelect、TimelineSelect。组件职责清晰。 |
| 状态管理 | ⭐⭐⭐⭐⭐ | `FormData` 已有 16 个字段，新增 `contentTemplateId`、`methodologyStageId`、`timelineEventId`、`keywords`、`templateVars`，共 21 个字段。Zustand 扁平状态，无嵌套，完全类型安全。 |
| 用户体验 | ⭐⭐⭐⭐ | 字段独立，用户可按任意顺序配置（先选人设或先选模板均可）。模板选择后展开变量表单，其他字段保持可见。符合渐进式披露原则。 |

#### 前端工程师结论

> **强烈推荐方案 B**。

>
> **用户体验关键点**：模板推荐卡片应在 Step 2 顶部展示（与平台、方法论联动），用户选择模板后，下方动态展开「模板变量」折叠面板，不遮挡其他字段。

### 2.3 后端工程师评审（权重 30%）

**评审专家**: 后端技术负责人  
**关注维度**: API 设计、数据库迁移、Pipeline 改造、数据完整性

#### 对方案 A（模板驱动）的评审

| 维度 | 评分 | 理由 |
|------|------|------|
| API 设计 | ⭐⭐ | CreateTaskRequest 需要接收 `content_template_id` + `template_variables: Dict[str, Any]`。后端需根据模板定义解析变量类型，将引用类型映射到 TaskORM 字段。API 契约从「显式字段」变为「隐式解析」，可维护性下降。 |
| 数据库迁移 | ⭐⭐⭐ | TaskORM 新增 `content_template_id` 即可，但 MarketingMethodology 和 ContentSeries ORM 化需要新建表 + 数据迁移（将内存中硬编码数据 seed 到 DB）。Alembic 当前已有双 head，迁移风险叠加。 |
| Pipeline 改造 | ⭐⭐ | Pipeline Step 2（MarketingMethodology）当前自动推荐 5A 阶段和结构模板。若向导已选 template 和 methodology，Step 2 需要跳过或校验。Step 5（keyword_inject）同理。改造涉及事件总线、Blackboard 状态传递、fail_strategy 调整，范围大。 |
| 数据完整性 | ⭐⭐ | 模板变量值与 TaskORM 字段的「双写」风险：如果用户通过 API 直接更新 task（绕过向导），只改了 `persona_id` 但没同步模板变量中的 persona 值，导致 Pipeline 消费不一致。 |

**小计**: (2 + 3 + 2 + 2) / 4 = **2.25 / 5.0**

#### 对方案 B（字段聚合）的评审

| 维度 | 评分 | 理由 |
|------|------|------|
| API 设计 | ⭐⭐⭐⭐⭐ | CreateTaskRequest 新增显式可选字段：`content_template_id`、`methodology_stage_id`、`timeline_event_id`、`keywords`、`template_vars`。契约清晰，向后兼容，旧客户端不传新字段即可。 |
| 数据库迁移 | ⭐⭐⭐⭐ | TaskORM 新增 3 个字段（`content_template_id` String、`methodology_stage_id` String、`timeline_event_id` UUID）。Alembic 单表加字段，零风险。Methodology 和 Series 保持内存中，无需迁移。 |
| Pipeline 改造 | ⭐⭐⭐ | Step 2/3/5 需感知向导选择，但改造范围明确：从 Blackboard 读取 `task.methodology_stage_id` / `task.content_template_id` / `task.prompt_variables.keywords`，若存在则使用向导值，否则保持自动推荐逻辑。回退策略清晰。 |
| 数据完整性 | ⭐⭐⭐⭐⭐ | 所有配置直接写入 TaskORM 字段，无间接解析，无二义性。Pipeline、审核、发布各阶段直接读取对应字段。 |

**小计**: (5 + 4 + 3 + 5) / 4 = **4.25 / 5.0**

#### 后端工程师结论

> **推荐方案 B**。
>
> 方案 A 的「变量解析映射」机制在当前后端架构下会引入严重的隐性耦合。每次新增变量类型，需要同时修改：模板 Schema、API 校验、Task 创建逻辑、Pipeline 消费逻辑、审核逻辑。方案 B 的显式字段虽然「不够优雅」，但在工程实践中是最安全的选择。
>
> **关键实施点**：
> 1. `keywords` 建议放入 `prompt_variables.keywords`（数组），与现有 `prompt_variables` 机制兼容，无需新增字段。
> 2. `template_vars`（模板结构变量值）也放入 `prompt_variables.template_vars`（字典）。
> 3. Pipeline Step 5（keyword_inject）的改造策略：**叠加模式**——向导选择的关键词作为「必用关键词」，Pipeline Step 5 自动注入的作为「补充关键词」，合并后去重注入 LLM Prompt。

|------|------|------|
| PRD 一致性 | ⭐⭐⭐ | 与 PRD 的冲突点相同（Step 2 从「配主题」扩展为「主题与策略」），但字段聚合模式与现有实现（多字段选择）更接近，PRD 的偏离是「增量扩展」而非「范式颠覆」，更易在 PRD 补丁中解释。 |
| 用户需求满足度 | ⭐⭐⭐⭐ | 满足全部 7 项配置需求（模板、Persona、Story、Series、Timeline、Keywords、Methodology）。用户体验是「多字段表单」而非「单模板驱动」，对于高频配置用户可能需要更多点击，但 Copilot 智能推荐可缓解。 |
| 实施风险 | ⭐⭐⭐⭐ | 预计 5-7 天完成。无 ORM 化负担，Pipeline 改造范围明确，Alembic 迁移简单。Mode C 下 Copilot 只需新增「推荐模板」Action Card 和「下一步」Quick Action，与现有模式一致。 |
| 长期演进 | ⭐⭐⭐ | 字段聚合模式在 3-4 个配置项时工作良好，但增长到 10+ 项时会变得臃肿。长期需向模板驱动迁移，方案 B 是过渡态。 |

**小计**: (3 + 4 + 4 + 3) / 4 = **3.5 / 5.0**

#### 产品经理结论

> **两个方案在用户满足度和长期演进上各有优劣，但实施风险差距显著**。
>
> 方案 A 是「正确但昂贵」的选择，在当前项目阶段（v4.0 Phase 1，尚有 5 条架构断层待补）实施风险过高。方案 B 是「足够好且可交付」的选择，能在 1 周内为用户提供完整价值。
>
> **建议**：实施方案 B，但前端组件设计时预留向模板驱动迁移的接口（如 TemplateVariableForm 组件抽象），降低未来迁移成本。

### 4.6 Copilot 智能推荐逻辑

**模板推荐**（后端 `GET /content-templates/recommend` 增强）：

```python
def recommend_templates(topic: str, platform: str, methodology_stage_id: Optional[str] = None):
    # 1. 基础过滤：tenant + platform + active
    candidates = query_content_templates(tenant_id, platform, "active")
    
    # 2. 评分
    scored = []
    for tmpl in candidates:
        score = 0
        # topic 包含匹配
        if topic.lower() in tmpl.prompt_template.lower():
            score += 10
        # methodology_stage 精确匹配（若传入）
        if methodology_stage_id and tmpl.extracted_structure.get("methodology_stage") == methodology_stage_id:
            score += 5
        # 使用次数
        score += min(tmpl.usage_count or 0, 5)
        scored.append((score, tmpl))
    
    # 3. 取 Top 3
    scored.sort(key=lambda x: -x[0])
    return [t for _, t in scored[:3]]
```

**Agent 推荐**（前端逻辑增强）：

function recommendAgent(platform: string, format: string, templateStructureType?: string): string | null {
  // 1. 基础映射（现有）
  const baseKey = `${platform}-${format}`
  const baseRecommendation = AGENT_RECOMMENDATIONS[baseKey]
  
  // 2. 模板结构类型加成（新增）
  if (templateStructureType) {
    // 例如：「避坑排雷型」→ 优先推荐图文 Agent
    // 例如：「剧情种草型」→ 优先推荐视频 Agent
    const structureAgentMap: Record<string, string> = {
      '避坑排雷型': 'content_forge_xhs_image',
      '剧情种草型': 'content_forge_xhs_video',
      // ...
    }
    const structureBoost = structureAgentMap[templateStructureType]
    if (structureBoost) return structureBoost
  }
  
  return baseRecommendation
}
