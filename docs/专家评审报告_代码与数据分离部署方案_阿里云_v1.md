# EcoDreamOmni 代码与数据分离部署方案 — 专家评估报告

> **评审日期**: 2026-05-31
> **评审目标**: 为阿里云生产环境部署设计"工程代码与数据存储分离"方案，确保系统升级时不影响数据
> **评审范围**: 基础设施架构、数据持久化、容器编排、配置管理、CI/CD 链路

---

## 一、执行摘要

当前项目在 **PostgreSQL/Redis 层面**已通过 Docker Named Volume 实现容器与数据的初步解耦，但存在 **3 个关键缺口**：

1. **上传文件绑定容器生命周期** — 素材库、缩略图、品牌知识文档全部落在容器内 `/app/uploads`，无 Volume 挂载，重建即丢失
2. **两套 Docker Compose 配置并存** — 根目录与 `docker/` 目录下的 compose 文件密码/端口/服务数均不一致，"CI 通过 ≠ 生产可用"
3. **运行时状态纯内存** — Harness Agent checkpoint 存储在进程内存，重启后状态归零

**本报告提供 3 个候选方案**，覆盖从 MVP 到规模化生产的全周期，推荐采用 **方案 B（SAE + 全托管云服务）** 作为当前阶段的最优解。

### 2.2 数据存储位置与风险等级

| 数据类型 | 存储位置 | 持久化方式 | 升级时风险 | 风险等级 |
|---------|---------|-----------|-----------|---------|
| 关系型数据库 (PG) | `/var/lib/postgresql/data` | Named Volume `postgres-data` | 低（Volume 与容器解耦） | 🟢 低 |
| 向量扩展 (pgvector) | 同上 | Named Volume | 低 | 🟢 低 |
| 缓存/队列 (Redis) | `/data` | Named Volume `redis-data` | 中（Redis 默认未开 AOF） | 🟡 中 |
| **上传文件** (uploads) | `/app/uploads` (容器内) | **❌ 无 Volume** | **高（重建即丢失）** | 🔴 **高** |
| **缩略图** | `/app/uploads/assets/...` | **❌ 无 Volume** | **高** | 🔴 **高** |
| **Stock 图片** | `/app/uploads/stock/...` | **❌ 无 Volume** | **高** | 🔴 **高** |
| **品牌知识文档** | `/app/uploads/brand_knowledge/...` | **❌ 无 Volume** | **高** | 🔴 **高** |
| Harness Checkpoints | 内存字典 `_checkpoints` | **❌ 无持久化** | 中（重启归零） | 🟡 中 |
| Alembic 迁移脚本 | Git 代码库 | 版本控制 | 无 | 🟢 低 |

### 2.3 已识别的关键风险

| # | 风险项 | 触发场景 | 后果 |
|---|-------|---------|------|
| R1 | **上传文件丢失** | `docker-compose up -d --build backend` 或 Dockerfile 变更后重建 | 素材库、品牌知识文档全部丢失，数据库记录指向 404 |
| R2 | **两套配置不一致** | CI 用 `docker/docker-compose.yml`，生产用根目录 `docker-compose.yml` | 密码/端口/服务数不同，"CI 通过"无法保证"生产可用" |
| R3 | **Python 版本冲突** | 根目录 Dockerfile 用 `python:3.14-slim`，`docker/` 用 `python:3.11-slim` | 依赖兼容性问题，构建失败或运行时异常 |
| R4 | **Redis 数据丢失** | Redis 默认未开启 AOF/RDB 持久化，容器异常退出 | Celery 队列、任务状态丢失 |
| R5 | **JWT 弱密钥** | `docker-compose.yml` 中 `${JWT_SECRET:-dev-secret-change-me}` | 若未覆盖环境变量，生产使用硬编码弱密钥 |

## 三、分离原则

### 3.1 "代码 vs 数据" 边界定义

```
┌─────────────────────────────────────────────────────────────┐
│                     无状态层（随代码升级而替换）                │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │ 容器镜像    │  │ 环境变量    │  │ 静态构建产物        │  │
│  │ (FastAPI)   │  │ (非敏感)    │  │ (React dist/)       │  │
│  └─────────────┘  └─────────────┘  └─────────────────────┘  │
│  特征：可任意重建、水平扩展、无唯一标识                        │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼ 通过网络/挂载访问
┌─────────────────────────────────────────────────────────────┐
│                     有状态层（独立于代码生命周期）              │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │ RDS PG      │  │ 云 Redis    │  │ OSS 对象存储        │  │
│  │ (结构化数据)│  │ (缓存/队列) │  │ (文件/图片/视频)    │  │
│  └─────────────┘  └─────────────┘  └─────────────────────┘  │
│  特征：不可重建、必须备份、有唯一标识（URL/Endpoint）          │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 升级安全准则

> **黄金法则**：任何一次 `docker build` 或 `docker-compose up --build` 必须满足：
> 1. 不删除或覆盖任何数据库文件
> 2. 不删除或覆盖任何用户上传文件
> 3. 不丢失任何运行中的任务状态
> 4. 配置通过外部注入，不硬编码在镜像内

## 四、候选方案

### 方案 A：保守渐进式 — ECS + Docker Compose + 托管数据服务

**目标**：最小改动，将当前 Docker Compose 架构直接迁移到阿里云 ECS，仅把数据层替换为托管服务。

```
┌─────────────────────────────────────────────────────────────┐
│                        阿里云 VPC                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │              ECS 实例 (2核4G 或更高)                  │   │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │   │
│  │  │ Nginx       │  │ FastAPI     │  │ Celery      │  │   │
│  │  │ (前端静态)  │  │ (后端API)   │  │ Worker+Beat │  │   │
│  │  └─────────────┘  └─────────────┘  └─────────────┘  │   │
│  │              Docker Compose on ECS                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                         │                                   │
│              ┌──────────┼──────────┐                        │
│              ▼          ▼          ▼                        │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────┐ │
│  │ RDS PostgreSQL  │  │ 云数据库 Redis  │  │ OSS / NAS   │ │
│  │ (高可用版)      │  │ (主从版)        │  │ (上传文件)  │ │
│  └─────────────────┘  └─────────────────┘  └─────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

**组件清单**：

| 组件 | 选型 | 月费估算 |
|------|------|---------|
| 计算 | ECS 2核4G（经济型 e） | ~40元 |
| 数据库 | RDS PostgreSQL 基础版 2核4G | ~19元 |
| 缓存 | 云数据库 Redis 256MB 主从 | ~25元 |
| 文件存储 | OSS 标准存储 50GB | ~6元 |
| 网络 | 公网 IP + 安全组 | 免费 |
| **合计** | | **~90元/月** |

**优势**：
- 改动最小，熟悉 Docker Compose 的团队可快速上手
- 成本最低，适合验证期/MVP
- 网络延迟最低（同 VPC 内网访问 RDS/Redis）

**劣势**：
- ECS 为单点，无自动扩缩容，升级需停机或蓝绿部署
- 仍需自行管理操作系统补丁、Docker 守护进程
- Docker Compose 多服务在同一主机，资源争抢

**适用场景**：预算敏感、团队无 K8s 经验、MVP 验证期

### 方案 B：平衡推荐式 — SAE + 全托管云服务 (Recommended)

**目标**：应用层完全 Serverless 化，数据层全部使用托管云服务，实现真正的"代码与数据分离"。

```
┌─────────────────────────────────────────────────────────────┐
│                        阿里云 VPC                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  SAE (Serverless 应用引擎)                           │   │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │   │
│  │  │ Nginx       │  │ FastAPI     │  │ Celery      │  │   │
│  │  │ (静态托管)  │  │ (Web应用)   │  │ (后台任务)  │  │   │
│  │  └─────────────┘  └─────────────┘  └─────────────┘  │   │
│  │  特征：自动弹性、按量计费、零服务器运维               │   │
│  └─────────────────────────────────────────────────────┘   │
│                         │                                   │
│              ┌──────────┼──────────┐                        │
│              ▼          ▼          ▼                        │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────┐ │
│  │ RDS PostgreSQL  │  │ 云数据库 Redis  │  │ OSS + CDN   │ │
│  │ (高可用版)      │  │ (1GB 主从高可用)│  │ (上传文件)  │ │
│  └─────────────────┘  └─────────────────┘  └─────────────┘ │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  云效 Flow (CI/CD) 或 GitHub Actions + ACR          │   │
│  │  代码提交 → 构建镜像 → 推送到 ACR → 自动部署到 SAE   │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**组件清单**：

| 组件 | 选型 | 月费估算 |
|------|------|---------|
| 计算 | SAE 2实例 1核2G + 弹性 | ~300元 |
| 数据库 | RDS PostgreSQL 高可用版 2核4G | ~348元 |
| 缓存 | 云数据库 Redis 1GB 主从高可用 | ~100元 |
| 文件存储 | OSS 200GB + CDN 100GB 流量 | ~50元 |
| 镜像仓库 | ACR 个人版 | 免费 |
| CI/CD | 云效 Flow 基础版 | 免费 |
| **合计** | | **~800元/月** |

**优势**：
- **代码与数据完全分离**：SAE 应用无状态，可随时替换版本而不触碰任何数据
- **零服务器运维**：无需管理操作系统、Docker、补丁
- **自动弹性**：流量高峰自动扩容，低谷自动缩容
- **内置灰度发布**：SAE 支持按流量比例灰度，升级零停机
- **配置外置**：SAE 内置 ConfigMap/Secret，与代码完全解耦

**劣势**：
- 成本高于 ECS（约 3~4 倍）
- 需要适应 Serverless 模型（如冷启动、状态外置）
- SAE 对长时间运行的后台任务（Celery Beat）有一定限制

**适用场景**：生产环境正式运行、团队希望减少运维负担、需要弹性扩缩容

## 六、推荐方案：方案 B（SAE + 全托管云服务）

### 6.1 推荐理由

1. **当前阶段最优性价比**：EcoDreamOmni 处于产品验证到早期生产的过渡阶段，SAE 的"按量计费 + 自动弹性"避免了资源浪费
2. **真正的代码-数据分离**：SAE 应用实例完全无状态，所有数据（数据库、缓存、文件）托管在独立云服务，升级应用仅需替换镜像，零数据风险
3. **运维负担最低**：团队可将精力集中在产品迭代，而非服务器维护
4. **平滑演进路径**：当业务增长需要微服务治理时，SAE 可平滑迁移到 ACK，无需重写架构

## 八、风险与缓解措施

| # | 风险 | 影响 | 缓解措施 |
|---|------|------|---------|
| 1 | **数据迁移过程中丢失** | 高 | 使用 DTS 增量迁移，保持双写直到切换完成；迁移前全量备份 |
| 2 | **OSS 访问延迟** | 中 | 使用内网 Endpoint（不经过公网）；大文件启用 CDN 加速 |
| 3 | **SAE 冷启动** | 低 | SAE 支持预留实例，可配置最小实例数为 1，避免冷启动 |
| 4 | **Celery Beat 在 SAE 中受限** | 中 | Celery Beat 单独部署为 SAE 后台任务应用；或使用阿里云定时任务触发 HTTP 回调 |
| 5 | **配置泄露** | 高 | 所有敏感配置通过 SAE Secret 管理，不在代码/Git 中出现 |
| 6 | **回退困难** | 中 | 切换流量前保留旧环境运行 1 周；DNS TTL 设置为 60 秒便于快速回退 |
| 7 | **成本超支** | 中 | SAE 配置最大实例数限制；RDS 选择基础版起步；OSS 配置生命周期规则自动归档旧文件 |

## 九、决策建议

### 如果您的情况符合以下任意一条：

| 条件 | 建议方案 |
|------|---------|
| 预算极度敏感（< 100元/月） | **方案 A**（ECS + Docker Compose），但需立即修复 uploads Volume 问题 |
| 追求"零运维 + 安全升级"，愿意为此支付 ~800元/月 | **方案 B**（SAE + 全托管）⭐ **推荐** |
| 团队有 K8s 经验，业务已进入规模化阶段 | **方案 C**（ACK Pro），或从方案 B 平滑演进 |
| 当前最紧迫的是"不丢数据" | 先执行 Phase 1（1~2 天），为 uploads 添加 Volume，再决定长期方案 |

### 最终建议

> **采用方案 B（SAE + 全托管云服务）**，按 Phase 1~6 分步实施。
>
> 理由：
> 1. EcoDreamOmni 当前处于"验证期向生产期过渡"阶段，SAE 的"自动弹性 + 零运维"特征与该阶段最匹配
> 2. 方案 B 真正实现了"代码与数据分离"——任何一次应用升级都不会触碰数据层
> 3. 实施周期短（约 7~10 天），团队学习成本低
> 4. 保留了向方案 C（ACK）平滑演进的路径，避免架构锁定
