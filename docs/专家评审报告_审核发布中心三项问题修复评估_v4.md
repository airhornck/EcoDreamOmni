### 问题 1：「全部」页签顺序与默认激活

#### 2.1.1 代码现状


#### 2.1.2 UE专家组诊断

**场景分析**：
运营人员打开审核发布中心时，首要诉求是「查看当前有哪些任务需要处理」，而非直接定位到某一特定状态。将「全部」置于末尾且默认显示「已通过」，导致：
1. **认知负担**：新进入页面时只能看到已审核通过的任务，遗漏了待审核和已驳回的任务
2. **操作冗余**：每次刷新页面后都需要手动点击「全部」才能获得全局视图
3. **信息架构违背**：「全部」作为聚合视图，在信息架构中应处于首位（最宽泛的筛选维度）

**UE结论**：按照「从全到精」的筛选惯例，「全部」应置于第一位并作为默认激活项。

#### 2.1.3 修复方案

| 维度 | 修复内容 | 工作量 |
|------|---------|--------|
| 类型定义 | `type TabKey = 'all' | 'pending' | 'approved' | 'revised' | 'rejected'` | 5min |
| 标签顺序 | 渲染数组改为 `['all', 'pending', 'approved', 'revised', 'rejected']` | 5min |
| 默认状态 | `useState<TabKey>('all')` | 5min |
| 过滤逻辑 | 保持现有 `filtered` 逻辑不变（`all` 已正确处理） | — |

**推荐标签顺序逻辑**：
```
全部 → 审核中 → 已通过 → 已打回 → 已驳回
```
此顺序按照「待处理 → 正向结果 → 负向结果」排列，符合运营工作流优先级。

**后端点亮逻辑**（`review_publish.py` 第157行）：

```python
can_publish_now=t.status == task_hub.TaskStatus.APPROVED_WAITING_PUBLISH
```

#### 2.2.2 UE专家组诊断

**当前设计的问题**：

| 问题 | 说明 | 影响 |
|------|------|------|
| 操作入口分散 | 卡片上既有审核操作（通过/驳回/打回）又有发布操作，且只在特定标签页显示 | 运营在不同标签页间切换时操作逻辑不一致 |
| 缺少统一详情入口 | 卡片整体可点击跳转，但没有显式的「查看详情」按钮，运营可能 unaware | 详情页使用率可能偏低 |
| 快捷操作与详情操作边界模糊 | 列表页应做「轻量快捷操作」，但审核决策（通过/驳回/打回）需要看详情才能做出准确判断 | 误操作风险 |

**产品规则缺失**：
「确认发布」按钮的点亮规则虽然在后端有实现（`APPROVED_WAITING_PUBLISH` 状态），但前端和产品文档中均未明确向运营解释「什么条件下可以发布」。

#### 2.2.3 UI专家组诊断

**列表卡片的信息密度**：
- 当前卡片高度约 120-140px，操作区在右侧垂直排列
- 当同时存在审核按钮 + 发布按钮时，操作区宽度占用过大，挤压了内容预览区
- 图标按钮（通过/驳回/打回）缺少文字说明，新运营人员需要 hover 才能理解含义

**UI建议**：
1. 列表卡片操作区简化为单一「查看详情」按钮
2. 审核/发布操作移至详情页内，确保运营在充分审阅内容后再做决策
3. 若需保留列表快捷操作，建议在 hover 状态下展开操作菜单（而非常驻显示）

#### 2.2.4 修复方案

**方案 A：极简列表（推荐）**

列表卡片仅保留「查看详情」按钮，所有审核/发布操作集中在详情页：

```tsx
<div className="flex items-center gap-2 ml-3">
  <Button size="sm" variant="outline" onClick={(e) => {
    e.stopPropagation()
    openDetail(item)
  }}>
    查看详情
  </Button>
</div>
```

**方案 B：分层操作（备选）**

保留列表快捷操作，但采用 hover 展开菜单模式：
- 默认显示：「查看详情」
- hover 展开：根据状态显示对应快捷操作
  - `pending` → 通过 / 驳回 / 打回
  - `approved_waiting_publish` → 确认发布

**「确认发布」点亮规则产品定义**：

| 条件 | 是否点亮 | 说明 |
|------|---------|------|
| 任务状态 = `APPROVED_WAITING_PUBLISH` | ✅ 点亮 | 人工审核已通过，等待运营最终确认 |
| 任务状态 = `HUMAN_WAIT` | ❌ 禁用 | 尚未完成人工审核 |
| 任务状态 = `RUNNING` | ❌ 禁用 | 工作流执行中，未到发布节点 |
| 已有 `cron_job_id` | ⚠️ 已配置循环 | 显示「已配置循环」标签，仍可点击确认发布 |
| 账号日配额不足 | ❌ 禁用 + Tooltip | 需后端在 `can_publish_now` 中增加配额校验 |

**后端增强**（`review_publish.py`）：

### 问题 3：详情页功能缺失

#### 2.3.1 代码现状

**当前详情页结构**（`ReviewPublishDetailPage.tsx`）：

```
左侧（内容审阅区）                    右侧（决策辅助区）
├── PlatformPreviewCard（只读）       ├── AgentSummaryCard（只读）
├── TopicReportPanel（可折叠）        ├── 审核历史（只读）
├── MarkdownBody（只读渲染）          └── 发布确认面板（条件显示）
```

**缺失功能清单**：

| 缺失功能 | 当前状态 | 用户需求 | 技术复杂度 |
|---------|---------|---------|-----------|
| 封面更换 | 仅 `img` 展示 | 支持素材库选取 + 本地上传（保存到素材库） | 中 |
| 标题编辑 | 纯文本展示 | 支持 Inline 编辑 + 保存 | 低 |
| 正文编辑 | Markdown 只读渲染 | 支持 Markdown 编辑器修改 + 保存 | 中 |
| 再次生成 | 无此按钮 | 点击后调用 ContentForge 重新生成 | 高 |
| 审核操作 | 无 | 详情页内直接通过/驳回/打回 | 低 |
| 通过审核按钮 | 无 | 在 pending 状态下显示「通过审核」 | 低 |

#### 2.3.2 架构专家组诊断

**根因分析**：

当前详情页定位为「只读审阅 + 发布确认」，缺失「内容修正」和「审核决策」能力。这是两个阶段的设计遗漏：

1. **内容修正层缺失**：`ReviewDetailResponse` schema 中没有设计内容修改的接口。`generated_content` 是只读返回的，没有 `PUT /api/review-publish-center/conclusions/{id}/content` 这样的更新端点。

2. **审核决策层割裂**：详情页没有接入 `human_in_loop` 的审核决策 API（`POST /api/human-in-the-loop/tasks/{id}/approve` 等），运营必须返回列表页才能做审核操作。

3. **再次生成链路未打通**：`ContentForge` 的重新生成需要调用工作流引擎回滚到内容生成节点，当前 `ReviewPublishDetailPage` 没有与 `workflow_engine` 的交互接口。

**数据流影响**：
```
用户修改标题/正文 → 需更新 task.prompt_variables.generated_content
                → 需触发合规重审（ComplianceGuard）
                → 需更新内容预览（content_preview）
                → 需记录修改历史（audit log）
```

#### 2.3.3 UE专家组诊断

**运营工作流分析**：

运营在详情页的典型操作序列应为：

```
1. 查看内容预览（封面/标题/正文/标签）
2. [可选] 对不满意的部分进行编辑或重新生成
3. [可选] 查看 Agent 摘要和选题报告
4. 做出审核决策（通过 / 驳回 / 打回）
5. 若通过，配置发布方式并确认发布
```

当前详情页只能完成步骤 1、3、5，缺失 2 和 4。

**交互设计建议**：

| 功能 | 交互模式 | 位置 |
|------|---------|------|
| 封面更换 | 点击封面图 → 弹出素材库 Drawer / 上传弹窗 | PlatformPreviewCard 顶部 |
| 标题编辑 | 点击标题 → inline input → 失焦自动保存 | PlatformPreviewCard 标题区 |
| 正文编辑 | 切换「编辑模式」→ Markdown 编辑器 | MarkdownBody 区域 |
| 再次生成 | 按钮触发 → 显示生成参数选项 → 确认 | 正文区顶部工具栏 |
| 审核操作 | 常驻底部操作栏 / 右侧决策区顶部 | 右侧决策区 |
| 通过审核 | 主按钮样式，与「确认发布」互斥显示 | 右侧决策区 |

#### 2.3.4 UI专家组诊断

**编辑状态的视觉区分**：

当进入编辑模式时，需要有明确的视觉反馈：
- 可编辑字段边框高亮（`border-primary`）
- 保存/取消按钮组跟随编辑区域
- 未保存离开时弹出确认对话框

**再次生成的加载状态**：
- 需要全屏或局部遮罩，防止运营在生成过程中做其他操作
- 显示生成进度（调用 ContentForge 的 SSE 流式返回）

#### 2.3.5 修复方案

**A. 后端 API 扩展**

```python
# review_publish.py 新增端点

class UpdateContentRequest(BaseModel):
    title: Optional[str] = None
    body: Optional[str] = None
    tags: Optional[List[str]] = None
    cover_image_url: Optional[str] = None

@router.put("/conclusions/{task_id}/content")
async def update_review_content(
    task_id: str,
    req: UpdateContentRequest,
    db: AsyncSession = Depends(get_db),
):
    """更新审核内容（标题/正文/标签/封面），触发合规重审标记"""
    t = await task_hub.get_task(db, task_id)
    if not t:
        raise HTTPException(404, detail="Task not found")
    
    # 合并更新到 prompt_variables
    generated = t.prompt_variables.get("generated_content", {})
    if req.title is not None:
        generated["title"] = req.title
    if req.body is not None:
        generated["body"] = req.body
        t.prompt_variables["content_preview"] = req.body[:200]
    if req.tags is not None:
        generated["tags"] = req.tags
    if req.cover_image_url is not None:
        generated["cover_image_url"] = req.cover_image_url
        t.prompt_variables["cover_image_url"] = req.cover_image_url
    
    t.prompt_variables["generated_content"] = generated
    t.prompt_variables["content_modified_at"] = datetime.utcnow().isoformat()
    t.prompt_variables["content_modified_by"] = "operator"
    
    await db.commit()
    return {"success": True}


@router.post("/conclusions/{task_id}/regenerate")
async def regenerate_content(
    task_id: str,
    db: AsyncSession = Depends(get_db),
):
    """触发内容重新生成，调用 WorkflowEngine 回滚到 ContentForge 节点"""
    t = await task_hub.get_task(db, task_id)
    if not t:
        raise HTTPException(404, detail="Task not found")
    
    # 打回修改状态，触发工作流回滚
    await task_hub.transition_task(db, task_id, "CONFIGURING")
    if t.execution_id:
        from src.services import workflow_engine
        workflow_engine.rollback_to_node(t.execution_id, node_type="content_generation")
    
    return {"success": True, "status": "CONFIGURING"}
```

## 四、风险与缓解

| 风险 | 严重度 | 缓解措施 |
|------|--------|---------|
| 运营在详情页误编辑内容后无法撤回 | 中 | 增加「修改历史」记录；提供「恢复原始版本」按钮 |
| 再次生成调用工作流引擎失败 | 中 | 后端增加 try-catch，失败时返回具体错误信息，前端提示用户 |
| 封面上传至素材库的权限问题 | 低 | 复用现有上传 API，权限体系已存在 |
| 编辑后合规状态失效 | 中 | 内容修改后自动清除原有合规结果，提示运营「内容已修改，建议重新审核」 |
| 并发编辑冲突 | 低 | 增加乐观锁或 `modified_at` 时间戳校验 |
