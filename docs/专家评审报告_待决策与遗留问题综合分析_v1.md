### 决策 3：W16 代码分割优化

**问题描述**: 是否对 `App.tsx` 路由结构做代码分割（lazy loading）。

**技术现状**:
- 当前 `App.tsx` 为同步导入所有页面组件，打包后 JS 主包约 **1.1MB**（gzip 后约 350KB）。
- 项目使用 Vite 6 + React 19，`React.lazy()` + `Suspense` 为标准方案。
- 按 `App.tsx` 路由表，可按模块拆分：Dashboard、TaskHub、ContentForge、ReviewPublish、Personas、Settings 等 6+ 个 chunk。

**选项对比**:

| 选项 | 工作量 | 首屏收益 | 风险 |
|------|--------|----------|------|
| **A. 授权实施** | 0.5d（`React.lazy()` 包裹 + `Suspense` fallback） | 首屏减少 ~40%（约 400KB → 200KB） | 极低 — 纯前端构建调整，无业务逻辑变更 |
| **B. 暂缓** | 0d | 无 | 无 |

**专家建议**:
> 建议 **选项 A（授权实施）**。理由：
> 1. 工作量仅 0.5 天，收益明确，**零业务逻辑风险**。
> 2. 1.1MB JS 在弱网环境下（3G/4G 移动端运营人员）加载时间 >3s，影响体验。
> 3. 代码分割与当前功能开发**不冲突**，可作为 W16 的"热身任务"（低风险 → 建立信心）。
> 4. 若担心 lazy loading 导致白屏，可给 `Suspense` 添加与当前 `PageHeader` 同风格的骨架屏。

**决策建议**: ✅ **授权实施，W16 优先做**（作为低风险开胃菜）。

### 决策 4：W19-W20 AccountPool 增强

**问题描述**: 是否新增状态机 Service + Celery 任务 + ORM 字段变更（每日配额 / 状态机 / 熔断）。

**技术现状**:

- `PlatformRule` L3 日配额检查当前为**硬编码/mock 状态**（`ReviewPublishDetailPage.tsx` 第 365-372 行显示"账号日配额充足"为静态文本）。
- `TaskORM` 中 `account_id` 为字符串外键，无联级配额校验。

**选项对比**:

| 选项 | 工作量 | 风险 | 数据变更 |
|------|--------|------|----------|
| **A. 授权实施（完整方案）** | 5d（状态机 + Celery 定时任务 + 配额 ORM + 熔断 Service + 测试） | 高 — 涉及 ORM 字段变更、数据库迁移、Celery 消费者新增 | `accounts` 表新增 `daily_quota`、`status` 字段；新增 `account_quota_logs` 表 |

> 4. 状态机和熔断延后到 **W21+ 稳定性专项**，与监控（Prometheus/Grafana）一起设计。

**决策建议**: ✅ **分拆实施**：W19 只做"每日配额"（影响最小），状态机和熔断延后。

### 问题 1：task-hub/create — PersonaStory 无法选择

**现象**: 进入 `http://localhost:5173/task-hub/create`，在 Step 2【人设与故事】中，"PersonaStory" 下拉框无法选择（灰色禁用或选项为空）。

**根因定位**:

| 维度 | 分析 |
|------|------|
| **前端绑定** | `TaskHubCreatePage.tsx` 第 324-337 行：`select` 的 `disabled` 条件为 `!formData.personaId \|\| personaStories.length === 0`。若 `personaStories` 为空数组，下拉框被禁用。 |
| **数据流** | `personaStories` 来自 `taskHubStore.fetchPersonaStories(personaId)`，调用 `/api/persona-stories?persona_id=xxx`。 |
| **后端响应** | `persona_story.py` 第 168-185 行：返回 `{ items: [...], total, limit, offset }`，其中每个元素为 `StoryOut`（含 `name` 字段）。 |
| **Store 处理** | `taskHubStore.ts` 第 198-206 行：`set({ personaStories: data.items \|\| [] })`。逻辑正确。 |

**核心根因判定**:
> 代码逻辑本身**无明显缺陷**。"无法选择"最可能的根因是：**数据库中未录入任何 PersonaStory 数据**，导致 `personaStories.length === 0`，进而 `disabled=true`。
>
> 但用户反馈"上一轮修复后仍存在"，说明数据可能已存在但仍无法选择。进一步排查发现：
> - `fetchPersonaStories` 为异步 action，若 API 返回延迟或网络错误，`error` 被设置但**无 UI 提示**（`taskHubStore.error` 未被页面消费）。
> - `TaskHubCreatePage.tsx` 未监听 `taskHubStore.error`，用户看不到"获取Story失败"的提示，误以为"无法选择"。

**修复方案**:
1. 在 `TaskHubCreatePage.tsx` Step 2 区域增加错误提示：`{taskHubStore.error && <p className="text-xs text-destructive">{taskHubStore.error}</p>}`
2. 若 `personaStories.length === 0` 且 `formData.personaId` 已选，显示引导文案："当前 Persona 下暂无故事线，请先前往 人设管理 → 故事线 创建。"
3. 在 `personaStories.length === 0` 时不 disabled 下拉框，改为显示空选项 + 提示，避免用户困惑。

**工作量**: 0.5d
**风险**: 🟢 极低

### 问题 3.1：review-publish-center — Agent 摘要"合规审核/互动预演/质量分"无法显示

**现象**: 详细页右侧【Agent 摘要】中，三项均显示"暂无 XX 数据"。

**根因定位**:

| 维度 | 分析 |
|------|------|
| **数据读取** | `AgentSummaryCard.tsx` 从 `detail.compliance_result`、`detail.prediction_result`、`detail.quality_score` 读取。 |
| **数据写入** | `task_hub.py` 第 707-728 行：工作流到达 HUMAN_APPROVAL 时，将 `execution.context` 合并到 `task.prompt_variables`。 |
| **模拟数据缺陷** | `task_hub.py` 第 513-532 行：`pool-predictor` Agent 返回的 `engagement_interval` 值为 **Python tuple** `(800, 1500)`，JSON 序列化后变为**数组 `[800, 1500]`**，但前端期望 **对象 `{ min, max, confidence }`**。 |
| **字段名不匹配** | 后端 `quality_score` 返回字段为 `overall`、`readability`、`engagement_potential`、`compliance_score`；前端 `AgentSummaryCard.tsx` 第 162-170 行期望的是 `title_attractiveness`、`body_completeness`、`tag_relevance`。 |
| **合规数据缺失** | `compliance-guard` Agent（第 650-656 行）返回 `compliance_result` 含 `level` 字段，但前端期望 `level` 为 `"pass"` 或 `"warning"`。`check_compliance` 实际返回值需进一步确认。 |

**核心根因判定**:
> **三重数据契约断裂**：
> 1. `engagement_interval` 数据结构：后端 tuple → 前端对象（**严重**）。
> 2. `quality_score` 字段命名：后端 `readability`/`engagement_potential` → 前端 `title_attractiveness`/`body_completeness`（**严重**）。
> 3. 若工作流未完整执行（如直接从 DRAFT 跳到 HUMAN_WAIT 的测试数据），`prompt_variables` 中无这些数据，前端正确显示"暂无数据"。

**修复方案**:
1. **统一数据结构**：修改 `task_hub.py` 第 513-532 行，将 `engagement_interval` 改为对象：
   ```python
   "engagement_interval": {
       "likes": {"min": 800, "max": 1500, "confidence": "medium"},
       ...
   }
   ```
2. **统一字段命名**：有两种选择：
   - **选项 A（改后端）**：将 `quality_score` 字段名对齐前端（`title_attractiveness`, `body_completeness`, `tag_relevance`）。
   - **选项 B（改前端）**：将前端 `AgentSummaryCard` 的字段名对齐后端（`readability`, `engagement_potential`）。
   > 建议 **选项 B（改前端）**，因为后端 `pool-predictor` 的字段命名更符合语义（`readability` = 可读性，不限于标题）。
3. **增加数据兜底**：在 `AgentSummaryCard.tsx` 中，当 `quality_score` 存在但字段名不匹配时，显示原始 JSON（避免空白）。

**工作量**: 0.5d
**风险**: 🟢 极低 — 纯字段映射调整

### 问题 3.2：review-publish-center — 选题报告点击报错，且未同步到 trend-scout

**现象**: 点击【选题报告】展开时报错；报告内容未同步到 `http://localhost:5173/trend-scout`。

**根因定位**:

| 维度 | 分析 |
|------|------|
| **点击报错** | `TopicReportPanel.tsx` 第 67 行：`topic.tags.slice(0, 2)`。后端 `_simulate_node_output`（`task_hub.py` 第 489-501 行）`trend-scout` Agent 返回的 `topics` 数组中，每个 topic **缺少 `tags` 字段**。调用 `.slice()` 时抛出 `TypeError: Cannot read properties of undefined`。 |
| **未同步到 trend-scout** | 当前代码中**无任何逻辑**将 `review-publish-center` 的选题报告写入 trend-scout 模块。`trend-scout` 作为独立 Agent 页面，与审核发布中心是**单向数据流**（trend-scout → 任务创建），无反向同步机制。 |

**核心根因判定**:
> 1. **崩溃根因**：后端模拟数据字段缺失（`tags`）→ 前端未做防御式编程。
> 2. **同步缺失根因**：产品流程设计上，选题报告应在**任务创建工作流**中由 `trend-scout` Agent 生成，并随任务一起流转到审核发布中心。当前审核发布中心看到的是**任务附带的报告副本**，而 trend-scout 页面显示的是**独立生成的报告**——两者数据源不同，自然不同步。

**修复方案**:
1. **防崩溃**：`TopicReportPanel.tsx` 第 67 行改为 `{(topic.tags || []).slice(0, 2)}`。
2. **补齐模拟数据**：`task_hub.py` 第 494-499 行，`trend-scout` 返回的 topic 增加 `tags: [base_topic]`、`estimated_engagement: 0`、`status: "adopted"`、`source_report: report_id`。
3. **同步机制（产品决策）**：
   - 若需求是"审核页采纳的选题应沉淀到 trend-scout 知识库"：需新增 API `POST /trend-scout/sync-from-task`，在审核通过时将 `topic_report` 写入 trend-scout 的 Topic 表。

### 问题 3.3：review-publish-center — 正文内容富文本修改点击报错，修改按钮不明显

**现象**: 点击【正文内容】的编辑区域报错；修改按钮（编辑图标）不易发现。

**根因定位**:

| 维度 | 分析 |
|------|------|
| **修改按钮不明显** | `MarkdownBody.tsx` 第 71-77 行：编辑按钮使用 `opacity-0 hover:opacity-100`，**默认完全透明**，用户必须将鼠标悬停在正文区域右上角才能看到。这属于 UX 设计缺陷。 |
| **点击报错** | `MarkdownBody.tsx` 使用 `@tiptap/react` + `StarterKit`。`useEditor` 的依赖数组为 `[isEditing]`（第 35 行），意味着每次进入/退出编辑模式都会**销毁并重建编辑器实例**。若 `body` 为 `undefined` 或 `null`，`editor.commands.setContent(body)` 可能报错。 |
| **潜在空值问题** | `ReviewPublishDetailPage.tsx` 第 279 行：`<MarkdownBody body={generated.body} ... />`。若 `generated` 为 `null` 或 `generated.body` 为 `undefined`，`MarkdownBody` 接收的 `body` 为 `undefined`，`draft` state 初始化为 `undefined`，TipTap `content: undefined` 可能导致内部错误。 |

**核心根因判定**:
> 1. **UX 缺陷**：`opacity-0` 导致编辑按钮不可见，用户不知道可以编辑。
> 2. **健壮性缺陷**：`body` 可能为 `undefined`，`useEditor` 的重建机制在高频切换下不稳定。

**修复方案**:
1. **按钮可见性**：将 `opacity-0 hover:opacity-100` 改为始终可见（`opacity-100`），或使用更明显的"编辑正文"文字按钮（放在 CardHeader 操作区）。
2. **空值防御**：`MarkdownBody.tsx` 第 22 行改为 `const [draft, setDraft] = useState(body || '')`；第 31 行 `content: draft || ''`。
3. **编辑器稳定性**：将 `useEditor` 的依赖数组改为 `[]`（只初始化一次），通过 `editor?.commands.setContent(body)` 动态更新内容，避免频繁重建。

**工作量**: 0.5d
**风险**: 🟢 极低

### 问题 3.4：review-publish-center — 封面图片素材库预览看不到图片

**现象**: 点击封面图片更换，弹窗中"素材库"标签页的图片不显示（空白或裂图）。

**根因定位**:

| 维度 | 分析 |
|------|------|
| **数据流** | `CoverPickerModal.tsx` 调用 `useAssetPoolStore().fetchAssets()`，获取 `/api/assets` 数据。 |
| **字段映射** | `assetPoolStore.ts` 第 24-38 行 `_mapAsset`：从后端字段 `url`/`file_url`/`thumbnail_url` 映射到前端 `url`/`thumbnail_url`。 |
| **图片渲染** | `CoverPickerModal.tsx` 第 152-157 行：`<img src={asset.thumbnail_url \|\| asset.url} ... />`。 |
| **后端数据** | 若后端返回的素材 `file_url` 为空字符串、或为本地上传的 base64 但格式错误、或为相对路径（无前缀），则 `img.src` 无效。 |
| **类型过滤** | `CoverPickerModal.tsx` 第 34 行：`assets.filter((a) => (a.type || '').startsWith('image'))`。若 `_mapAsset` 的 `type` 推断失败（如 `meta_mime_type` 为空且文件名无后缀），`type` 被设为 `'unknown'`，该素材被过滤掉。 |

**核心根因判定**:
> **双向可能**：
> 1. **后端数据问题**：素材表中没有图片素材，或 `file_url` 字段为空/无效。
> 2. **前端过滤问题**：`_mapAsset` 的类型推断逻辑在特定数据格式下失败，导致图片素材被错误排除。
>
> 最可能场景：用户通过"本地上传"添加图片时，`createAsset` 发送 base64 到 `/api/assets/upload`，后端保存后返回的 `file_url` 可能不是可直接访问的 URL（如存储在本地磁盘路径 `/uploads/xxx.jpg`，而非 `http://localhost:8000/uploads/xxx.jpg`）。前端 `img` 标签无法加载本地路径。

**修复方案**:
1. **URL 补全**：在 `_mapAsset` 中，若 `url`/`file_url` 为相对路径，补全为后端基础 URL（如 `import.meta.env.VITE_API_BASE_URL` + 路径）。
2. **类型推断增强**：支持更多图片后缀（`.webp`, `.bmp`, `.svg`），并优先使用后端返回的 `mime_type`。
3. **空状态引导**：若 `imageAssets.length === 0`，提示"暂无图片素材，请切换到"本地上传"标签页添加"。
4. **本地上传预览优化**：上传 base64 后，在素材库中立即显示（当前需要重新 fetch，可改为乐观更新）。

**工作量**: 0.5d
**风险**: 🟢 极低
