### 问题 1：Dashboard「新建任务」与 TaskHub「新建任务」指向不一致

#### 现状核查

| 入口 | 当前行为 | 目标路径 |
|------|----------|----------|
| Dashboard 顶部按钮（"新建任务"） | `navigate('/content-forge')` | `/content-forge` |
| Dashboard 快捷操作（"新建内容任务"） | `href: '/content-forge?mode=create'` | `/content-forge?mode=create` |
| Dashboard 快捷操作（"新建互动剧本"） | `href: '/task-hub?type=interaction'` | `/task-hub?type=interaction`（参数未消费） |
| TaskHub「新建任务」按钮 | `openDrawer()` → 本地 Drawer | `/task-hub`（Drawer） |

#### 根因分析

Dashboard 的「新建任务」按钮**直接跳转到 ContentForge**，而非 TaskHub 的新建任务 Drawer。这是两个完全不同的创建路径：
- **ContentForge**：面向"内容生成"（直接写内容、选草稿）
- **TaskHub Drawer**：面向"任务编排"（选账号、选模板、选平台、设策略）

由于用户要求**注释掉 ContentForge**（问题 2），Dashboard 的「新建任务」按钮将成为**死链**。

#### 专家结论

**UE 专家组**：Dashboard 作为运营驾驶舱，「新建任务」应当跳转到**统一的任务创建入口**。当前指向 ContentForge 是历史遗留（ContentForge 早期被当作快速创建入口）。在 Wizard 改造完成后，Dashboard 应统一指向 `/task-hub/create`。

**技术专家组**：在 ContentForge 被注释掉后，Dashboard 的 `navigate('/content-forge')` 和快捷操作 `href: '/content-forge?mode=create'` 将触发 404 或空白页。必须同步修正。

| Dashboard「新建内容任务」快捷操作 | `DashboardPage.tsx:60` | 跳转到已注释页面，404 |
| Dashboard 智能选题「采用」按钮 | `DashboardPage.tsx:276` | 跳转到已注释页面，404 |
| Sidebar 导航「内容工坊」 | `Sidebar.tsx:66` | 导航到已注释页面，404 |
| App.tsx 路由 `/content-forge` | `App.tsx:43-44` | 路由存在但组件被注释，白屏或 404 |
| TaskHub Detail Drawer「预览/干预」按钮 | `TaskHubPage.tsx:976` | 跳转到 `/content-forge/:taskId`，如果路由被移除则 404 |
| ReviewPublishCenter 「前往修改」按钮 | `ReviewPublishCenterPage.tsx:432` | 跳转到 `/content-forge/:taskId` |

#### 专家结论

**架构专家组**：ContentForge 是任务驱动流程中的关键环节（Phase 3：内容生成与干预）。如果完全注释掉，以下链路断裂：
```
TaskHub → Detail Drawer → 「预览/干预」→ ContentForge → 生成/修改内容 → 提交审核
```

**建议**：
1. **短期**：如果必须注释，应将所有指向 ContentForge 的入口同步隐藏或重定向。
2. **中期**：ContentForge 的核心能力（内容生成、合规高亮、AI 辅助写作）应当被整合到 **TaskHub 的任务详情页**或**Wizard 的 Step 2（内容设定）**中，而非简单下线。
3. **长期**：保留 ContentForge 作为「独立内容编辑器」的能力，但从 TaskHub 进入时自动加载任务上下文。

### 问题 3：有了 `review-publish-center`，`compliance` / `human-in-the-loop` / `publisher` 是否还有必要

#### 三页职责对比矩阵

| 页面 | 核心职责 | 与 ReviewPublishCenter 的关系 | 是否可合并 |
|------|----------|------------------------------|------------|
| **Compliance** | 四层风控扫描（L1-L4）、规则库管理、批量扫描、扫描历史、一键修复 | RPC 只展示 `compliance_result` 摘要，无扫描能力 | ❌ **不可合并** —— 扫描工具与结论展示是两种不同能力 |
| **HumanInLoop** | 待审核任务列表、通过/驳回/打回决策 | RPC 有「审核中」Tab 展示待审核任务，但**无审核操作按钮** | ✅ **可合并** —— 将审核决策能力（通过/驳回/打回）添加到 RPC 的「审核中」Tab，即可下线 HITL |
| **Publisher** | 发布任务全生命周期管理（列表+日历）、草稿选择、立即/定时发布、发布执行 | RPC 只处理「审核通过后的发布确认」，无日历视图、无草稿管理、无历史追踪 | ❌ **不可合并** —— 发布中心是独立的管理域 |

#### 专家结论

**UE 专家组**：
- **Compliance 保留**：合规扫描是风控基础设施，运营人员需要独立使用它做内容预检。RPC 中的合规摘要只是结果展示。
- **HumanInLoop 可合并下线**：当前 HumanInLoop 页面（201 行）功能极简，与 RPC 的「审核中」Tab 高度重叠。建议将 HITL 的「通过/驳回/打回」操作直接放入 RPC 的 pending 列表卡片中，实现「一站式审核」。下线后可减少一个导航入口，降低认知负担。
- **Publisher 保留**：Publisher 的日历视图、发布历史、草稿选择是完整的发排系统，RPC 的发布确认只是其中一个子集。

**技术专家组**：
- 合并 HumanInLoop 到 RPC 的前端工作量约 0.5 天（将 `handleDecision` 逻辑迁移到 RPC 的 pending 列表项中）。
- 下线 HumanInLoop 后需移除：`pages/HumanInLoopPage.tsx`、路由、Sidebar 导航、后端 `/human-in-the-loop/pending` 的独立调用（保留 approve/reject/revise API，改由 RPC 调用）。

- 前端需补充：`updateAgent` store 方法 + 编辑表单 UI（可复用创建表单的 DOM 结构）。
- 预估工作量：0.5 天。

### 问题 5：`llm-cockpit` 白屏且无法返回

#### 根因链分析（技术专家组深度诊断）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        根因链：LLM Cockpit 白屏                                │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ① 后端 API 返回格式 ≠ 前端期望格式                                           │
│     ├── GET /api/llm-hub/models                                              │
│     │   返回: {"items": [...], "total": N}                                   │
│     │   前端期望: 数组 [...]                                                   │
│     │   实际代码: set({ models: data || [] })                                 │
│     │   结果: models = {"items": [...], "total": N} （对象！）                 │
│     │                                                                        │
│     └── 组件渲染: models.map((m) => ...)                                     │
│         在对象上调用 .map() → TypeError: models.map is not a function        │
│         → React 渲染崩溃 → 整个应用卸载（白屏）                                │
│                                                                             │
│  ② "无法返回" 的解释                                                         │
│     └── React 18 未捕获渲染错误会导致整棵树卸载                                │
│         Sidebar、Header、Outlet 全部消失                                       │
│         用户看到纯空白页，点击任何导航无效                                     │
│                                                                             │
│  ③ 附加数据结构不匹配问题（非致命但功能异常）                                  │
│     └── /api/llm-hub/scope-configs                                           │
│         返回字段: current_model（后端）                                        │
│         前端期望: model_name                                                  │
│         结果: 应用范围表格中「当前模型」列显示空值                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 代码证据

**后端** (`apps/backend/src/api/llm_hub.py:147-161`)：
```python
@router.get("/models", response_model=dict)
async def list_models(...):
    models = await lhs.list_models(...)
    enriched = [...]
    return {"items": enriched, "total": len(enriched)}  # ← 返回对象
```

**前端 Store** (`apps/frontend/src/stores/llmCockpitStore.ts:88-98`)：
```typescript
fetchModels: async () => {
  const res = await fetch('/api/llm-hub/models', { headers: authHeaders() })
  const data = await res.json()
  set({ models: data || [] })  # ← data 是 {"items": [...], "total": N}
}
```

#### 专家结论

**前端技术专家组**：这是典型的「前后端契约断裂」问题。修复方案明确：

1. **立即修复**（1 行）：`set({ models: data.items || [] })`
2. **附加修复**：`scopeConfigs` 的数据结构映射（`current_model` → `model_name`）
3. **防御性编程**：为所有 API 响应增加 `Array.isArray()` 检查，避免类似崩溃
4. **长期措施**：引入 ErrorBoundary，防止单个页面错误导致整个应用白屏

### 2.2 信息架构优化建议（基于用户 5 个问题）

#### 建议 1：统一「新建任务」入口（解决问题 1）

**当前问题**：Dashboard 和 TaskHub 的「新建任务」指向不同页面。

**推荐方案**：
- **短期**（ContentForge 被注释期间）：Dashboard「新建任务」按钮统一指向 `/task-hub`，并自动打开 Drawer。
- **长期**（Wizard 上线后）：Dashboard「新建任务」统一指向 `/task-hub/create`。

#### 建议 2：ContentForge 下线策略（解决问题 2）

**不要简单注释**，建议按以下顺序处理：

| 优先级 | 操作 | 影响 |
|--------|------|------|
| P0 | 隐藏 Sidebar「内容工坊」导航入口 | 用户不可见 |
| P0 | 修正 Dashboard 所有指向 ContentForge 的链接 | 消除死链 |
| P0 | TaskHub Detail Drawer 的「预览/干预」按钮改为打开**内联编辑器**或**重定向到 ContentForge 的简化版** | 保留干预能力 |
| P1 | 将 ContentForge 的核心编辑能力嵌入 TaskHub 任务详情页 | 链路闭环 |
| P2 | 完全移除 ContentForge 路由和组件 | 清理代码 |

#### 建议 3：合并 HumanInLoop 到 ReviewPublishCenter（解决问题 3）

**合并方案**：

```
ReviewPublishCenter（审核发布中心）
├── Tab: 已通过   ← 现有（发布确认）
├── Tab: 已驳回   ← 现有
├── Tab: 已打回   ← 现有
├── Tab: 审核中   ← 增强：增加通过/驳回/打回操作按钮
│   └── 任务卡片上直接显示 [通过] [驳回] [打回]
│   └── 点击任务 → 详情 Drawer → 完整内容预览 + Agent 摘要 + 审核操作
└── Tab: 全部     ← 现有
```

**合并后收益**：
- 减少一个导航分组入口
- 运营人员在一个页面完成「审核决策 + 发布确认」
- HumanInLoopPage 可下线（减少 201 行维护代码）

#### 建议 4：AgentOrchestra 增加编辑能力（解决问题 4）

在 Agent 列表的「操作」列增加「编辑」图标按钮，点击后：
1. 展开编辑表单（复用创建表单的字段结构）
2. 调用 `PUT /api/agents/:id` 更新
3. 刷新列表

#### 建议 5：系统治理分组内的 Bug 修复（解决问题 5）

- 修复 LLM Cockpit 白屏（1 行代码）
- 增加 ErrorBoundary（约 30 行），防止未来类似问题导致全局白屏
