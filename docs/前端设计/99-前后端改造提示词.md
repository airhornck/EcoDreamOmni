## 三、禁止事项（21 条）

- ❌ 跳过 HTML 预览直接开发
- ❌ HTML 预览未审核通过即进 Step 2/3
- ❌ **工作区放置业务按钮**（保存/提交/确认/新建/删除/发布/生成/通过/驳回）
- ❌ Copilot-Driven 页面未通过 Copilot 执行操作
- ❌ index.css 主题不一致时开发新页面
- ❌ 使用 any/unknown（normalize 入参 Record<string, unknown> 除外）
- ❌ 直接从 `response.data` 取数据（未解包 `.data`）
- ❌ API 调用 camelCase（后端必须用 snake_case）
- ❌ 原始枚举值直接显示（xhs → 小红书）
- ❌ 组件直接 fetch/axios（必须通过 Store/Query）
- ❌ 页面文件 > 400 行未拆分
- ❌ 重复定义 index.css 已有 utility class
- ❌ 未更新数据字典/契约/开发计划即提交
- ❌ 测试覆盖率 < 60%
- ❌ 未处理 VALIDATION_ERROR/RATE_LIMITED/AGENT_DEGRADED 等
- ❌ dangerouslySetInnerHTML 渲染用户输入
- ❌ 页面无 AI 原生体验
- ❌ 未上报 Copilot 上下文
- ❌ P0 未完成时开发 P1
- ❌ 发现后端缺口不记录、不输出补充文档
- ❌ 后端未提供 `copilot_followup` 导致 Copilot 状态流转断裂

## 八、FAQ

**Q1: 工作区「保存」按钮算不算业务按钮？**
> 算。所有保存/提交/确认/新建/删除/发布/生成/通过/驳回均禁止放在工作区，必须通过 Copilot Action Card。唯一例外是编辑器格式化工具栏（粗体/斜体等编辑辅助）。

**Q2: 已完成的页面（工作台/内容生产）如何回炉？**
> 按 `Copilot-Workspace-交互模式规范_v2.0.md` §8.1-8.2 执行：移除工作区按钮 → Copilot 提供对应 Action Cards → 更新 Store。HTML 预览需重建审核。

**Q3: 后端 API 未就绪，前端如何继续？**
> 使用统一 Mock（后端需求文档 §9.2），基于契约 Schema 生成 mock，必须包含 `copilot_followup` 字段确保 Copilot 流转可测试。

**Q4: Action Card API 调用失败如何处理？**
> Card 进入 `failed` 状态，展示错误 + [重试]。网络错误自动重试 3 次（指数退避）。错误码按 `01-API接口契约.md` 处理。

**Q5: 契约变更流程？**
> Step 2 冻结后，变更需：1) 更新 `01-API接口契约.md`；2) 通知对方；3) 15min 对齐会议。变更方负责更新对方 mock/定义。
