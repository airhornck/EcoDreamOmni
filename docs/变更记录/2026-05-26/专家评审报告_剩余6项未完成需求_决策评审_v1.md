## 一、评审纪律与架构红线

在提出任何方案之前，必须先明确 **EcoDreamOmni 的架构分层红线**（引自 `AGENTS.md`）：

| 层级 | 职责 | 能否直接操作DB |
|------|------|---------------|
| **Function层** (`*_function.py`, `*_orm.py`) | 基础数据CRUD、向量检索、文件上传、规则评估 | ✅ 可以 |
| **API层** (`api/*.py`) | HTTP路由、请求校验、依赖注入 | ✅ 可以（通过Function层） |
| **Agent层** (`task_hub.py`, `agent_orchestra.py`, `harness/`) | 编排、状态机、工作流驱动、HITL | ❌ **禁止** |
| **Skill层** (`skill_hub.py`, `*_service.py`) | 工具注册、技能执行、ReAct循环 | ❌ **禁止** |

**评审原则**：
- 凡是涉及数据库Schema变更、ORM操作、文件IO、HTTP外部调用的，**必须下沉到Function层或API层**。
- Agent/Skill层只允许通过Function API间接访问数据。
- 违反红线的方案一票否决。

### 需求5：AssetPool STS Token / 签名URL

#### 2.1 组织架构专家组

| 维度 | 评估 |
|------|------|
| **PRD对齐** | PRD FUNC-1 提到"版权链留存>=2年"，STS签名URL是云存储安全最佳实践 |
| **职责边界** | 云存储SDK调用 → Function层（`file_upload.py` 扩展）；Token生成 → API层 |
| **优先级建议** | **P2** — 当前本地存储已满足MVP，签名URL是安全增强项 |

#### 2.2 技术专家组

**关键决策点**：是否必须上云？

| 存储方案 | 优点 | 缺点 | 适用阶段 |
|---------|------|------|---------|
| 本地磁盘（当前） | 简单、无额外成本、开发快 | 无CDN、无容灾、URL永久暴露 | MVP |
| 阿里云OSS | 国内访问快、STS临时凭证成熟 | 增加成本、需要阿里云账号 | 生产 |
| MinIO（自建S3） | 兼容S3协议、可私有部署 | 增加运维负担 | 私有化部署 |

**推荐方案：渐进式**

```
MVP阶段（当前）：保持本地存储，增加访问控制
  └─ 增加文件访问鉴权中间件（非公开URL需Bearer Token）

生产阶段：增加OSS适配层
  Function层 [EXTEND]：
    src/core/file_upload.py
      ├─ save_upload_file()  [MODIFY]
      │   └─ 若 settings.STORAGE_MODE == "oss"，上传至OSS
      │   └─ 若 settings.STORAGE_MODE == "local"，保持本地写入
      └─ get_file_url(asset_id) -> str  [NEW]
         └─ local模式: 返回静态路径
         └─ oss模式: 返回 presigned_url(expires=3600)

API层 [NEW]：
  └─ POST /assets/sts-token
       └─ 调用阿里云 STS AssumeRole，返回临时凭证
```

**依赖包**：`oss2>=2.18.0`（阿里云）或 `boto3>=1.34`（AWS S3）

**工作量**: 
- MVP鉴权：0.5天
- OSS适配层：2天
- 前端URL适配：0.5天

**架构红线审查**: ✅ 通过。文件IO在Function层，Token生成在API层。

#### 2.3 UI/UE专家组

- 图片/视频预览时，如果URL即将过期（<5分钟），自动刷新签名。
- 增加存储模式配置页面（仅管理员可见）。

#### 2.4 综合判定

| 属性 | 判定 |
|------|------|
| **实现层级** | 基础代码层（Function文件IO扩展） |
| **Agent/Skill介入** | ❌ 不需要 |
| **工作量** | 0.5天（MVP鉴权）/ 3天（完整OSS） |
| **风险等级** | 🟡 中（涉及云厂商依赖和成本） |
| **推荐决策** | **建议分阶段**：先实施MVP鉴权（0.5天），OSS上云等生产环境确定后再实施 |
