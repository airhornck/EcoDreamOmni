## Phase 2-1: Timeline↔CronHub 绑定

### 目标
打通 TimelineLibrary 与 CronHub 之间的单向绑定：Timeline 事件配置 cron 表达式后，自动在 CronHub 中创建/更新/删除对应的定时任务。

### 修改文件

| 文件 | 变更点 |
|------|--------|
| `apps/backend/src/models/timeline_library_orm.py` | 新增 `cron_job_id VARCHAR(64)` 字段 |
| `apps/backend/src/services/timeline_library_function.py` | `create_event`: 有效 cron 则自动 `cron_hub.create_job()`，回写 job ID；`update_event`: cron 变化时替换 job（删旧建新）；`delete_event`: 级联删除关联 job；`event_to_dict`: 返回 `cron_job_id` |
| `apps/backend/src/api/timeline.py` | `TimelineEventOut` schema 增加 `cron_job_id` |
| `apps/backend/src/main.py` | lifespan 启动时执行 `ALTER TABLE timeline_events ADD COLUMN cron_job_id`（兼容已有数据库） |
| `apps/backend/tests/test_timeline_library_orm.py` | fixture 增加表结构自动迁移（`checkfirst=True` + `ALTER TABLE`）；新增 4 个绑定测试 |

### 绑定行为矩阵

| Timeline 操作 | CronHub 行为 |
|---------------|-------------|
| 创建事件 + `cron_expression` | 自动创建 job，`event.cron_job_id = job.id` |
| 创建事件 + 无 cron | 不创建 job，`cron_job_id = None` |
| 更新事件 + cron 变化 | 删除旧 job → 创建新 job → 更新 `cron_job_id` |
| 删除事件 | 若 `cron_job_id` 存在，先 `delete_job()` 再删事件 |
| 无效 cron | 非阻塞，跳过 CronHub 同步，Timeline 事件仍创建成功 |

### 导入映射规则

| 格式 | 解析策略 | Entry 映射 |
|------|----------|-----------|

| **Word** | 整篇提取 | 单个 entry，`name = filename` |
| **Excel** | 逐行读取 | 第一行 = headers，每行一个 entry，`name = 第一列值`，`content = 后续列的 header: value` |

## Phase 3-2: AssetPool 文件访问鉴权（MVP）

### 目标
当前文件通过 `/uploads/...` 静态文件服务直接暴露，任何知道 URL 的人都能访问。MVP 方案：增加鉴权 API endpoint，替代公开 URL。

### 修改文件

| 文件 | 变更点 |
|------|--------|
| `apps/backend/src/api/asset_pool.py` | 新增 `GET /assets/{asset_id}/download` 和 `GET /assets/{asset_id}/thumbnail`，需 Bearer Token 鉴权；检查 asset 存在且未删除；通过 `FileResponse` 返回本地文件 |

### 行为

| 场景 | 响应 |
|------|------|
| 合法用户 + 有效 asset | `FileResponse` 返回文件 |
| asset 不存在或已删除 | 404 |
| 文件在磁盘上不存在 | 404 |
| 未鉴权 | 401 |

> 注：`/uploads/` 静态文件服务仍保留用于向后兼容，前端可逐步迁移至鉴权 endpoint。
