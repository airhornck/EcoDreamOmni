### VF-1: task_hub.py — Agent 层直接持有 `AsyncSession`

#### 当前状态
- **导入**: `from sqlalchemy.ext.asyncio import AsyncSession`（第 13 行）
- **参数污染**: `db: AsyncSession` 出现在 **28 个函数签名**中
- **直接 DB 操作**: 无（已下沉到 `task_function.py`）
- **本质问题**: Agent 层虽然不再直接 `db.execute()`，但仍然**耦合了 DB session 类型**，破坏了层间隔离

#### 涉及功能范围
| 功能 | 影响 |
|------|------|
| TaskHub 创建/更新/删除 | API 层需改为直接调用 `task_function.py` |
| 工作流启动 (`start_workflow`) | 内部调用 `brand_knowledge_function`、`persona_story_service`、`llm_hub` 时需调整参数传递 |
| 批量任务创建 | 循环内调用 `task_function.create_task_in_db` 替代 `task_hub.create_task` |
| HITL 审批桥接 | `human_in_loop.py` 调用 `task_hub` 时不再需要传 `db` |
| Celery 任务恢复 | `celery_tasks.py` 调用 `task_hub` 时不再需要传 `db` |

#### 修复方案

**步骤 1**: 删除 `task_hub.py` 中所有 `db: AsyncSession` 参数
```python
# 改造前
async def create_task(db: AsyncSession, ...): ...

# 改造后
async def create_task(...): ...
    # 内部通过 get_db() 或让 API 层传入 domain 对象
```

**步骤 2**: API 层 (`api/task_hub.py`) 直接调用 `task_function.py` 做 CRUD
```python
# api/task_hub.py
from src.services import task_function as tf

@router.post("/tasks")
async def create_task(data: CreateTaskRequest, db: AsyncSession = Depends(get_db)):
    task = await tf.create_task_in_db(db, ...)
    # Agent 层只做编排
    await task_hub.notify_task_created(task.id)
```

**步骤 3**: `task_hub.py` 保留纯编排逻辑
- `start_workflow` / `submit_human_decision` / 状态机 shortcut
- 所有 DB 操作委托给 `task_function.py`

#### 工作量与风险
- **工作量**: 0.5 天（删除参数 + API 层调整 + 测试）
- **风险**: 🟢 低。无行为变更，纯参数传递链调整。
- **测试重点**: `test_task_hub.py`、`test_task_status_machine.py`、`test_human_in_loop_v2.py`

### VF-2: human_in_loop.py — HITL 层传递 `db` 参数

#### 当前状态
- **导入**: `AsyncSession`（第 13 行）
- **参数污染**: `db: AsyncSession` 出现在 **8 个函数签名**
- **直接 DB 操作**: 无（全部委托给 `task_hub`）
- **本质问题**: HITL 层作为 Agent/Service 的边界，不应接触 session 类型

#### 涉及功能范围
| 功能 | 影响 |
|------|------|
| 内容风险检测 | 纯内存计算，无影响 |
| 任务审批/驳回/修订 | 改为调用无 `db` 参数的 `task_hub` 函数 |
| 批量审批 | 循环调用无 `db` 参数的 `task_hub` 函数 |
| 审核统计 | 纯内存聚合，无影响 |

#### 修复方案

**依赖前置条件**: 必须先完成 **VF-1 (task_hub.py)**，因为 `human_in_loop.py` 的全部 DB 交互都通过 `task_hub`。

**步骤 1**: 删除 `human_in_loop.py` 的 `AsyncSession` 导入和 `db` 参数

**步骤 2**: 调整函数体，调用重构后的 `task_hub`
```python
# 改造前
async def approve_task(db, task_id, reviewer, reason):
    task = await task_hub.get_task(db, task_id)
    await task_hub.transition_task(db, task_id, ...)

# 改造后
async def approve_task(task_id, reviewer, reason):
    task = await task_hub.get_task(task_id)  # task_hub 内部自管 session
    await task_hub.transition_task(task_id, ...)
```

#### 工作量与风险
- **工作量**: 0.3 天（纯参数删除）
- **风险**: 🟢 低。强依赖 VF-1 完成顺序。
- **测试重点**: `test_human_in_loop_v2.py`

### VF-3: auth_service.py — Service 层直接操作 UserORM

#### 当前状态
- **导入**: `AsyncSession` + `src.models.user`（第 6、9 行）
- **DB 操作**: 4 个函数直接调用 `create_user` / `get_user_by_email` / `update_user_mfa` / `get_user_by_id`
- **本质问题**: 用户认证是核心横切关注点，但 UserORM CRUD 未下沉到 Function 层

#### 涉及功能范围
| 功能 | 影响 |
|------|------|
| 用户注册 | `api/auth.py` → `auth_service.register_user` → `create_user` |
| 用户登录 | `api/auth.py` → `auth_service.authenticate_user` → `get_user_by_email` |
| MFA 设置 | `api/auth.py` → `auth_service.setup_mfa` → `update_user_mfa` |
| MFA 验证 | `api/auth.py` → `auth_service.verify_mfa_code` → `get_user_by_id` |
| 当前用户解析 | `src/core/dependencies.get_current_user` → `get_user_by_id` |

> ⚠️ **注意**: `src/core/dependencies.py` 也调用了 `get_user_by_id`，这意味着提取 Function 层时，`dependencies.py` 的调用点也需要调整。

#### 修复方案

**步骤 1**: 新建 `src/services/auth_function.py`
```python
# auth_function.py — Function 层
from sqlalchemy.ext.asyncio import AsyncSession
from src.models.user import UserORM

async def create_user(db: AsyncSession, email: str, password_hash: str, ...) -> UserORM: ...
async def get_user_by_email(db: AsyncSession, email: str) -> Optional[UserORM]: ...
async def get_user_by_id(db: AsyncSession, user_id: str) -> Optional[UserORM]: ...
async def update_user_mfa(db: AsyncSession, user_id: str, mfa_secret: str) -> bool: ...
```

**步骤 2**: `src/models/user.py` 中的 DB 操作函数迁移到 `auth_function.py`
- 当前 `user.py` 混合了模型定义和 CRUD 函数，这是另一个架构问题
- 将 `create_user`, `get_user_by_email`, `get_user_by_id`, `update_user_mfa` 从 `user.py` 迁移到 `auth_function.py`

**步骤 3**: `auth_service.py` 改为调用 `auth_function`
```python
# auth_service.py — Service 层（无 AsyncSession 导入）
from src.services.auth_function import create_user, get_user_by_email, ...

async def register_user(db, email, password, ...):
    # 或者：让 API 层直接调用 auth_function，auth_service 只保留 token 逻辑
    user = await create_user(db, email=email, ...)
    return {"user_id": str(user.id), "access_token": create_access_token(...)}
```

**步骤 4**: `src/core/dependencies.py` 改为调用 `auth_function`
```python
# dependencies.py
from src.services.auth_function import get_user_by_id

async def get_current_user(credentials, db: AsyncSession = Depends(get_db)):
    ...
    user = await get_user_by_id(db, user_id)
```

#### 工作量与风险
- **工作量**: 0.5 天（新建文件 + 迁移 4 个函数 + 3 个调用点调整）
- **风险**: 🟡 中。`dependencies.py` 是全局入口，一旦出错所有 API 都无法鉴权。
- **测试重点**: `test_auth.py`（全覆盖注册/登录/MFA/Token 刷新）

### VF-6: celery_tasks.py — Celery Worker 直接创建 `AsyncSessionLocal`

#### 当前状态
- **导入**: `AsyncSession` + `AsyncSessionLocal`（第 39–40 行）
- **DB 操作**: `async with AsyncSessionLocal() as db:`（第 45 行），内部调用 `task_hub.get_task` / `transition_task` / `update_task`
- **本质问题**: Celery Worker 直接持有 session 工厂，且内部复刻了 `task_hub` 的工作流驱动逻辑

#### 涉及功能范围
| 功能 | 影响 |
|------|------|
| 流水线任务异步执行 | `pipeline_service.py` 触发 `run_pipeline_task` |
| 工作流恢复 | CronHub / 手动触发 `execute_workflow` |
| 定时发布 | 发布调度器触发 `scheduled_publish` |
| 心跳检测 | 健康检查触发 `heartbeat` |

#### 修复方案

**步骤 1**: 新建 `src/services/celery_tasks_function.py`

```python
# celery_tasks_function.py
from src.core.database import AsyncSessionLocal
from src.services import task_function as tf
from src.services import workflow_engine as we

async def drive_workflow(task_id: str):
    """Function 层封装：内部自管 session 生命周期。"""
    async with AsyncSessionLocal() as db:
        task = await tf.get_task_from_db(db, task_id)
        if not task:
            return {"error": "task_not_found"}
        # 调用 workflow engine（workflow engine 也是 Function 层）
        result = await we.execute_next_node(db, task)
        await db.commit()
        return result
```

**步骤 2**: `celery_tasks.py` 变为纯任务装饰器壳
```python
# celery_tasks.py — 仅保留 @shared_task 装饰和 asyncio.run() 桥接
from src.services.celery_tasks_function import drive_workflow

@shared_task
def execute_workflow(task_id: str):
    return asyncio.run(drive_workflow(task_id))
```

**步骤 3**: 消除与 `task_hub.py` 的 workflow 驱动逻辑重复
- 当前 `celery_tasks.py` 第 32–94 行复刻了 `task_hub._resume_workflow()` 的循环
- 应将 `task_hub._resume_workflow()` 提取为 `workflow_engine.execute_next_node()`，供 Agent 层和 Celery Worker 共同调用

#### 工作量与风险
- **工作量**: 1 天（新建文件 + workflow 循环提取 + 消除重复 + Celery 集成测试）
- **风险**: 🟡 中。Celery Worker 是异步任务执行的最后防线，一旦出错会导致任务堆积。
- **测试重点**: `test_task_hub.py`（workflow 恢复）、`test_cron_hub.py`（任务调度）、手动触发 Celery task 验证

## 五、专家建议

### 组织架构专家组
> "VF-1（task_hub 去 session 化）是解锁其他修复的前置条件，应优先实施。VF-3（auth）虽然独立，但涉及全局鉴权入口，建议在低峰期窗口执行。VF-4（llm_hub）代码量最大，建议分 2 个迭代完成：先提取模型注册和范围配置（高频调用），再提取用量日志和成本分析（低频但复杂查询）。"

### 技术专家组
