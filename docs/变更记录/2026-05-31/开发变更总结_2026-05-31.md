## 追加变更 — 工作流执行修复（2026-05-31 第二轮）

> **需求概述**: 修复工作流在审核通过和发布确认后无法恢复执行的问题
> **PRD 章节**: §10.4 / §10.6

### 根因分析

| 问题点 | 根因 | 影响 |
|--------|------|------|
| `human_in_loop.approve_task` | 审核通过后仅改变任务状态为 `APPROVED_WAITING_PUBLISH`，未恢复 workflow execution | publisher 节点永远不会执行 |
| `review_publish.confirm_publish` | 确认发布后仅改变任务状态为 `RUNNING`，未恢复 workflow execution | publisher 节点永远不会执行 |
| `celery_tasks_function.drive_workflow` | Celery 恢复时直接调用 `execute_next_node()`，未传入 `node_output`（缺少 `_simulate_node_output` 调用） | 节点执行无真实输出，execution context 为空 |

### 修改文件清单

| 文件 | 类型 | 说明 |
|------|------|------|
| `apps/backend/src/services/task_hub.py` | 修改 | `_resume_workflow` → `resume_workflow_execution`（公共函数）；`_simulate_node_output` → `simulate_node_output`（公共函数） |
| `apps/backend/src/services/human_in_loop.py` | 修改 | `approve_task` 最终通过后调用 `resume_workflow_execution` |
| `apps/backend/src/api/review_publish.py` | 修改 | `confirm_publish` 确认发布后调用 `resume_workflow_execution` |
| `apps/backend/src/services/celery_tasks_function.py` | 修改 | `drive_workflow` PAUSED 时调用 `resume_workflow_execution`；RUNNING 时调用 `simulate_node_output` 生成真实输出 |

# 结果: 44 passed

## 追加变更 — 账号池对抗、代理与隔离机制修复（2026-05-31 第八轮）

> **需求来源**: `docs/诊断报告_账号池对抗与代理_最终版.md`

### 详细修复说明

#### P0-1: 发布结果持久化

**问题**: `tasks` 表无 `published_url`/`platform_post_id`/`published_at`/`publish_error` 字段，无法验证发布真伪。

**修复**:
- `task_orm.py`: 新增 4 个发布审计字段
- `task_hub.py` `simulate_node_output(publisher)`: 发布成功后写入 `published_url` / `platform_post_id` / `published_at`；失败写入 `publish_error`
- Alembic migration `d9e3f4a5b6c7`: 为现有 `tasks` 表添加这 4 列

```python
# 发布成功后自动写入 DB
await update_task(db, task.id,
    published_url=result.get("published_url"),
    platform_post_id=result.get("platform_post_id"),
    published_at=_now(),
)
```

#### P0-2: Cookie 来源优先级

**问题**: `main.py` seed 使用 `demo_cookie`，而 `publish_to_xhs` 优先读取 `account.cookie`，导致发布始终用假 Cookie。

**修复**:
- `main.py` seed: 若 `REDNOTE_COOKIE` 环境变量已配置，用真实 Cookie 替代 `demo_cookie`
- `xhs_publisher.py`: 检测占位符 Cookie（`demo_cookie` / `placeholder` / 空），自动回退到 `settings.REDNOTE_COOKIE`

```python
if not cookie or cookie in ("demo_cookie", "", "placeholder"):
    cookie = settings.REDNOTE_COOKIE
```

#### P0-3: 账号池持久化

**问题**: `_account_pool_db` 纯内存字典，容器重启后所有账号、Cookie、代理绑定全部丢失。

**修复**:
- 新建 `account_pool_orm.py`: `AccountPoolEntryORM` 表（24 个字段，完整映射 `AccountPoolEntry`）
- `account_pool.py`: 新增 `load_pool_from_db()` / `save_pool_entry_to_db()` / `sync_pool_to_db()` / `delete_pool_entry_from_db()`
- `main.py` lifespan: 启动时从 DB 加载账号池；若为空则 seed 并保存
- `api/account_pool.py`: create/update/delete/quota 路由全部改为 `async`，操作后立即 `await save_pool_entry_to_db()`
- Alembic migration `c8d2e3f4a5b6`: 创建 `account_pool_entries` 表

#### P1-4: 代理配置

**问题**: 零代理配置，所有账号共享 ECS 公网 IP，小红书风控关联风险极高。

**修复**:
- `config.py`: 新增 8 个代理环境变量（HTTP + SOCKS5 各 host/port/user/pass）
- `main.py` startup: 若环境变量配置了代理，自动创建 `ProxyConfigEntry` 并绑定到 seed 的 xhs 账号
- `.env.prod.example`: 添加代理配置模板（用户填入真实代理凭据即可生效）

PROXY_SOCKS5_HOST=51.77.190.247

> **安全注意**: 代理凭据通过 `.env.prod` 环境变量注入，**绝不硬编码在代码中**。

#### P1-5: 健康检查 API NameError

**问题**: `api/publisher.py` 的 `xhs_health_check` 和 `xhs_account_status` 使用了 `settings.REDNOTE_COOKIE`，但未导入 `settings`。

**修复**: 添加 `from src.core.config import settings` 导入。

#### P2-6: XhsClient 缓存泄漏

**问题**: `_XHS_CLIENTS` 字典只增不减，内存泄漏 + 旧 Cookie 复用风险。

**修复**:
- LRU 淘汰：缓存上限 50，超限后删除最旧条目
- 访问时将 key 移到末尾（MRU）
- 新增 `invalidate_xhs_client_cache(cookie, user_agent)` 函数
- `account_pool.py` update/delete: Cookie 或代理变更时自动清除对应缓存

#### P2-7: 指纹死代码清理

**问题**: `fingerprint_engine.py` 生成 `canvas_noise` / `webgl_noise`，但 `requests`-based `XhsClient` 完全无法使用这些字段。

**修复**: 移除两个字段的生成，添加注释说明它们仅适用于未来的 Playwright 迁移。
