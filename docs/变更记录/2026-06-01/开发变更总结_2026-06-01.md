# 开发变更总结 — 2026-06-01

> **需求概述**: 修复审核发布中心详情页封面选择弹窗的三大问题：素材库列表获取失败、本地上传不稳定、缺少格式限制与持久化
> **PRD 章节**: §1.1 AssetPool / §10.6 Human-in-the-Loop（封面选择为 PRD 定义功能的实现细节；文件格式限制为新增安全约束）

## 修改文件清单

| 文件 | 类型 | 说明 |
|------|------|------|
| `apps/backend/src/services/asset_pool_function.py` | 修改 | `create_asset` 合并 kwargs 到 asset_data，修复 `meta_mime_type`/`meta_width`/`meta_height`/`meta_file_size` 丢失问题（+6 行） |
| `apps/frontend/src/components/review/CoverPickerModal.tsx` | 修改 | 1) 本地上传改用 `uploadAssetFile`（multipart）替代 `createAsset`（base64 JSON）；2) 添加 `.jpg/.jpeg/.png` 格式限制（代码层 + input accept 属性）；3) 上传后自动刷新素材库列表并切换到素材库标签（约 30 行改动） |
| `apps/backend/tests/test_asset_pool_orm.py` | 修改 | 新增 2 个复现测试：`test_create_asset_preserves_meta_mime_type`、`test_derived_asset_type_from_meta_mime_type` |
| `docs/PRD偏差报告.md` | 修改 | 新增「十二、素材库封面选择偏差」章节，记录 3 项偏差及修复状态 |


## 风险与后续

- **回归风险**: `create_asset` 合并 kwargs 后，其他调用方（如 `upload_asset` JSON 端点）若传入未预期的 kwargs 字段，也会被写入数据库。已通过 `hasattr(AssetORM, key)` 过滤，仅写入 ORM 已定义字段，风险极低。
- **待办**: 确认生产环境中 `save_upload_file` → `storage.save()` 的本地/OSS 存储是否正常生成缩略图（本地开发环境已通过 `_generate_thumbnail` 测试）

# 开发变更总结 — 2026-06-01（追加）

> **需求概述**: TaskHub 任务创建第四步「发布确认」新增循环执行预设与自定义日期时间配置，并修复 CronHub job 自动触发机制

## 修改文件清单

| 文件 | 类型 | 说明 |
|------|------|------|
| `apps/frontend/src/pages/TaskHubCreatePage.tsx` | 修改 | 1) 新增「每晚8点」预设；2) 新增「固定循环/自定义循环」子模式切换；3) 自定义循环支持起始日期/结束日期/时间点选择；4) 表单状态扩展 `cronDateStart`/`cronDateEnd`/`cronTime`/`cronMode`（约 80 行改动） |
| `apps/backend/src/api/task_hub.py` | 修改 | `CreateTaskRequest` 扩展 `cron_date_start`/`cron_date_end`；创建任务时保存到 `prompt_variables` |
| `apps/backend/src/services/task_hub.py` | 修改 | `create_task` 扩展 `cron_date_start`/`cron_date_end` 参数，保存到 `prompt_variables` |
| `apps/backend/src/api/review_publish.py` | 修改 | `ConfirmPublishRequest` 扩展 `cron_date_start`/`cron_date_end`；`confirm_publish` 将日期范围保存到 `target_params` |
| `apps/backend/src/services/cron_hub.py` | 修改 | `execute_job` 增加日期范围校验：当前日期在范围外 → SKIP |
| `apps/backend/src/services/celery_tasks.py` | 修改 | 新增 `check_and_execute_cron_jobs` Celery 任务，每分钟检查并触发到期的 CronJob |
| `apps/backend/src/celery_app.py` | 修改 | `beat_schedule` 新增 `check-cron-jobs-every-minute`（每分钟执行） |
| `apps/backend/tests/test_cron_hub.py` | 修改 | 新增 4 个复现测试：日期范围前 SKIP、日期范围后 SKIP、日期范围内 RUN、Celery 调度触发 |

## 调用关系变更

```
前端: TaskHubCreatePage.tsx → createTask(payload) → POST /api/tasks
后端: api/task_hub.py → task_hub.create_task(...) → prompt_variables["cron_date_start"/"cron_date_end"]
       ↓
      review_publish.py::confirm_publish → cron_hub.create_job(target_params={date_range_start, date_range_end})
       ↓
      celery beat (每分钟) → check_and_execute_cron_jobs() → cron_hub.execute_job()
       ↓
      cron_hub.execute_job() → 日期范围校验 → RUNNING / SKIPPED
```

## 风险与后续

- **回归风险**: `celery_app.py` beat_schedule 修改后，Celery Beat 容器需要重启才能生效。Docker Compose 已配置 `celery-beat` 服务，重启 `docker-compose up -d` 即可生效。
- **待办**: `cron_hub` 当前为内存存储（`_job_db`），容器重启后 job 丢失。后续 Phase 需考虑持久化到 PostgreSQL。
- **待办**: `check_and_execute_cron_jobs` 当前每分钟检查一次，精度为分钟级。如需秒级精度，需调整检查频率或采用更精确的调度机制。
