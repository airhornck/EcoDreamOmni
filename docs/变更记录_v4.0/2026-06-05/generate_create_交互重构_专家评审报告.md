## 二、Q1 详细评审：`/generate/create` 交互范式

### 2.1 PRD 真源定位

**`Copilot-Workspace-交互模式规范_v2.0.md` §3.3 / §8.2**（明确且未变更）：

```
创建向导 /generate/create:
  之前: 4 步向导，每步有 [下一步] [确认创建]
  之后: 4 步内容展示（配置预览），无提交按钮
        用户在 Copilot 中确认配置并点击「创建任务」
```

**`后端需求补充_内容生产_Copilot-Driven_2026-06-05.md` §6.3**（创建向导页 Action Cards 配置）：

```json
{
  "cards": [
    {
      "id": "confirm-create",
      "type": "confirm",
      "title": "✅ 确认创建",
      "description": "任务：猫咪驱虫避坑指南\n平台：小红书\nAgent：小红书图文生成（成功率 94%）\n账号：小艾养猫记",
      "actions": [
        { "id": "confirm", "label": "确认创建", "variant": "primary" },
        { "id": "back", "label": "返回修改", "variant": "secondary" }
      ]
    }
  ],
  "quick_actions": [
    { "icon": "chevron-left", "label": "上一步", "command": "prev_step" },
    { "icon": "chevron-right", "label": "下一步", "command": "next_step" }
  ]
}
```

### 2.2 用户提出的"四个卡片式编辑"方案分析

**方案描述**：将 Stepper 的 4 步改为 4 个独立卡片，每个卡片可编辑，完成后统一"运行任务"。

**与 PRD 的偏差**:

| 维度 | PRD 定义 | 用户方案 | 偏差 |
|------|---------|---------|------|
| 工作区职责 | 纯展示（配置预览） | 可编辑卡片 | ❌ 工作区承担编辑操作 |
| 提交入口 | Copilot Action Card | 工作区"运行任务"按钮 | ❌ 违反 Mode C |
| 步骤导航 | Stepper 点击切换 | 卡片间切换 | ⚠️ 交互范式变更 |
| 状态管理 | Copilot 驱动状态机 | 工作区自主管理 | ❌ 与 Copilot-Driven 冲突 |

**Mode C 合规风险**:

- 若每个卡片可编辑，则卡片内需要「保存」「确认」按钮 → **违反 Mode C 红线**（工作区禁止保存/提交/确认按钮）
- 若存在"运行任务"按钮在工作区 → **违反 Mode C 红线**（工作区禁止创建/运行按钮）
- 若改为 Copilot Action Card 驱动卡片编辑，则用户需要在 Copilot 和多个卡片间频繁切换 → **认知负担增加**

### 2.3 各维度评分（Q1）

| 维度 | 评分 | 理由 |
|------|------|------|
| 架构 | ⭐⭐ / 5 | 与 Copilot-Driven 架构方向冲突，引入双轨制风险 |
| 前端 | ⭐⭐ / 5 | 需重写 TaskHubCreatePage 为全新范式，现有 Stepper 组件废弃 |
| 后端 | ⭐⭐⭐ / 5 | 接口无需变更，但状态机逻辑需重新设计 |
| 产品 | ⭐⭐ / 5 | 与 PRD 定义冲突，Mode C 合规风险高 |
| **均值** | **2.3 / 5** | **< 3.0，不满足实施条件** |

### 2.4 Q1 结论

**驳回**。PRD v4.0 已明确定义 `/generate/create` 为 Stepper 纯展示 + Copilot 确认创建，当前实现（`TaskHubCreatePage.tsx`）与此一致。若用户有强烈的交互优化需求，应作为 **PRD v4.1 变更提案** 走正式的 PRD 修订流程，而非在本次修复中直接实施。

**替代建议**（不改动核心范式）:
- 保持 Stepper 纯展示，但优化每步内容的视觉呈现（更清晰的配置卡片样式）
- 在 Step 3（发布确认）增强配置摘要的可视化（与 Copilot 确认创建 Action Card 呼应）
- Copilot 面板在 Step 3 时主动高亮「确认创建」Action Card

## 三、Q2 详细评审：Copilot 对话与 Action Cards 共存策略

### 3.1 现状分析

**当前布局**（`AICopilotPanel.tsx`）:

```
┌─────────────────────────┐  ← aside (flex flex-col)
│  Header (shrink-0)      │
├─────────────────────────┤
│  ContextBar (shrink-0)  │
├─────────────────────────┤
│  MessageHistory         │  ← flex-1，占满剩余空间
│  (overflow-y-auto)      │
├─────────────────────────┤
│  ActionCardStack        │  ← 消息级 legacy Cards
├─────────────────────────┤
│  PageActionCardArea     │  ← 页面级 v4.0 Cards
│  (max-h-64)             │
├─────────────────────────┤
│  QuickActionBar         │  ← 快捷操作
├─────────────────────────┤
│  InputBox (shrink-0)    │  ← 输入框
└─────────────────────────┘
```

**问题**: 当 `MessageHistory` 消息量 > 5-6 条时，页面级 Action Cards 被推到面板底部，用户需要滚动 MessageHistory 才能看到 Cards。

### 3.2 方案对比

| 方案 | 描述 | Mode C 合规 | 技术可行 | UX 影响 | 推荐度 |
|------|------|:----------:|:-------:|:-------:|:------:|
| **A** | 保持现状（垂直堆叠） | ✅ | ✅ | ⚠️ Cards 不可见 | ⭐⭐ |
| **B** | MessageHistory 限高，Cards 始终可见 | ✅ | ✅ | ✅ 操作优先 | ⭐⭐⭐⭐ |
| **C** | Cards 在 MessageHistory 上方 | ✅ | ✅ | ⚠️ 消息被压 | ⭐⭐⭐ |
| **D** | 标签页切换（聊天 / 操作） | ✅ | ✅ | ❌ 增加操作路径 | ⭐⭐ |

### 3.3 推荐方案 B：分区固定策略

```
┌─────────────────────────┐
│  Header                 │
├─────────────────────────┤
│  ContextBar             │
├─────────────────────────┤
│  MessageHistory         │  ← max-h-[40%] 或 flex-shrink
│  (可滚动，但不无限扩张)  │
├─────────────────────────┤
│  ── 可拖拽分割线 ──      │  ← 用户可调整比例（未来增强）
├─────────────────────────┤
│  ActionCardStack        │  ← 始终可见区域
│  PageActionCardArea     │
├─────────────────────────┤
│  QuickActionBar         │
├─────────────────────────┤
│  InputBox               │
└─────────────────────────┘
```

**关键变更**:
1. `MessageHistory` 移除 `flex-1`，改为 `max-h-[50%]` 或基于内容的自适应高度
2. Action Cards 区域保证最小可见高度（如 `min-h-[160px]`）
3. 当无消息时，MessageHistory 显示欢迎语，不占空间
4. 当有消息时，MessageHistory 可滚动，但 Action Cards 始终在输入框上方可见

### 3.4 各维度评分（Q2）

| 维度 | 评分 | 理由 |
|------|------|------|
| 架构 | ⭐⭐⭐⭐ / 5 | 纯前端布局调整，不影响后端架构 |
| 前端 | ⭐⭐⭐⭐ / 5 | Flex 布局微调，技术成熟 |
| 后端 | ⭐⭐⭐⭐⭐ / 5 | 无后端变更 |
| 产品 | ⭐⭐⭐⭐ / 5 | 解决 Cards 不可见问题，不引入新交互范式 |
| **均值** | **4.3 / 5** | **≥ 3.0，满足实施条件** |

### 3.5 Q2 结论

**有条件采纳**。方案 B（分区固定策略）可直接实施，但需满足以下条件：

| 编号 | 条件 | 优先级 |
|------|------|--------|
| C-Q2-1 | `MessageHistory` 最大高度限制（如 `max-h-[50%]`），确保 Action Cards 始终可见 | P0 |
| C-Q2-2 | 无消息时 MessageHistory 收缩，不占用 Action Cards 空间 | P0 |
| C-Q2-3 | HTML 预览验证：5+ 条消息 + 2+ Action Cards 时的布局效果 | P0 |
| C-Q2-4 | 移动端适配（如有）：Action Cards 可横向滑动或折叠 | P1 |
