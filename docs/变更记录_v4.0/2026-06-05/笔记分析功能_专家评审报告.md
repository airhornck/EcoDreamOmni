## 二、需求文档核心问题（按严重程度排序）

### 🔴 P0 — 致命问题

| # | 问题 | 影响 | 改造方案 |
|---|------|------|---------|
| 1 | **Mode C 严重违规** | 需求设计 8 个业务按钮（上传/分析/生成/保存/导出等），与 v4.0 Copilot-Driven 范式直接冲突；若按此实现，Phase 4 需全面返工 | 全部按钮迁移至 Copilot Action Cards；工作区仅保留编辑控件和只读展示 |
| 2 | **新建独立页面/路由** | 与现有 实验室 爆款复刻功能高度重叠（~70%），另起炉灶造成重复建设 | 作为 实验室 「增强分析模式」，不新建 `/viral-analyzer` |
| 3 | **ViralTemplate 与 ContentTemplate 分裂** | 两套模板体系并存，用户认知混乱，维护成本翻倍 | `AnalysisReport` 作为 Skill 内部中间产物不存储；通过 `TemplateConverter` 映射为 `ContentTemplate` |
| 4 | **API 路由未纳入 Skill 体系** | `/api/viral/*` 独立 REST API 破坏 Agent-First 架构，形成「架构飞地」 | 扩展 `/lab/*` 接口；分析能力封装为 `content_structural_analysis` Skill |

### 🟠 P1 — 严重问题

| # | 问题 | 影响 | 改造方案 |
|---|------|------|---------|
| 5 | **纯规则引擎方案上限低** | jieba + 正则匹配无法处理网络新词、反讽、emoji 组合等；情绪识别基于关键词匹配，准确性存疑 | LLM 主路径（通过 LLM Hub）+ 轻量规则校准层；关键词库冷启动从 750+ 降至 50-100 |
| 6 | **爆款评分公式未经验证** | 权重无业务依据（30/25/20/15/10），无历史数据校准，可能沦为「数字装饰」 | 评分权重可配置化；初期标注「Beta」；或降级为状态指示器 |

## 三、最佳方案（综合四位专家意见）

### 3.1 功能定位

**实验室 · 爆款笔记分析增强模式**（非独立功能）

在现有 实验室 「爆款复刻」链路中，将 `StructureParseZone` 从「基础结构解析」升级为「深度结构分析」：

```
现有链路（保留）:
ViralInputZone → StructureParseZone → TemplateGenZone → VariableReplaceZone → DiffPreviewZone
                    ↓ 增强
              【深度分析面板】（新增，折叠式）
              • 结构类型标签（6种 + 置信度）
              • 关键词高亮（5维度，原文标注）
              • 爆款评分（简化版：结构完整性 + 关键词丰富度）
              • 成功因子 Top3
```

**边界原则**：
- 实验室 只做「分析 + 模板生成」，不做内容生成（生成由 `/generate` 负责）
- 变量替换和 Diff 预览保留在 实验室（现有功能）
- 所有操作通过 Copilot Action Cards 触发

### 3.2 前端架构（Mode C 合规）

```
LabPage.tsx（重构后）
├── Canvas 区域（flex-1）
│   └── Tab 导航（笔记编辑 / 分析预览 / 报告详情 / 模板预览）
│       ├── Tab 1: NoteEditorZone     ← 替代 ViralInputZone（结构化输入）
│       ├── Tab 2: AnalysisPreviewZone ← 新增（关键词高亮 + 评分环 + 情绪曲线）
│       ├── Tab 3: ReportDetailZone   ← 替代 StructureParseZone（JSON 树 + 详情）
│       └── Tab 4: TemplatePreviewZone ← 替代 TemplateGenZone（公式可视化 + 约束）
├── Copilot 区域（320px）
│   └── Action Cards 状态机:
│       ├── 默认态: 「📤 粘贴笔记」Action Card
│       ├── 输入后: 「🔍 分析结构」Action Card
│       ├── 分析完成: 「📋 查看报告」+ 「📐 生成模板」Action Cards
│       └── 模板生成后: 「💾 保存模板」+ 「📥 导出模板」Action Cards
└── （删除 ActionBar.tsx、现有 ViralInputZone/StructureParseZone/TemplateGenZone 重构）
```

**删除清单**：
- `ActionBar.tsx`（整文件删除，功能上收 Copilot）
- 现有 `ViralInputZone` 中的「AI 结构解析」按钮
- 工作区所有保存/生成/导出按钮

### 3.3 后端架构（Skill 化 + LLM 优先）

**API 设计**（扩展 `/lab/*`，不新建 `/api/viral/*`）：

| 方法 | 路径 | 功能 | 说明 |
|------|------|------|------|
| POST | `/api/lab/analyze` | 爆款笔记分析 | 替代原 `/api/viral/analyze`；返回 AnalysisReport |
| POST | `/api/lab/template` | 从分析报告生成模板草稿 | 替代原 `/api/viral/template`；内部转换为 ContentTemplate 格式 |
| GET | `/api/lab/history` | 分析历史 | 复用 ContentTemplate 列表，`source=viral_analyzer` 过滤 |
| GET | `/api/lab/keywords` | 关键词库查询 | 供前端高亮配色参考 |
| GET | `/api/lab/categories` | 赛道分类 | 复用现有分类枚举 |
| POST | `/api/content-templates` | 保存模板 | 复用现有接口（已有）|
| GET | `/api/content-templates/{id}` | 获取模板 | 复用现有接口（已有）|

**分析引擎三层架构**：

```
┌─────────────────────────────────────────────────────────┐
│  Layer 1: LLM 结构化分析（主路径）                        │
│  • 通过 LLM Hub 调用文本模型（qwen-max / deepseek-chat）  │
│  • 输入: 笔记全文 + 少样本分析 Prompt                      │
│  • 输出: JSON（structure_type, keyword_matches,           │
│           emotion_curve, title_analysis, ...）           │
├─────────────────────────────────────────────────────────┤
│  Layer 2: 规则校准层（轻量）                              │
│  • 最小关键词库（MVP 50-100 核心词）做交叉验证             │
│  • 修正 LLM 幻觉（structure_type 归一化到 6 种）          │
│  • 评分权重可配置（`structure_definitions.scoring_weights`）│
├─────────────────────────────────────────────────────────┤
│  Layer 3: Template Converter（Skill 内部）               │
│  • 将 keywordFormulas + constraints 翻译为                │
│    标准 ContentTemplate.prompt_template + variables       │
│  • 示例: "{数字}个{痛点}" → "{{数字}}个{{痛点}}"         │
└─────────────────────────────────────────────────────────┘
```

**数据库设计**（不新建 `viral_templates` 表）：

```sql
-- 复用现有 content_templates 表，增加字段:
ALTER TABLE content_templates
  ADD COLUMN source VARCHAR(50) DEFAULT 'manual',  -- 'manual' | 'viral_analyzer' | 'ai_generated'
  ADD COLUMN source_content_id VARCHAR(36),        -- 溯源笔记ID
  ADD COLUMN analysis_report JSONB;                -- 关联的分析报告（可选）

-- 新增结构定义表（6种结构类型 + 评分权重）
CREATE TABLE structure_definitions (
  id SERIAL PRIMARY KEY,
  structure_type VARCHAR(50) NOT NULL,  -- '种草测评型' | '干货合集型' | ...
  description TEXT,
  scoring_weights JSONB NOT NULL,       -- {"completeness": 0.5, "keyword_richness": 0.5}
  keyword_patterns JSONB,               -- 结构识别关键词正则
  created_at TIMESTAMP DEFAULT NOW()
);

-- 新增最小关键词库表（MVP 50-100 条）
CREATE TABLE keyword_library (
  id SERIAL PRIMARY KEY,
  keyword VARCHAR(100) NOT NULL,
  dimension VARCHAR(50) NOT NULL,       -- 'structure' | 'function' | 'emotion' | 'industry' | 'effect'
  weight FLOAT DEFAULT 1.0,
  applicable_structures VARCHAR(50)[],  -- 适用结构类型
  created_at TIMESTAMP DEFAULT NOW()
);
```

### 3.4 Phase 规划（修正版）

| Phase | 周期 | 内容 | 交付标准 |
|-------|------|------|---------|
| **P0 架构整改** | 1 周 | ① 前端 Mode C 改造（删除 ActionBar，重构 Zone）② API 契约更新 ③ 数据模型统一方案确认 | 工作区无业务按钮；契约文档更新；DB 迁移脚本 |
| **P1 MVP** | 2 周 | ① `content_structural_analysis` Skill（LLM 主路径 + 规则校准）② Copilot Action Cards 状态机 ③ 分析结果可视化（关键词高亮 + 评分环 + 结构标签）④ 模板 Converter 层 | 测试覆盖率 ≥ 60%；TypeScript 0 errors |
| **P2 产品化** | 2 周 | ① 情绪曲线图（recharts）② 报告详情 JSON 树 ③ 模板公式可视化 ④ 历史记录（复用 ContentTemplate 列表）⑤ 关键词库管理后台 MVP | HTML 预览审核通过 |
| **P3 优化** | 1 周 | ① 评分权重运营可配置 ② 分析结果缓存（Redis，TTL 7天）③ 性能优化（关键词高亮 Worker）④ E2E 测试 | Lighthouse ≥ 90 |

**总计**: 6 周（含 1 周架构整改）

### 3.5 MVP 裁剪清单

| 功能 | 原方案 | 最佳方案 | 理由 |
|------|--------|---------|------|
| 情绪曲线图表 | Phase 1 | Phase 2 | 对核心用户价值低，LLM 生成更准确 |
| 报告详情 JSON 树 | Phase 1 | Phase 2 / Admin 可见 | 内容运营不需要 JSON 树 |
| 模板公式可视化 | Phase 1 | Phase 2 | MVP 用纯文本公式即可 |
| 导出模板（JSON 下载）| Phase 1 | Phase 2 | 模板在系统内使用即可 |
| 历史记录列表 | Phase 1 | Phase 2（复用 ContentTemplate 列表）| 功能重复 |
| 互动数据输入 | Phase 1 | Phase 2 | 可选字段，评分公式未经验证 |
| 封面图片上传/预览 | Phase 1 | Phase 2 | 不做 OCR 则封面不参与分析 |
| 关键词匹配权重表格 | Phase 1 | 不开发 | 信息过载 |
