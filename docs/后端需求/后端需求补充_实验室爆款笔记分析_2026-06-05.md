## 一、需求变更摘要

### 1.1 原需求 → 最佳方案变更

| 变更项 | 原需求 | 最佳方案 | 变更原因 |
|--------|--------|---------|---------|
| 功能定位 | 独立新功能「ViralAnalyzer」 | 实验室 「增强分析模式」 | 与现有 实验室 重叠 70%，避免重复建设 |
| 路由 | 新建 `/api/viral/*` | 扩展 `/api/playground/*` | 不新建架构飞地，API 代表「爆款笔记分析」能力 |
| 模板模型 | 新建 `ViralTemplate` | 复用 `ContentTemplate` | 避免双模板体系，降低维护成本 |
| 分析引擎 | 纯规则引擎（jieba + 关键词库） | LLM 主路径 + 规则校准 | 规则引擎上限低，LLM 更准确 |
| 关键词库 | 6 大维度 750+ 词 | MVP 50-100 核心词（校准集） | 降低运营负债，冷启动更快 |
| 前端范式 | 按钮面板（8 个按钮） | Copilot Action Cards（Mode C） | v4.0 架构红线 |
| Phase 周期 | 5 周（2+2+1） | 6 周（1+2+2+1） | 增加 1 周架构整改 |

### 1.2 新建数据库表

```sql
-- 结构定义表（6种爆款结构 + 评分权重）
CREATE TABLE structure_definitions (
  id SERIAL PRIMARY KEY,
  structure_type VARCHAR(50) NOT NULL UNIQUE,
  description TEXT,
  scoring_weights JSONB NOT NULL DEFAULT '{"completeness": 0.5, "keyword_richness": 0.5}',
  keyword_patterns JSONB DEFAULT '[]',
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

-- 最小关键词库（MVP 50-100 条，作为 LLM 校准集）
CREATE TABLE keyword_library (
  id SERIAL PRIMARY KEY,
  keyword VARCHAR(100) NOT NULL,
  dimension VARCHAR(50) NOT NULL,  -- 'structure' | 'function' | 'emotion' | 'industry' | 'effect'
  weight FLOAT DEFAULT 1.0,
  applicable_structures VARCHAR(50)[],
  is_active BOOLEAN DEFAULT TRUE,
  created_at TIMESTAMP DEFAULT NOW(),
  updated_at TIMESTAMP DEFAULT NOW()
);

-- content_templates 扩展字段（复用现有表）
ALTER TABLE content_templates
  ADD COLUMN IF NOT EXISTS source VARCHAR(50) DEFAULT 'manual',
  ADD COLUMN IF NOT EXISTS source_content_id VARCHAR(36),
  ADD COLUMN IF NOT EXISTS analysis_report JSONB;
```

## 三、分析引擎设计

### 3.1 三层架构

```
Layer 1: LLM 结构化分析（主路径）
  └── 通过 LLM Hub 调用文本模型
  └── 输入: 笔记全文 + 少样本分析 Prompt
  └── 输出: 结构化 JSON（structure_type, keyword_matches, emotion_curve, ...）

Layer 2: 规则校准层（轻量）
  └── 最小关键词库（50-100 核心词）交叉验证
  └── 修正 LLM 幻觉（structure_type 归一化）
  └── 评分权重可配置

Layer 3: Template Converter
  └── 将分析结果转换为 ContentTemplate 格式
  └── keywordFormulas → prompt_template + variables
```

### 3.2 LLM Prompt 设计

```
你是一位小红书爆款内容分析专家。请对以下笔记进行结构化分析。

【笔记内容】
标题: {{title}}
正文: {{content}}

【输出要求】（严格 JSON 格式）
{
  "structure_type": "种草测评型|干货合集型|避坑排雷型|教程攻略型|对比测评型|个人故事型",
  "structure_confidence": 0.0-1.0,
  "viral_score": 0-100,
  "keyword_matches": {
    "structure": [{"keyword": "", "position": 0, "weight": 0.0}],
    "function": [...],
    "emotion": [{"keyword": "", "position": 0, "weight": 0.0, "intensity": 0.0}],
    "industry": [...],
    "effect": [...]
  },
  "title_analysis": {"pattern": "", "contains_number": false, "length": 0},
  "hook_analysis": {"hook_type": "", "effectiveness": 0.0},
  "body_analysis": {"sections": 0, "has_story": false},
  "cta_analysis": {"cta_type": "", "effectiveness": 0.0},
  "emotion_curve": [{"segment": 0, "emotion": "", "intensity": 0.0}],
  "success_factors": [""]
}
```

### 3.3 评分算法

```python
def calculate_viral_score(analysis_result, weights_config):
    """
    爆款评分 = 加权求和，权重可配置
    默认权重: completeness=0.5, keyword_richness=0.5 (MVP 简化版)
    Phase 2 扩展: emotion_curve, interaction_weight, emoji_strategy
    """
    scores = {
        'completeness': _score_structure_completeness(analysis_result),
        'keyword_richness': _score_keyword_coverage(analysis_result),
        'emotion_curve': _score_emotion_variety(analysis_result),
        'interaction_weight': _score_interaction_data(analysis_result),
        'emoji_strategy': _score_emoji_usage(analysis_result)
    }
    
    total = sum(scores[k] * weights_config.get(k, 0) for k in scores)
    return min(100, max(0, round(total)))
```

## 四、依赖与约束

### 4.1 新增依赖

| 依赖 | 用途 | 版本 | 备注 |
|------|------|------|------|
| `jieba`（已有） | 中文分词（校准层） | — | 项目已有 |
| `asyncpg`（已有） | 异步 PostgreSQL | — | 项目已有 |
| LLM Hub（已有） | 结构化分析主路径 | — | 复用 `services/llm_hub.py` |

### 4.2 关键约束

1. **所有 LLM 调用必须通过 LLM Hub**（工程纪律 §2.5）
2. **Agent 禁止直接访问数据库**（工程纪律 §2.1），分析 Skill 通过 Service 层操作
3. **评分权重可配置化**（存入 `structure_definitions.scoring_weights`）
4. **分析结果不持久化**（`AnalysisReport` 为中间产物，仅 `ContentTemplate` 存储）
5. **关键词库 MVP 50-100 条**（非 750+）

### 4.3 种子数据要求

运营团队需在 Phase 1 开始前提供：
- 6 种结构类型的定义和示例（每种 3-5 个示例笔记）
- 50-100 个核心高频关键词（按维度分类，含权重建议）
- 赛道分类与现有 `platform_content_type_styles` 的映射关系
