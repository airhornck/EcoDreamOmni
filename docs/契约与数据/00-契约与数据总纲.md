## 三、契约与数据的使用场景

### 场景 1：前后端并行开发
```
前端开发者 ←── 01-API接口契约.md ──→ 后端开发者
     │                              │
     │  约定：                        │
     │  • 错误码含义一致              │
     │  • 枚举值前后端同步            │
     │  • 响应格式统一                │
     │  • 幂等性保证                  │
```

### 场景 2：数据库 Schema 变更评审
```
开发者提交 Alembic 迁移脚本
        │
        ▼
对照 02-数据库ER图.md 检查：
  • 字段命名是否符合规范（snake_case）
  • 索引是否合理
  • 外键关系是否正确
  • 租户隔离字段（tenant_id）是否添加
        │
        ▼
评审通过 → 合并到 main
```

### 场景 3：状态机 Bug 排查
```
用户反馈：Pipeline 卡在"生成中"状态无法推进
        │
        ▼
对照 03-核心业务状态流转.md 检查：
  • 当前状态是否合法
  • 状态转换条件是否满足
  • 是否有未处理的异常导致状态悬停
        │
        ▼
定位根因 → 修复 → 补充回归测试
```

## 四、与数据词典的关系

```
契约与数据/          vs.    数据词典_v4.0/
─────────────────           ─────────────────
01-API接口契约.md           02-后端API路由层.md
  ├── 抽象契约（错误码/       ├── 具体端点（每个路由的
  │   枚举/幂等性/限流）       │   路径/参数/响应）
  └── 人工维护真源            └── 脚本自动生成

02-数据库ER图.md            01-数据库模型层.md
  ├── 全局 ER 关系图           ├── ORM 类定义
  ├── 表结构清单               ├── 字段详细注释
  └── 设计决策（为什么这样设计） └── 从代码提取

03-核心业务状态流转.md        05-模块调用关系矩阵.md
  ├── 状态机定义               ├── 模块间调用关系
  ├── 转换条件                 ├── 数据流向
  └── 业务规则                 └── 技术实现
```

> **区别**：`契约与数据/` 是**设计层**文档（为什么这样设计、业务规则是什么），`数据词典_v4.0/` 是**实现层**文档（代码里具体怎么写的）。两者互补，设计变更时先改契约文档，再同步到数据词典。

## 五、维护规范

1. **契约优先**：任何 API 变更、Schema 变更、状态变更，必须先更新 `契约与数据/` 中的对应文档，再写代码。
2. **版本标记**：每个契约文档顶部标注版本号，变更时在文档末尾追加变更记录。
3. **评审强制**：数据库 Schema 变更必须通过 ER 图评审；API 变更必须通过接口契约评审。
4. **自动化校验**：CI 中增加契约校验步骤（如 OpenAPI Schema 与代码一致性检查）。
