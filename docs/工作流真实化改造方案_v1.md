## 一、当前工作流节点真实状态全景图

标准内容生产工作流（`content_creation_standard`）共 8 个节点：

```
[0] 热点侦察(trend-scout) → [1] 结构分析(marketing-methodology) → [2] 框架生成(cf-outline) → [3] 正文生成(cf-body) → [4] 合规审核(compliance-guard) → [5] 互动预演(pool-predictor) → [6] 人工审核(HUMAN_APPROVAL) → [7] 发布(publisher)
```

| 节点 | agent_id | 当前状态 | 数据真实性 | 关键代码位置 |
|------|----------|----------|------------|--------------|
| **热点侦察** | `trend-scout` | 🔴 **纯模拟** | 基于 `task_name` 模板拼接的假 topic report | `task_hub.py:489-502` |
| **结构分析** | `marketing-methodology` | 🔴 **纯模拟** | 固定标题 + 固定 4 个 sections | `task_hub.py:504-511` |
| **框架生成** | `content-forge` / `cf-outline` | 🟡 **未实现** | 工作流模板定义了节点，但 `_simulate_node_output` 中无 `cf-outline` 分支，实际被跳过 | `task_hub.py:550-648`（无 cf-outline 处理） |
| **正文生成** | `content-forge` / `cf-body` | 🟢 **真实 LLM** | 调用 DeepSeek/OpenAI API，支持 BrandKnowledge RAG + PersonaStory 注入 + LLM Hub 动态配置 | `content_generator.py:125-165` |
| **合规审核** | `compliance-guard` | 🟢 **真实规则引擎** | jieba 分词 + 正则表达式，L1/L2/L3 三层合规扫描 | `compliance_engine.py:183-251` |
| **互动预演** | `pool-predictor` | 🔴 **纯模拟** | 固定数值（likes 800-1500、views 12000、quality_score 87） | `task_hub.py:513-532` |
| **人工审核** | `HUMAN_APPROVAL` | 🟢 **真实人工** | 人工在 review-publish-center 做决策 | `review_publish.py` |
| **发布** | `publisher` | 🟡 **部分真实** | 小红书（`xhs` 库）✅ 真实；抖音/视频号 ❌ mock | `xhs_publisher.py` / `playwright_publisher.py` |

**总结**: 8 个节点中，**3 个真实**（正文生成、合规审核、人工审核）、**1 个部分真实**（发布）、**4 个纯模拟**（热点侦察、结构分析、框架生成、互动预演）。

## 二、真实化改造方案（分节点）

### 2.1 热点侦察（trend-scout）— 优先级 P0

**当前问题**: 返回假数据，topic report 中的 topics 只有 1 个且缺少必填字段（`tags`、`estimated_engagement`、`source_report`、`status`），导致前端 `.slice` 崩溃。

**真实化目标**: 基于真实的小红书热搜/搜索建议数据生成 topic report。

#### 方案 A：小红书搜索建议 API（推荐）

利用已接入的 `xhs` 库的 `get_suggest_topic()` 接口，基于任务关键词获取真实的小红书话题数据。

```python
# 改造后逻辑示意
from src.services.xhs_publisher import _get_xhs_client

async def _real_trend_scout(topic: str, platform: str, account_id: str) -> dict:
    # 使用账号池中的 cookie 初始化 XhsClient
    account = get_pool_entry(account_id)
    client = _get_xhs_client(cookie=account.cookie, user_agent=account.fp.user_agent)
    
    # 1. 获取搜索建议话题
    suggest_topics = client.get_suggest_topic(topic)
    
    # 2. 搜索相关笔记获取热门标题（可选）
    search_result = client.get_note_by_keyword(topic, page=1, page_size=10)
    
    # 3. 组装 topic_report
    topics = []
    for idx, t in enumerate(suggest_topics[:5]):
        topics.append({
            "id": f"topic-{idx}",
            "title": t.get("name", topic),
            "source_report": report_id,
            "estimated_engagement": estimate_engagement(t),  # 基于搜索结果数量估算
            "tags": [topic, t.get("name", "")],
            "status": "candidate",
        })
    
    return {
        "topic_report": {
            "report_id": report_id,
            "selected_topic": topic,
            "topics": topics,
            "5a_stage": infer_5a_stage(topic),  # 基于关键词推断
            "audience_fit_score": calculate_fit(topic, persona),
        },
        "agent_summary": f"TrendScout: 从小红书获取 {len(topics)} 个真实话题建议",
    }
```

**所需凭证**:
- ✅ 小红书 Cookie（账号池已支持 per-account cookie）
- ⚠️ 注意：频繁调用可能触发风控，建议配合代理池使用

**工作量**: 1.5d（API 封装 + 数据组装 + 测试）
**风险**: 🟡 中 — 小红书 API 可能限频/返回异常，需做好降级（fallback 到模拟数据）

#### 方案 B：第三方热搜数据源

接入第三方小红书数据平台（如灰豚数据、千瓜数据等）的 API，获取行业热搜词、热门笔记标题。

**所需凭证**:
- 第三方平台 API Key（需采购）

**工作量**: 2d（调研 + 接入 + 测试）
**风险**: 🟡 中 — 依赖外部商业服务，成本不确定

### 2.2 结构分析（marketing-methodology）— 优先级 P1

**当前问题**: 返回固定标题和 4 个固定 sections（"环境降温"、"饮食补水"等），与选题无关。

**真实化目标**: 基于选题 topic 和平台特性，生成真实的内容结构大纲。

#### 方案：LLM 驱动的结构分析

利用已有的 LLM 调用能力（`call_llm`），让 LLM 根据选题生成内容结构。

```python
async def _real_marketing_methodology(topic: str, platform: str, trend_data: dict) -> dict:
    system_prompt = f"""你是一位{platform}平台的内容策略专家。
基于以下选题和趋势数据，生成一个吸引人的内容结构大纲。

选题：{topic}
平台：{platform}
趋势参考：{trend_data}

要求：
1. 标题要符合平台调性，带 emoji，有吸引力
2. 内容结构 3-5 个部分，每部分有清晰的小标题
3. 考虑用户的阅读习惯和平台的推荐机制
4. 输出 JSON 格式：{{"title": "", "sections": ["", ""]}}
"""
    
    raw = call_llm(system_prompt, f"请为「{topic}」生成内容结构大纲")
    outline = json.loads(raw)
    
    return {
        "outline": outline,
        "agent_summary": f"MarketingMethodology: LLM 生成「{topic}」内容框架",
    }
```

**所需凭证**: 无新增（复用现有 LLM Hub / DeepSeek API Key）

**工作量**: 0.5d（Prompt 工程 + 集成）
**风险**: 🟢 极低 — 复用已有 LLM 链路，无外部依赖

### 2.3 框架生成（cf-outline）— 优先级 P1

**当前问题**: 工作流模板定义了 `cf-outline` 节点，但 `_simulate_node_output` 中**无处理分支**，该节点实际被 fallback 到默认模拟输出。

**真实化目标**: 基于结构大纲生成详细的正文框架（每部分的要点、案例、互动点）。

#### 方案：LLM 驱动的框架生成

复用 `content_generator.py` 的能力，增加一个 `generate_outline` 函数。

```python
# 在 content_generator.py 中新增
def generate_outline(topic: str, platform: str, structure: dict, persona: dict = None) -> dict:
    """基于结构大纲生成详细的内容框架。"""
    system_prompt = _build_outline_prompt(persona, platform, topic, structure)
    user_prompt = f"请为「{topic}」生成详细的内容框架，包含每个部分的具体要点。"
    
    raw = call_llm(system_prompt, user_prompt)
    return json.loads(raw)
```

**所需凭证**: 无新增

**工作量**: 0.5d（新增函数 + Prompt 工程 + 工作流集成）
**风险**: 🟢 极低

### 2.4 互动预演（pool-predictor）— 优先级 P2

**当前问题**: 返回固定数值（likes 800-1500、views 12000），与真实内容无关。

**真实化目标**: 基于内容质量、历史数据、平台趋势，给出相对合理的互动预测区间。

#### 方案 A：基于历史数据的简单模型（推荐）

利用账号池中的历史发布数据（如果已存储），建立简单的线性回归或规则模型。

```python
# 伪代码
def predict_engagement(content: dict, account_history: list) -> dict:
    # 1. 内容特征：字数、标签数、标题长度、是否含 emoji
    # 2. 账号特征：粉丝数、历史平均互动率
    # 3. 时间特征：发布时段
    # 4. 规则预测（MVP 阶段无需 ML 模型）
    
    base_likes = 50  # 新号基础曝光
    if account_history:
        avg_likes = sum(h.likes for h in account_history[-10:]) / 10
        base_likes = max(base_likes, avg_likes * 0.8)
    
    # 内容质量加成
    quality_bonus = 1.0
    if len(content["body"]) > 500:
        quality_bonus += 0.3
    if len(content["tags"]) >= 5:
        quality_bonus += 0.2
    
    predicted_likes = int(base_likes * quality_bonus)
    
    return {
        "engagement_interval": {
            "likes": {"min": int(predicted_likes*0.5), "max": int(predicted_likes*2), "confidence": "low"},
            "comments": {"min": int(predicted_likes*0.05), "max": int(predicted_likes*0.2), "confidence": "low"},
            "collects": {"min": int(predicted_likes*0.1), "max": int(predicted_likes*0.5), "confidence": "low"},
        },
        "viral_probability": 0.1,  # 保守估计
        "best_publish_time": "19:00-21:00",
    }
```

**所需数据**:
- 账号历史发布数据（当前 `publish_task` 表已有 `published_url`、`published_at` 等字段，但缺少互动数据回采）
- **新增需求**: 需要定时任务回采已发布笔记的互动数据（likes/comments/collects）

**工作量**: 2d（历史数据模型 + 互动数据回采任务 + 集成）
**风险**: 🟡 中 — 需要稳定的互动数据回采机制

#### 方案 B：纯 LLM 预测（快速实现）

让 LLM 基于内容和平台特性给出预测区间（明确标注为"参考区间"）。

```python
system_prompt = """你是一位小红书运营数据分析师。基于以下内容和账号信息，给出一个合理的互动预测区间。
重要：必须标注这是"基于经验的参考区间，非平台真实数据"。"""
```

**工作量**: 0.5d
**风险**: 🟢 极低 — 但预测准确性有限

> **建议**: MVP 阶段采用**方案 B（LLM 预测）**，快速实现且无需额外数据基础设施。等账号积累了 20+ 条真实发布数据后，再升级到方案 A。

### 2.5 发布（publisher）— 优先级 P0

**当前状态**: 小红书（`xhs` 库）✅ 真实；抖音/视频号 ❌ mock。

#### 小红书发布 — 已真实 ✅

代码位置: `xhs_publisher.py`
- 使用 `xhs` 库（v0.2.13）调用小红书创作者 API
- 支持 cookie 登录、话题解析、图片上传、私密/公开发布
- 有账号健康检查（`check_account_status`）
- 支持代理池

**运行条件**:
1. 账号池中配置的账号必须有有效的**小红书 Cookie**
2. 或全局配置 `REDNOTE_COOKIE` 环境变量

#### 抖音发布 — 需改造

抖音没有公开的创作者 API，需通过 **Playwright 浏览器自动化**实现：

```
技术方案：Playwright + rebrowser-patches（已有 browser_pool.py 基础设施）
步骤：
1. 使用 Playwright 打开抖音创作者中心 (https://creator.douyin.com/)
2. 注入 Cookie 完成登录
3. 模拟上传视频/图文、填写标题、选择封面、添加话题
4. 点击发布按钮
```

**所需凭证**:
- 抖音账号的 Cookie（`sessionid`、`msToken` 等）
- 或扫码登录后的 Cookie 导出

**工作量**: 3d（Playwright 脚本 + 反爬对抗 + 测试）
**风险**: 🔴 高 — 抖音反爬严格，Cookie 易失效，脚本维护成本高

#### 视频号发布 — 需改造

视频号同样没有公开 API，需 Playwright 自动化：

```
技术方案：Playwright
步骤：
1. 打开视频号助手 (https://channels.weixin.qq.com/)
2. 微信扫码登录（或注入 Cookie）
3. 模拟发布流程
```

**所需凭证**:
- 微信账号的 Cookie（`wxuin`、`sid` 等）
- 或定期扫码登录

**工作量**: 3d
**风险**: 🔴 高 — 微信风控极严格，Cookie 存活时间短

## 五、风险分析与应对

| 风险 | 等级 | 描述 | 应对措施 |
|------|------|------|----------|
| 小红书 Cookie 失效 | 🟡 中 | Cookie 过期或被风控导致 API 调用失败 | ① 每个节点执行前检查 Cookie 有效性；② 失效时降级到模拟数据；③ 账号池页面增加"检测账号状态"按钮 |
| 小红书 API 限频 | 🟡 中 | 频繁调用搜索/发布接口触发限流 | ① 增加请求间隔（sleep 1-2s）；② 使用代理池轮换 IP；③ 失败时指数退避重试 |
| 抖音/视频号反爬升级 | 🔴 高 | Playwright 脚本被平台识别并封禁 | ① 使用 rebrowser-patches（已有）+ stealth 插件；② 定期更新脚本；③ 限制发布频率 |
| LLM API 调用失败 | 🟡 中 | DeepSeek/OpenAI 服务不可用或限流 | 已有 fallback 逻辑（`_fallback_generate`），但需确保 fallback 内容质量可接受 |
| 合规规则漏判 | 🟡 中 | 真实内容被平台判定违规 | ① 持续更新 `compliance_engine.py` 的关键词库；② 发布前增加"平台预检"（调用小红书的内容预审 API，如果可用） |
| 数据格式不匹配 | 🟢 低 | 真实 API 返回的数据结构变化导致前端报错 | ① 所有外部数据接入增加 schema 校验；② 异常时降级到 mock |
