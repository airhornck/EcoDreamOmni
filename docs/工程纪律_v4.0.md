## 二、v4.0 新增架构红线

### 2.1 Agent 禁止直接操作数据库

**❌ 错误示例**：
```python
# ContentForge Agent 直接查询数据库
class ContentForgeAgent:
    def generate_content(self, persona_id: str):
        # 直接 SQL 查询 —— 违反红线！
        persona = db.query(Persona).filter(Persona.id == persona_id).first()
        ...
```

**✅ 正确做法**：
```python
class ContentForgeAgent:
    def generate_content(self, persona_id: str):
        # 通过 Function API 获取数据
        persona_data = self.function_call(
            "knowledge_base.read_persona",
            {"persona_id": persona_id, "tenant_id": self.tenant_id}
        )
        ...
```

**红线规则**：
- Agent 只能调用 Function API 获取数据
- Agent 禁止直接执行 SQL、ORM 查询、Redis 读写
- 只有 Function Layer（knowledge_base / account_pool / analytics 等）可以访问数据库
- 违反此规则的代码必须在 Code Review 中被拒绝合并

### 2.2 Event Bus 优先于直接调用

**❌ 错误示例**：
```python
# Pipeline 直接调用 Agent
class PipelineExecution:
    def execute(self):
        agent = self.agent_registry.get("content_forge_001")
        result = agent.generate_content(data)  # 直接调用 —— 违反红线！
        ...
```

**✅ 正确做法**：
```python
class PipelineExecution:
    def execute(self):
        # 通过 Event Bus 发布任务
        self.event_bus.publish("task.assign", {
            "agent_id": "content_forge_001",
            "task_data": data,
            "pipeline_execution_id": self.execution_id,
            "tenant_id": self.tenant_id
        })
        
        # 等待 Agent 完成后通过事件回调
        result = self.event_bus.wait_for(
            f"task.completed.{self.execution_id}",
            timeout=300
        )
        ...
```

**红线规则**：
- Agent 间通信必须通过 Event Bus（Redis Streams / 消息队列）
- Pipeline 编排层通过 Event Bus 发布任务，不直接调用 Agent
- 直接调用仅允许在：Function 层内部调用、前端→后端 API 调用
- Swarm 模式依赖 Event Bus 实现并行执行

### 2.3 MCP 协议预留

**预留要求**：
- 所有对外暴露的 Agent/Skill 接口，接口定义必须兼容 MCP 协议结构
- 预留 `mcp_server` 目录结构（空壳即可）
- Function API 的输入输出 Schema 使用 JSON Schema，便于未来转为 MCP Tool 描述
- 不强制实现 MCP Server，但接口设计不能与之冲突

**示例**：
```python
# 当前：Function API
@function_api("content.generate")
def generate_content(params: ContentGenerateParams) -> ContentGenerateResult:
    ...

# 未来可无缝转为 MCP Tool（结构兼容）
# MCP Tool 描述：
# {
#   "name": "content.generate",
#   "description": "...",
#   "inputSchema": { ... },  # 与 ContentGenerateParams 的 JSON Schema 一致
#   "outputSchema": { ... }  # 与 ContentGenerateResult 的 JSON Schema 一致
# }
```

### 2.4 租户隔离强制检查

**❌ 错误示例**：
```python
# 查询忘记加 tenant_id 过滤
contents = db.query(ContentDraft).filter(ContentDraft.status == "published").all()
```

**✅ 正确做法**：
```python
# 必须注入 tenant_id 过滤
contents = db.query(ContentDraft).filter(
    ContentDraft.tenant_id == current_tenant_id,
    ContentDraft.status == "published"
).all()
```

**红线规则**：
- 所有业务查询必须包含 `tenant_id` 过滤（系统表除外）
- 应用层通过 SQLAlchemy query filter 自动注入 `tenant_id`
- ORM 层实现 `TenantQuery` 基类，自动追加 `tenant_id` 条件
- 违反此规则的代码必须通过 Code Review 拦截

### 2.5 模态路由必须使用 LLM Hub

**❌ 错误示例**：
```python
# 直接调用特定模型
response = openai_client.chat.completions.create(
    model="gpt-4o",
    messages=messages
)
```

**✅ 正确做法**：
```python
# 通过 LLM Hub 路由
result = llm_hub.generate(
    modality="text",
    input=messages,
    tenant_id=tenant_id,
    preferred_model="qwen-max",  # 国内首选
    priority="high"
)
# LLM Hub 自动处理：路由 → 限流 → 成本记录 → 兜底切换
```

**红线规则**：
- 所有 LLM 调用必须通过 LLM Hub
- 禁止直接实例化模型客户端（OpenAI / 百炼 等）
- LLM Hub 负责：路由决策、限流、成本记录、兜底切换、质量反馈

## 三、Prompt 工程红线

### 3.1 六层 Prompt 必须完整拼接

**六层结构**（顺序不可变）：
```
Layer 1: PlatformFormat（平台格式硬约束）
Layer 2: Structure Template（结构模板）
Layer 3: Brand Knowledge（品牌知识注入）
Layer 4: Keywords（关键词注入）    ← v4.0 新增 Step 5
Layer 5: Persona（人设/语气参数）
Layer 6: PlatformContentTypeStyle（风格 DNA）  ← v4.0 新增
```

**❌ 错误示例**：
```python
# 只拼接了 3 层
prompt = f"""
{platform_format}
{structure_template}
{persona}
生成内容：{topic}
"""
```

**✅ 正确做法**：
```python
from prompt_builder import SixLayerPromptBuilder

builder = SixLayerPromptBuilder()
prompt = builder.build(
    platform_format=platform_format,          # Layer 1
    structure_template=structure_template,    # Layer 2
    brand_knowledge=brand_knowledge,          # Layer 3
    keywords=keywords,                         # Layer 4
    persona=persona,                          # Layer 5
    platform_style=platform_style             # Layer 6
)
# builder 自动处理：去重、排序、注入标记
```

**红线规则**：
- 内容生成必须完整使用六层 Prompt
- 每层必须独立可替换（配置化）
- PlatformContentTypeStyle 的 `tone_preset` 必须映射到 Layer 6
- 缺少任何一层的 Prompt 视为不合格

### 3.2 Prompt 模板化规范

- 所有 Prompt 片段必须存储在数据库或配置中心，禁止硬编码
- Prompt 模板使用 Jinja2 语法，变量命名使用 `snake_case`
- Prompt 变更必须通过版本控制（Prompt 版本号 + 变更记录）
- 不同环境（dev/staging/prod）的 Prompt 必须隔离

## 四、代码质量红线

### 4.1 测试覆盖率 ≥ 80%

| 层级 | 覆盖率要求 | 检查方式 |
|------|-----------|---------|
| Unit Test | ≥ 80% | pytest-cov |
| Integration Test | ≥ 60% | 手工 + 自动化 |
| E2E Test | 核心流程全覆盖 | Playwright |

**红线规则**：
- PR 合并前必须通过覆盖率检查
- 新代码覆盖率 < 80% 的 PR 被拒绝合并
- 测试文件命名：`test_{module_name}.py`
- 测试数据使用 Factory Boy，禁止使用生产数据

### 4.2 类型检查 0 错误

```bash
# 后端（Python）
python -m mypy app/ --strict

# 前端（TypeScript）
npx tsc --noEmit
```

**红线规则**：
- CI 流水线中类型检查必须 0 错误
- 新代码必须包含完整类型注解（Python）或类型声明（TypeScript）
- 禁止使用 `Any` / `unknown` 作为逃逸类型

### 4.3 Lint 0 错误

```bash
# 后端
ruff check app/ --fix
black app/ --check

# 前端
eslint src/ --max-warnings=0
prettier src/ --check
```

### 4.4 错误码必须来自全局字典

**❌ 错误示例**：
```python
raise HTTPException(status_code=400, detail="参数错误")
```

**✅ 正确做法**：
```python
from app.errors import ErrorCode, raise_validation_error

raise_validation_error(
    code=ErrorCode.VALIDATION_ERROR,
    message="标题不能为空，且长度不超过20字",
    field_errors=[{"field": "title", "message": "标题不能为空"}]
)
```

## 五、文档维护纪律

### 5.1 契约先行原则

**开发流程**：
```
1. 设计阶段：更新 契约与数据/ 中的对应文档
2. 评审阶段：Schema/API/状态机评审通过
3. 开发阶段：按契约实现代码
4. 测试阶段：验证契约一致性
5. 上线阶段：文档与代码同步上线
```

**红线规则**：
- 任何 API 变更、Schema 变更、状态变更，必须先更新契约文档
- Alembic 迁移脚本必须与 `02-数据库ER图.md` 一致
- OpenAPI Schema 必须与 `01-API接口契约.md` 一致
- 状态机代码必须与 `03-核心业务状态流转.md` 一致

### 5.2 文档连锁更新

变更触发链：
```
代码变更（API/Schema/State）
    │
    ├──→ 契约与数据/ 对应文档更新
    │
    ├──→ 数据词典_v4.0/ 对应文档更新（或重新生成）
    │
    ├──→ 文档总纲_v4.0.md 状态更新
    │
    └──→ 开发Checklist.md 进度更新
```

## 六、安全红线

### 6.1 敏感信息处理

| 类型 | 存储方式 | 传输方式 | 日志处理 |
|------|---------|---------|---------|
| API Key | 加密存储（AES-256-GCM） | HTTPS | 脱敏输出 |
| JWT Token | Redis（TTL） | Header | 不记录完整 Token |
| 用户密码 | Argon2 哈希 | HTTPS | 不记录 |
| 数据库连接串 | 环境变量 / Secret Manager | 本地 | 不记录 |

### 6.2 审计日志

所有以下操作必须记录审计日志：
- 用户登录/登出/Token 刷新
- 内容发布/删除/修改
- Agent 状态变更
- Pipeline 人工干预（暂停/恢复/终止）
- 权限变更
