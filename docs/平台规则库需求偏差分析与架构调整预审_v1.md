## 第一章：PRD 业务真源对齐（强制检查）

### 1.1 该需求是否存在于 PRD 中？

**结论：PRD 中未定义用户所描述的「平台 API 发布格式规范库」。**

PRD 中 `PlatformRule` 的定义（§1.5.1、§2.5）：
> "PlatformRule 是 EcoDream Omni 的**平台合规真源**。存储法律红线（L1）、平台静态规则（L2）、账号状态规则（L3）、动态风控规则（L4）。所有内容发布前的**合规预检**、跨平台适配、**发布约束查询**，均以此 Function 为真源。"

PRD 中 `PlatformRule` 与 `Publisher` 的关系（§10.8 Workflow 对齐矩阵）：
> "Publisher 节点执行前，Workflow Engine 调用 PlatformRule Engine 预检（L3/L4），**违规则阻断**。"

### 1.2 用户新定义与 PRD 的偏差

| 维度 | 用户新定义 | PRD 定义 | 偏差说明 |
|------|-----------|----------|---------|
| **核心关注点** | API 发布时的**格式结构**和**字段约束**（标题长度、图片尺寸、字段类型） | **合规风控规则**（法律红线、静态关键词、账号状态限制、动态风控策略） | 用户关注「接口 Schema/格式规范」，PRD 关注「运营合规与风控策略」 |
| **规则性质** | 平台 API 的**技术接口规范** | 法律/平台/账号/动态**四层业务规则** | 一个是技术契约，一个是业务策略 |
| **与 Publisher 关系** | Publisher **按格式要求组装请求体** | Publisher 执行前做**合规预检**，违规则阻断 | 用户定义是「格式适配」，PRD 定义是「合规 gate」 |
| **数据来源** | `D:\project\lumina\data\platforms\*.yml` | 数据库 `platform_rules` 表 + 硬编码 Python | 数据源完全不同 |

### 1.3 需求定性

**这是一个「偏差需求」**——用户重新定义了「平台规则库」的业务内涵，与 PRD 中的定义不一致。

**根据需求对齐约束**：

### 2.2 当前实现调研结论

#### 前端实现（`PlatformRulesPage.tsx`）
- 当前是 **L1-L4 规则列表 + 条件编辑器 + 规则试跑面板**
- 用户期望的是 **平台字段清单 + Schema 浏览器 + 格式校验器**
- 前端 Store (`platformRulesStore.ts`) 数据模型围绕 `layer`/`condition_json`/`action`，与 YAML 中的 `content_formats`/`audit_rules`/`content_dna` 完全不匹配

#### 后端实现
- **API**: 13 个端点全部为合规风控设计（`evaluate`、`attribution`、`load-yaml` 等）
- **Service**: `platform_rule_function.py` 核心逻辑是关键词匹配/正则匹配/频率检查
- **ORM**: `PlatformRuleORM` 字段为 `layer`/`condition_json`/`action`/`priority`，无 `content_formats` 字段
- **YAML**: `load-yaml` 端点期望 `rules[]` 结构，与真实 `*_v2024.yml` 的 `content_formats`/`audit_rules` 结构完全不匹配

#### 数据库模型对比

```
当前 PlatformRuleORM 字段：
  id, platform, layer(l1/l2/l3/l4), name, description,
  condition_json(关键词DSL), action(block/warn/suggest),
  priority, enabled, version, effective_from, effective_until,
  applicable_lifecycle, created_by, updated_by, tenant_id

用户期望的数据结构（来自 xiaohongshu_v2024.yml）：
  platform_id: "xiaohongshu"
  content_dna: [{element, value, validation_skill}]
  audit_rules: [{category, forbidden_terms, audit_skill}]
  content_formats:
    图文:
      pic_num: {max, min, default}
      title: {max_chars, min_chars, default}
      content: {max_chars, min_chars, default}
      tags: {max_count}
      pic_size: {max}
      pic_format: {supported}
      pic_resolution: {ratio, min}
    视频:
      title: {max_chars, min_chars, default}
      video_size: {max}
      video_duration: {max, default}
      video_format: {recommended}
      video_resolution: {recommended}
```

      min_chars?: number               # 最小字符数

## 第八章：可选修改方案

### 方案 A：完全重构（推荐用户选择）

**思路**：将「平台规则库」彻底从「合规风控引擎」重构为「平台 API 发布格式规范库」。

**实施步骤**：
1. 新增 ORM 模型：`PlatformSchemaORM`、`PlatformContentFormatORM`、`PlatformFieldConstraintORM`
2. 新增 Service：`platform_schema_service.py` — YAML 解析 + Schema 校验
3. 重写 API：`GET /platform-rules/schemas`、 `GET /platform-rules/{platform_id}/formats`、 `POST /platform-rules/validate-format`
4. 重写前端：平台 Schema 浏览器 + 格式校验器
5. 废弃或迁移旧版 L1-L4 合规规则到 `ComplianceGuard` 模块

**优点**：完全符合用户定义，数据结构清晰，Publisher 可直接查询格式要求组装请求。
**缺点**：工程量大（预估 3-5 天），旧数据迁移复杂，可能影响 Workflow 中的预检逻辑。

### 方案 B：双轨并行（折中方案）

**思路**：保留现有 L1-L4 合规规则引擎，新增「平台 API 格式规范」作为独立模块。

**实施步骤**：
1. 保留现有 `platform_rules` 表和 API（合规风控不变）
2. 新增模块：`api/platform_schemas.py` + `services/platform_schema_service.py` + `models/platform_schema_orm.py`
3. 前端新增子页面：`/platform-rules/schemas`（在现有平台规则库页面下增加 Tab 切换）
4. Publisher 调用新 API 获取格式约束

**优点**：不影响现有合规风控功能，风险低，可渐进实施。
**缺点**：两个概念同名「平台规则库」，可能导致用户混淆。需要明确区分「合规规则」和「发布格式」。

### 方案 C：最小改动（快速落地）

**思路**：在现有 `PlatformRulesPage` 中增加一个「平台格式规范」Tab，后端新增简单的 YAML 读取和展示 API。

**实施步骤**：
1. 后端新增：`GET /platform-rules/yaml-platforms-v2` — 正确解析 `*_v2024.yml` 结构
2. 前端新增：在现有页面增加「格式规范」Tab，展示 YAML 中的 `content_formats`
3. 暂不做 Schema 校验和 Publisher 集成

**优点**：半天即可实现，用户可立即看到平台格式数据。
**缺点**：仅为展示，无法做格式校验，Publisher 无法调用。
