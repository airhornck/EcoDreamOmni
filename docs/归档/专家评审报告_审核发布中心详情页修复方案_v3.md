## 一、问题诊断：三方专家联合评审

### 1.1 用户原始问题陈述

| # | 问题描述 | 严重程度 |
|---|---------|---------|
| 1 | 打开「审核详情」时，无法看到「内容预览」的详情，即真正生成的文章、封面、标题等内容 | 🔴 Critical |

### 1.2 架构设计专家组 — 根因诊断

#### 诊断 1：数据模型与内容真源断裂

**证据链**：

```
// 后端 review_publish.py 第188行
t.prompt_variables.get("content_preview", "")  ← 返回纯字符串

// 真源数据 demo_output/real_flow.json
generated: {
  title: "🔥40℃高温预警！猫咪中暑急救指南，铲屎官必看！",
  body: "喵~各位铲屎官们！最近这天气热得...",
  tags: ["猫咪防暑", "夏季养猫", ...],
  cover_image_url: "..."
}
```

**根因**：`prompt_variables.content_preview` 被存储为**纯文本字符串**，丢失了 `title/body/tags/cover_image_url` 的结构化信息。审核发布中心详情 API 直接返回这个字符串，前端只能做 `whitespace-pre-wrap` 的原样展示，运营人员无法快速辨识「标题是什么」、「标签有哪些」、「封面长什么样」。

**影响**：
- 运营审核时需要从一团文字中人工提取标题和正文，效率极低
- 封面图即使存在，也只是被压缩到 `h-40` 的小图展示，无法判断视觉合规性
- 标签无法独立展示，无法快速核对是否含敏感词或商业标记

#### 诊断 2：「选题报告」未注入工作流上下文

**证据链**：

```
// review_publish.py ReviewDetailResponse schema（第55-73行）
class ReviewDetailResponse(BaseModel):
    task_id: str
    ...
    agent_summary: str
    compliance_result: Dict[str, Any]
    prediction_result: Dict[str, Any]   ← 互动预演结果
    quality_score: Dict[str, Any]
    cover_image_url: Optional[str]
    # ❌ 缺少 topic_report: Dict[str, Any]
```

**根因**：工作流执行时，TrendScout 节点产出的选题报告（`topics` 数组 + 选中选题的详细分析）**未写入** `task.prompt_variables`。审核发布中心详情接口自然也无法返回。

**影响**：运营审核时看不到「这篇内容为什么选这个选题」、「选题的预估互动量是多少」、「竞品分析结论是什么」，丢失了审核决策的重要上下文。

#### 诊断 3：「互动预演」数据结构存在但展示缺失

**证据链**：

```
// 前端 ReviewPublishCenterPage.tsx 第347-351行
{Object.keys(detailData.compliance_result || {}).length > 0 && (
  <div className="p-2 rounded bg-success/10 text-success">
    <strong>合规：</strong>{JSON.stringify(detailData.compliance_result)}
  </div>
)}
// ❌ prediction_result 根本没有被渲染
```

**根因**：前端代码中只渲染了 `compliance_result` 和 `quality_score`，`prediction_result`（互动预演结果）虽然在 schema 中存在，但在 JSX 中被遗漏。且即使渲染，也是 `JSON.stringify` 的粗暴展示。

**影响**：运营无法看到 PoolPredictor 的预估互动区间（点赞/评论/收藏），无法据此判断内容是否值得发布。

### 1.3 UE专家组 — 交互体验诊断

#### 诊断 4：480px 右侧 Drawer 不适合「深度审阅」场景

**场景分析**：

审核发布中心的用户（运营人员）核心任务是：
1. **仔细阅读**生成的正文（通常 300-800 字）
2. **查看**封面图的视觉合规性（是否含未授权素材、是否比例正确）
3. **核对**标签是否完整、是否含商业标记
4. **审阅**Agent摘要（合规结果、互动预演区间、质量分）
5. **查看**选题报告（为什么选这个方向）
6. **确认**发布策略（立即/定时/循环）

**右侧 Drawer 的问题**：

| 问题 | 说明 | 运营效率影响 |
|------|------|------------|
| 视口过窄 | 480px ≈ 手机宽度，正文换行频繁，长段落阅读困难 | 阅读速度降低 40% |
| 垂直堆叠 | 所有模块（预览+摘要+历史+确认）垂直排列，滚动距离长 | 信息获取效率低 |
| 封面图压缩 | `h-40` 仅 160px 高，无法判断视觉细节 | 需额外打开图片检查 |
| 无对比空间 | 无法同时看到「内容预览」和「Agent摘要」的完整信息 | 需反复上下滚动 |
| 发布确认拥挤 | 定时发布 + Cron表达式 + 预检摘要挤在 480px 内 | 误操作风险增加 |

**专家结论**：审核详情不是「快速查看」场景，而是「深度审阅 + 决策确认」场景。右侧 Drawer 适合前者，**完全不适合后者**。

#### 诊断 5：信息层级混乱 — 缺少「内容预览」的结构化展示

当前 Drawer 中的「内容预览」区域：

```tsx
<Card>
  <CardHeader>内容预览</CardHeader>
  <CardContent>
    {detailData.cover_image_url && (
      <img src={detailData.cover_image_url} className="w-full h-40 object-cover rounded-lg" />
    )}
    <div className="text-sm whitespace-pre-wrap">{detailData.content_preview}</div>
  </CardContent>
</Card>
```

**问题**：
- 标题与正文混在一起，没有视觉层级
- 标签不可见（从 `flow_data.json` 看标签是重要的平台元数据）
- 平台标识不可见（运营需确认内容是否匹配目标平台风格）
- 没有「平台模拟预览」（如小红书的卡片样式预览）

### 1.4 UI专家组 — 视觉与布局诊断

#### 诊断 6：当前配色与信息密度不足以承载六类信息模块

需要展示的信息模块（按 PRD 真源）：

1. **选题报告**（趋势报告）— 结构化报告，含多个话题的对比分析
2. **内容预览** — 图文混合，需模拟平台真实展示效果
3. **Agent 摘要** — 含合规结果（多层级）、互动预演（数据卡片）、质量分（进度条）
4. **审核历史** — 时间线样式，多记录
5. **发布确认** — 表单区域，含单选/日期输入/文本输入/快捷按钮
6. **状态提示** — 驳回/打回/通过的差异化视觉反馈

**480px Drawer 的信息密度极限**：

```
480px - 32px padding*2 = 416px 可用宽度
- 内容预览正文：每行约 28 个汉字（16px 字体）
- 一个标签页需要 6 个模块垂直堆叠
- 预估滚动高度：1800px+
```

**UI专家结论**：六类模块在 480px 内堆叠，无论怎么压缩，都会导致：
- 单个模块的 padding/margin 被迫缩小，视觉层级丢失
- 发布确认表单中的日期选择器和 Cron 快捷按钮排列混乱
- 运营在长时间滚动后产生「决策疲劳」

## 二、修复方案：三方联合设计

### 2.1 信息架构重设计（架构专家组）

#### 方案：「审核详情」升级为独立全宽页面（或 80% 宽模态框）

**交互模式对比**：

| 维度 | 当前方案：右侧 Drawer 480px | 推荐方案：全宽详情页 / 80% 模态 |
|------|---------------------------|-------------------------------|
| 适用场景 | 快速查看、轻量编辑 | 深度审阅、复杂决策 |
| 视口宽度 | 480px | 100% 页面宽 或 80% 视口宽（~1200px） |
| 信息承载量 | ≤3 个模块 | ≥6 个模块 + 侧边栏 |
| 阅读体验 | 差（频繁换行） | 良好（接近平台真实宽度） |
| 决策效率 | 低（需反复滚动） | 高（左右分栏，一目了然） |
| 实现成本 | 已存在 | 新增路由 + 组件拆分 |

**推荐决策**：

> **UE专家组**：审核发布中心列表页点击任务 → **跳转到独立详情页** `/review-publish-center/:taskId`，而非打开 Drawer。列表页保留「快捷操作」（通过/驳回/打回/确认发布）在卡片上，点击卡片标题/区域进入详情页深度审阅。
>
> **架构专家组**：同意。但需保留列表页 URL 参数 `?task_id=xxx` 的自动展开能力（向后兼容），同时新增独立路由。
>
> **UI专家组**：独立详情页可采用「左侧内容预览 + 右侧 Agent 摘要与决策」的双栏布局，内容区宽度接近小红书笔记的真实展示宽度（~750px），运营审核时有「真实感」。

### 2.2 数据模型修复（架构专家组）

#### 修复 1：结构化内容预览

**当前问题**：`content_preview` 是纯字符串

**修复方案**：工作流执行时将内容以结构化形式写入 `prompt_variables`：

```python
# 工作流 ContentForge 节点完成时
generated = {
    "title": "...",
    "body": "...",
    "tags": ["..."],
    "cover_image_url": "...",
    "platform": "xhs",
    "content_type": "note",
}

# 写入 task.prompt_variables 时同时保存结构化和字符串版本
await task_hub.update_task(
    db, task_id,
    prompt_variables={
        **existing_vars,
        "generated_content": generated,           # 新增：结构化内容
        "content_preview": generated["body"][:200], # 保留：列表展示用摘要
        "draft_id": draft_id,
    }
)
```

**后端 API Schema 变更**：

```python
# review_publish.py
class ReviewDetailResponse(BaseModel):
    task_id: str
    task_name: str
    platform: str
    status: str
    
    # 内容预览 — 结构化升级
    content_preview: str                    # 保留：兼容旧数据 + 列表摘要
    generated_content: Optional[Dict[str, Any]] = None  # 新增：结构化内容
    # 包含 { title, body, tags, cover_image_url, platform, content_type }
    
    # Agent 摘要
    agent_summary: str
    compliance_result: Dict[str, Any]
    prediction_result: Dict[str, Any]       # 互动预演结果
    quality_score: Dict[str, Any]
    injection_context: Dict[str, Any]
    
    # 选题报告 — 新增
    topic_report: Optional[Dict[str, Any]] = None
    # 包含 { selected_topic, topics[], report_id, estimated_engagement }
    
    cover_image_url: Optional[str]          # 冗余保留，便于快速访问
    review_history: List[Dict[str, Any]]
    risk_level: str
    can_publish: bool
    account_id: str
    account_name: str
    draft_id: Optional[str]
```

#### 修复 2：选题报告注入工作流

**当前问题**：TrendScout 产出的选题报告未关联到 Task

**修复方案**：工作流 Phase 1（选题决策）完成后，将选题报告写入 task：

```python
# workflow_engine.py / TrendScout 节点执行后
report = await trend_scout_service.create_trend_report_v2(
    query=topic_query,
    persona_id=task.persona_id,
    platform=task.platform,
)

# 将选题报告关联到 task
await task_hub.update_task(
    db, task.id,
    prompt_variables={
        **task.prompt_variables,
        "topic_report": {
            "report_id": report.id,
            "selected_topic": report.selected_topic,
            "topics": [t.model_dump() for t in report.topics],
            "estimated_engagement": report.estimated_engagement,
            "5a_stage": report.methodology_stage,
            "audience_fit_score": report.audience_fit_score,
        }
    }
)
```

#### 修复 3：互动预演结果规范化

**当前问题**：`prediction_result` 可能为空或格式不一致

**修复方案**：PoolPredictor 节点执行后，规范化输出：

```python
# workflow_engine.py / PoolPredictor 节点执行后
prediction = {
    "engagement_interval": {
        "likes": {"min": 25, "max": 60, "confidence": "medium"},
        "comments": {"min": 5, "max": 15, "confidence": "medium"},
        "collects": {"min": 8, "max": 20, "confidence": "medium"},
    },
    "viral_probability": 0.35,
    "best_publish_time": "09:00-11:00",
    "note": "内部参考区间，非平台真实数据",
}

await task_hub.update_task(
    db, task.id,
    prompt_variables={
        **task.prompt_variables,
        "prediction_result": prediction,
    }
)
```

### 2.3 UI 重设计方案（UI专家组）

#### 方案 A：独立详情页（推荐）

**路由**：`/review-publish-center/:taskId`

**布局**：左右双栏，左侧 65% 内容审阅，右侧 35% 决策辅助

```tsx
// ReviewPublishDetailPage.tsx 伪代码结构
export function ReviewPublishDetailPage() {
  const { taskId } = useParams()

      {/* 顶部导航栏 */}
      <DetailHeader task={detail} onBack={() => navigate('/review-publish-center')} />
      
      {/* 双栏主体 */}
      <div className="flex-1 flex overflow-hidden">
        {/* 左侧：内容审阅区 */}
        <div className="flex-1 overflow-y-auto p-6 bg-secondary/20">

          <TopicReportPanel report={detail?.topic_report} />
          <MarkdownBody body={detail?.generated_content?.body} />
        </div>
        
        {/* 右侧：决策辅助区 */}
```

**组件设计规范**：

| 组件 | 位置 | 设计要点 |
|------|------|---------|
| `PlatformPreviewCard` | 左侧顶部 | 模拟小红书/抖音真实卡片样式，含封面图、标题、正文前3行、标签行、互动数据（模拟） |
| `TopicReportPanel` | 左侧中部 | 可折叠面板，默认折叠。展开后展示选题列表表格（话题/预估互动/5A阶段/选中状态） |
| `MarkdownBody` | 左侧下部 | 完整正文渲染，支持 Markdown 语法高亮、合规违规词汇高亮（复用 ContentForge 的 `highlightCompliance`） |

| `ReviewHistoryTimeline` | 右侧中部 | 时间线样式，审核人/决策/原因/时间 |
| `PublishConfirmPanel` | 右侧底部 | 发布方式单选 + 日期选择 + Cron 配置 + 快捷模板 + 预检摘要 + 确认按钮 |

### 2.4 UE 交互流程优化（UE专家组）

#### 优化 1：列表页保留快捷操作，减少不必要的详情页跳转

运营人员 80% 的操作是「审核通过 + 确认发布」，不需要看详情。列表页卡片上直接提供快捷操作：

```
审核中 标签页：
┌────────────────────────────────────────────────────────┐
│ 任务名称                    [✅通过] [🔄打回] [❌驳回]  │
│ 小红书 · 账号A · 2026-05-25 14:32                       │
│ 内容预览前 120 字...                                    │
└────────────────────────────────────────────────────────┘

已通过 标签页：
┌────────────────────────────────────────────────────────┐

└────────────────────────────────────────────────────────┘
```

#### 优化 2：详情页提供「平台模拟预览」

运营审核时最关心的问题：「这篇内容在小红书上看起来是什么样的？」

左侧 `PlatformPreviewCard` 按平台渲染：

**小红书笔记卡片**：
- 16:9 封面图（点击可放大）
- 标题（两行截断）
- 正文前 4 行（带省略号）
- 底部：作者头像 + 昵称 + 点赞/收藏图标（模拟）
- 右下角：发布时间（模拟）

**抖音图文卡片**：
- 9:16 封面图
- 底部叠加标题文字
- 右侧：点赞/评论/分享图标列（模拟）

#### 优化 3：选题报告默认折叠，避免干扰核心审阅

运营人员首次进入详情页时，最关心的是「内容本身好不好」。选题报告是「背景信息」，默认折叠，点击展开。
