## 一、方案概述

### 1.1 改造目标

将当前分散的「TaskHub → ContentForge → HumanInLoop → Publisher」独立页面，改造为**以任务为单一流转载体**的串联流程，运营从「创建任务」到「内容发布」全程在同一上下文中完成，无需跳转多个页面。

### 1.2 改造范围

| 优先级 | 模块 | 改造内容 | 预估工作量 |
|--------|------|----------|-----------|
| **P0** | TaskHub 新建任务抽屉 | 增强配置能力，增加平台过滤、注入摘要 | 2d |
| **P0** | 任务执行详情页（新增） | 系统执行进度监控 + 干预入口 | 3d |
| **P0** | ContentForge 任务上下文模式 | 从独立生成改为任务详情预览/干预 | 3d |
| **P0** | HumanInLoop 增强 | 增加内容预览、Agent摘要、发布确认 | 3d |
| **P1** | 前端路由串联 | 从 TaskHub → 执行详情 → 审核台 → 发布的导航流 | 1d |
| **P1** | 后端 Task 模型扩展 | 增加 story_id/series_id 关联、执行上下文 | 1d |

**总预估：约 13 人天（不含测试）**

### 1.3 核心原则

1. **复用优先**：最大化利用现有页面和 API，不新建大量页面
2. **系统能力最大化**：Harness ReAct + Workflow Engine 串行 Pipeline + 多 Agent 协作已有完整能力，改造重点是「调用链串联」而非「重写 Agent」
3. **技术稳定性**：基于 490+ 后端测试 + 50+ 前端测试的基线，所有改动必须有测试保护
4. **向后兼容**：ContentForge 独立生成模式保留，新增「任务上下文模式」为可选入口

@router.get("/tasks/{task_id}/execution-context", response_model=ExecutionContextResponse)
def get_execution_context(task_id: str):
    """聚合任务执行所需的全部上下文，供前端展示"""
    task = task_hub.get_task(task_id)
    if not task:
        raise HTTPException(status_code=404, detail="Task not found")
    
    # 聚合信息
    context = {
        "task": _to_task_response(task),
        "brand_knowledge_injection": _get_brand_knowledge_summary(task.prompt_variables),
        "timeline_recommendation": _get_timeline_recommendation(task.platform),
        "platform_rule_status": _get_platform_rule_pre_check(task.account_id, task.platform),
        "persona_story_context": _get_persona_story_context(task.persona_story_id, task.current_node_id),
        "content_series_context": _get_content_series_context(task.content_series_id),
    }
    return context
```

**技术说明**：

- 不修改任何基础功能数据，仅做聚合查询

# 新增接口：GET /task-hub/tasks/{task_id}/progress
@router.get("/tasks/{task_id}/progress")
def get_task_progress(task_id: str):
    """聚合任务在各 Phase 的执行进度"""
    task = task_hub.get_task(task_id)
    if not task:
        raise HTTPException(status_code=404)
    
    # 从 WorkflowExecution 获取节点执行记录
    execution = workflow_engine.get_execution_by_task_id(task_id)
    node_executions = []
    if execution:
        node_executions = workflow_engine.get_node_executions(execution.id)
    
    # 构建 Phase 映射（Workflow Template 节点 → Phase）
    template = workflow_engine.get_template(task.workflow_template_id)
    phases = []
    for idx, node in enumerate(template.nodes):
        phase = _map_node_to_phase(node)
        node_exec = next((ne for ne in node_executions if ne.node_index == idx), None)
        phases.append({
            "phase_index": idx,
            "phase_name": phase.name,
            "agent_id": node.agent_id,
            "status": node_exec.status.value if node_exec else "PENDING",
            "started_at": node_exec.started_at if node_exec else None,
            "ended_at": node_exec.ended_at if node_exec else None,
            "duration_ms": node_exec.duration_ms if node_exec else None,
            "output_summary": _summarize_output(node_exec.output_context) if node_exec else None,
        })
    
    return {
        "task_id": task_id,
        "current_phase_index": task.current_node_index,
        "overall_status": task.status.value,
        "phases": phases,
    }
```

## 五、多Agent调用能力最大化利用

### 5.1 已有 Agent 协作链

当前系统已有 **标准内容生产工作流模板**（8节点）：

```
Node 0: TrendScout      (AGENT)    → 热点侦察/选题洞察
Node 1: MarketingMethodology (AGENT) → 结构分析/AIPL模板匹配
Node 2: ContentForge    (AGENT)    → 框架生成
Node 3: ContentForge    (AGENT)    → 正文生成
Node 4: ComplianceGuard (AGENT)    → 四层合规审核
Node 5: PoolPredictor   (AGENT)    → 互动预演（FAIL=CONTINUE，非阻断）
Node 6: HumanApproval   (HUMAN)    → 人工审核（24h超时）
Node 7: Publisher       (AGENT)    → 定时发布
```

### 5.2 Harness 能力集成点

| Harness 能力 | 当前状态 | 改造中的利用方式 |
|--------------|----------|-----------------|
| **ReAct 循环** (Think→Act→Observe→Verify) | 已实现 | 每个 AGENT 节点内部调用 Harness Session，ReAct 步骤记录到 execution context |
| **Planning 引擎** (任务分解+todo执行) | 已实现 | ContentForge 生成前自动分解为：analyze → outline → draft → review |
| **Context Manager** (窗口/压缩/pin) | 已实现 | 跨节点上下文传递，人工审核时 pin 关键信息 |
| **State Graph** (检查点/回滚) | 已实现 | 打回修改时回退到指定节点，保留已执行节点的上下文 |
| **Tool Registry** | 已实现 | ComplianceGuard、PoolPredictor 等通过 Tool Registry 调用 |

### 5.3 改造后的 Agent 调用流

```
运营创建任务 → TaskHub.create_task()
  → 状态 = DRAFT
  → 运营点击「启动」→ transition(QUEUED)
  
Workflow Engine 自动执行：
  → Node 0: Harness.create_session(goal="热点侦察: {topic}")
    → ReAct: Think → 调用 trend-scout tool → Observe → Verify
    → 输出：选题报告 → 写入 execution context
  
  → Node 1: Harness.create_session(goal="结构分析: {topic}")
    → ReAct: Think → 调用 marketing-methodology tool → Observe
    → 输出：内容结构模板 → 写入 context
  
  → Node 2-3: Harness.create_session(goal="内容生成: {topic}")
    → Planning: 分解为 outline → body
    → ReAct 执行每个 todo
    → 输出：标题+正文+封面 → 写入 context + 创建 ContentDraft
  
  → Node 4: Harness.create_session(goal="合规审核")
    → ReAct: 调用 compliance-check tool (L1→L2→L3→L4)
    → 输出：合规报告 → 写入 context
  
  → Node 5: Harness.create_session(goal="互动预演")
    → 调用 engagement-predict tool
    → 输出：预测区间 → 写入 context
  
  → Node 6: 状态 = HUMAN_WAIT
    → 前端 HumanInLoop 加载审核详情
    → 运营决策：
      - 通过 → transition(COMPLETED) → Node 7 执行
      - 驳回 → transition(FAILED) → 进 DLQ
      - 打回 → transition(RUNNING) → 回退到指定节点
  
  → Node 7: Publisher
    → 调用 publish-schedule tool
    → 输出：发布结果
```

**关键设计**：改造不涉及重写任何 Agent，仅通过 **Workflow Engine 的节点编排** 和 **Harness 的 Session 管理** 将已有能力串联。
