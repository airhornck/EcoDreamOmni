# 问题分析报告：Task Hub 创建任务后列表不可见

> **问题现象**：在 `http://localhost:5173/task-hub/create` 中成功创建任务后，跳转回 `http://localhost:5173/task-hub` 的"任务列表"中，未出现新创建的任务。
>
> **报告日期**：2026-05-25
> **分析方式**：组织 4 个专家组（前端页面层、前端状态管理层、后端 API 层、路由与数据流层）并行代码审查

---

## 1. 问题摘要

经专家组联合分析，该问题的**直接根因已定位**：前端 `taskHubStore.fetchTasks()` 对后端 API 响应格式的解析假设错误，将后端返回的 JSON 数组当作对象解析，导致任务列表永远为空数组。此外，后端 `list_tasks` 的 `parent_task_id` 过滤条件存在边界 bug，在特定查询参数下会进一步加剧"列表为空"的现象。

| 严重程度 | 问题 | 位置 | 影响 |
|---------|------|------|------|
| 🔴 Critical | 前端 fetchTasks 响应格式解析错误 | `taskHubStore.ts:143-144` | 任务列表**永远为空**，与是否创建任务无关 |
| 🔴 High | 后端 parent_task_id 空字符串过滤陷阱 | `services/task_hub.py:342` | 若前端携带 `parent_task_id=` 参数，新建任务被过滤 |
| 🟡 Medium | 后端富化字段始终为 null | `api/task_hub.py:134-165` | 列表中账号名、人设名等显示空白 |
| 🟡 Medium | DLQ API 路径前后端不匹配 | `taskHubStore.ts:162-172` / `api/task_hub.py` | DLQ Tab 404 报错 |
| 🟡 Medium | Draft → Queued 状态转换非法 | `TaskHubPage.tsx:354-361` | 用户点击"启动"按钮触发 409 CONFLICT |
| 🟡 Low | 列表无排序 | `services/task_hub.py:327-351` | 新任务可能排在列表末尾，不易发现 |
| 🟡 Low | new_series_name 字段被后端忽略 | `CreateTaskRequest` 缺少该字段 | 新建系列名称功能无效 |
| 🟡 Low | 前端测试覆盖缺失 | `TaskHubCreatePage.test.tsx` / `TaskHubPage.test.tsx` | 无创建→列表的集成场景测试 |

## 3. 根因分析（Root Cause Analysis）

### 3.1 致命根因：前端 fetchTasks 响应格式解析错误

#### 代码证据

**后端 API**（`apps/backend/src/api/task_hub.py` 第 230-238 行）：
```python
@router.get("/tasks", response_model=List[TaskResponse])
async def list_tasks(...):
    tasks = await task_hub.list_tasks(db, status, account_id, parent_task_id)
    return [_to_task_response(t) for t in tasks]   # ← 返回的是 JSON 数组 []
```

**前端 Store**（`apps/frontend/src/stores/taskHubStore.ts` 第 138-148 行）：
```ts
fetchTasks: async () => {
    set({ isLoading: true, error: null })
    try {
      const res = await fetch('/api/task-hub/tasks', { headers: authHeaders() })
      if (!res.ok) throw new Error(`获取任务列表失败: ${res.status}`)
      const data = await res.json()
      set({ tasks: data.tasks || [], isLoading: false })   // ← BUG：data 是数组，data.tasks = undefined
    } catch (err) { ... }
  }
```

#### 时序分析

```
TaskHubCreatePage
  └── handleSubmit()
        └── await createTask(payload)           ← POST /api/task-hub/tasks ✅ 成功
              ├── 后端返回 201，任务写入 PostgreSQL
              ├── await get().fetchTasks()       ← 第一次 GET /api/task-hub/tasks
              │     ├── 后端返回 JSON 数组 [...]
              │     ├── data.tasks → undefined
              │     └── set({ tasks: [] })       ← ❌ 列表被清空
              └── return true
        └── navigate('/task-hub')
              └── TaskHubPage 重新挂载
                    └── useEffect(() => fetchTasks())
                          └── 第二次 GET /api/task-hub/tasks
                                ├── 后端返回 JSON 数组 [...]
                                ├── data.tasks → undefined
                                └── set({ tasks: [] })   ← ❌ 再次被清空
```

**结论**：无论后端数据库中有多少条任务记录，`fetchTasks` 始终将 `tasks` 设为空数组。这是"创建任务后在列表看不到"的**最直接、最致命**的根因。

> **旁证**：同一文件中 `fetchWorkflowTemplates` 已存在兼容性处理：
> ```ts
> // Backend returns array directly, not { templates: [...] }
> set({ workflowTemplates: Array.isArray(data) ? data : (data.templates || []) })
> ```
> 这说明项目存在"后端直接返回数组"的历史债务，但 `fetchTasks` 未做相同处理。

### 3.2 高优先级根因：后端 parent_task_id 空字符串过滤陷阱

#### 代码证据

**后端服务层**（`apps/backend/src/services/task_hub.py` 第 327-351 行）：
```python
async def list_tasks(db, status, account_id, parent_task_id):
    query = select(TaskORM)
    if status:
        query = query.where(TaskORM.status == status)
    if account_id:
        query = query.where(TaskORM.account_id == account_id)
    if parent_task_id is not None:               # ← ⚠️ 关键差异
        query = query.where(TaskORM.parent_task_id == parent_task_id)
    ...
```

#### 问题分析

三个过滤参数的判断逻辑**不一致**：

| 参数 | 判断方式 | 空字符串 `""` 的行为 |
|------|---------|-------------------|
| `status` | `if status:` | ❌ 不加 WHERE（正确） |
| `account_id` | `if account_id:` | ❌ 不加 WHERE（正确） |
| `parent_task_id` | `if parent_task_id is not None:` | ✅ **加 `WHERE parent_task_id = ''`**（**Bug**）|

新建任务的 `parent_task_id` 在数据库中是 `NULL`。在 SQL 中，`NULL = ''` 的结果是 `NULL`（假值），因此这些任务被过滤掉了。

**触发场景**：如果前端在 `fetchTasks` 中不慎携带 `?parent_task_id=` 参数（例如通过某个 filter 状态），即使修复了响应格式问题，列表仍然会为空。

### 3.3 其他关联问题

#### 3.3.1 后端富化字段始终为 null

`_to_task_response` 未 JOIN 关联表，导致 `account_name`、`persona_name`、`story_name`、`workflow_template_name` 等字段永远返回 `None`。前端列表中这些值显示为空白或 `null`。

#### 3.3.2 DLQ API 路径不匹配

前端调用 `/api/task-hub/dlq`，但后端 `task_hub.py` 中完全没有 `/dlq` 路由。DLQ 实际存在于 `cron_hub.py`，路径为 `/cron-hub/dlq`。

#### 3.3.3 Draft → Queued 状态转换非法

前端"启动"按钮将任务状态从 `draft` 直接改为 `queued`，但后端状态机只允许 `DRAFT → CONFIGURING` 或 `DRAFT → CANCELLED`。后端会返回 409 CONFLICT。

#### 3.3.4 列表无排序

`list_tasks` 查询缺少 `ORDER BY created_at DESC`，新创建的任务可能排在列表末尾，增加"看不到"的感知。

## 5. 修复建议（按优先级排序）

### 5.1 立即修复（P0）

**修复 fetchTasks 响应解析**（`apps/frontend/src/stores/taskHubStore.ts` 第 143-144 行）：

```ts
// 修改前
const data = await res.json()
set({ tasks: data.tasks || [], isLoading: false })

// 修改后
const data = await res.json()
set({ tasks: Array.isArray(data) ? data : (data.tasks || []), isLoading: false })
```

> 建议同步检查 `fetchContentSeries` 和 `fetchDLQ` 是否存在相同的解析问题。

**修复 parent_task_id 过滤条件**（`apps/backend/src/services/task_hub.py` 第 342 行）：

```python
# 修改前
if parent_task_id is not None:

# 修改后
if parent_task_id:
```

### 5.2 短期修复（P1）

1. **补充列表排序**：在 `list_tasks` 中添加 `ORDER BY created_at DESC`
2. **修正 Draft 启动按钮**：前端将 `updateTaskStatus(id, 'queued')` 改为调用 `/configure` 或 `/queue` 快捷接口
3. **修正 DLQ API 路径**：将前端 DLQ 调用路径从 `/api/task-hub/dlq` 改为 `/api/cron-hub/dlq`，或在 `task_hub.py` 中增加 DLQ 路由代理

### 5.3 中期修复（P2）

1. **后端富化字段**：在 `list_tasks` 或 `_to_task_response` 中加入 JOIN 查询，填充关联实体名称
2. **处理 new_series_name**：在 `CreateTaskRequest` 中添加 `new_series_name` 字段，并在服务层实现"不存在则创建系列"的逻辑
3. **补充测试覆盖**：
   - `test_list_tasks_no_filter_after_create`：验证创建任务后无过滤列表能正确返回
   - `test_list_tasks_empty_parent_task_id`：验证 `parent_task_id=""` 不会过滤掉数据
   - 前端测试：覆盖"创建任务 → 跳转列表 → 断言新任务存在"的完整链路

### 5.4 长期优化（P3）

1. **统一 API 响应格式**：明确前后端契约，统一列表接口返回 `{ items: [], total: n }` 格式，避免"数组 vs 对象"的歧义
2. **消除内存缓存 `_task_db`**：当前 `list_tasks` 依赖内存缓存，在多进程/多副本部署下会导致数据不一致
3. **添加租户隔离（tenant_id）**：`TaskORM` 有 `tenant_id` 字段，但 `Task` dataclass 和 `create_task` 均未处理
