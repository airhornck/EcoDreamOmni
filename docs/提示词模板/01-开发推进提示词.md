# 提示词模板：开发推进（按 Phase 执行）

> **使用场景**：每次进入新 Phase、每日开发开始时、任务卡壳时调用
> **输入**：当前 Phase 编号 + 具体任务编号（如 P1-1）
> **输出**：该任务的完整上下文 + 执行步骤 + 验收标准 + 关联文档索引
> **关联文档**：`docs/v4.0_开发Checklist.md`、`docs/架构设计/07-现有代码基线改造设计.md`

---

## 提示词正文（复制使用）

```
你正在推进 EcoDream Omni v4.0 的开发任务。

【当前任务上下文】
- Phase: {phase_number}（如 Phase 0 / Phase 1 / ... / Phase 7）
- 任务编号: {task_id}（如 P0-1 / P1-2 / P2-3）
- 任务名称: {task_name}
- 开发者: {developer_name}
- 开始日期: {start_date}

【你必须遵循的文档体系】（按优先级读取）
1. 主推进文档: docs/架构设计/07-现有代码基线改造设计.md
   → 读取该 Phase 的任务清单、技术方案、工作量估算
2. 执行清单: docs/v4.0_开发Checklist.md
   → 读取对应 Phase 的逐任务 Checklist，逐项执行并打勾
3. 工程纪律: docs/工程纪律_v4.0.md
   → 检查是否违反 6 条架构红线（Agent禁DB/EventBus优先/MCP预留/租户隔离/LLMHub路由/Prompt六层）
4. 设计约束: docs/契约与数据/
   → API变更查 01-API接口契约.md，Schema变更查 02-数据库ER图.md，状态变更查 03-核心业务状态流转.md
5. 需求真源: EcoDream_Omni_PRD_v4_AI_Native_Architecture.md
   → 需求争议时回来对真源

【你的执行流程】
步骤 1 - 读取上下文
  读取上述 5 份文档中与本任务相关的章节，提取：
  - 技术方案（改造设计文档）
  - 检查清单（Checklist）
  - 架构红线（工程纪律）
  - 契约约束（契约与数据）
  - 需求依据（PRD 章节）

步骤 2 - 设计先行（契约优先）
  如果本任务涉及以下任何一项，必须先更新契约文档，再写代码：
  □ 新增/修改 API 端点 → 更新 docs/契约与数据/01-API接口契约.md
  □ 新增/修改数据库表或字段 → 更新 docs/契约与数据/02-数据库ER图.md
  □ 新增/修改状态或状态转换 → 更新 docs/契约与数据/03-核心业务状态流转.md
  □ 新增/修改枚举值 → 更新 01-API接口契约.md §三
  □ 新增/修改错误码 → 更新 01-API接口契约.md §二

步骤 3 - 开发实现
  按 Red-Green TDD 流程执行：
  1. 写测试（验证新行为 / 回归旧行为）
  2. 运行测试，确认失败（Red）
  3. 写实现代码
  4. 运行测试，确认通过（Green）
  5. 重构优化

步骤 4 - 质量门禁（必须全部通过）
  □ 测试覆盖率 ≥ 80%（pytest --cov-fail-under=80）
  □ 后端 Lint 0 errors（ruff check src/ tests/）
  □ 后端类型检查 0 errors（mypy src/）
  □ 前端 Lint 0 errors（npm run lint）
  □ 前端类型检查 0 errors（npx tsc --noEmit）
  □ 前端测试通过（npm run test -- --run）
  □ Docker Compose Build 通过（docker compose build）
  □ Alembic 迁移可升级/降级（alembic upgrade head && alembic downgrade -1）
  □ 数据词典已同步更新（docs/数据词典_v4.0/ 对应文件）
  □ 契约文档已同步更新（docs/契约与数据/ 对应文件）
  □ 工程纪律红线未违反（对照 docs/工程纪律_v4.0.md 检查）
  □ PRD 偏差报告已检查（无新增偏差）

步骤 5 - 文档连锁更新
  代码完成后，按以下顺序更新文档：
  1. 契约与数据/ → 确认契约文档与代码一致
  2. 数据词典_v4.0/ → 更新 ORM/API/Service/Store 映射
  3. 文档总纲_v4.0.md → 如有新增文档，更新 §4 分类清单
  4. PRD偏差报告_v4.0.md → 登记本次实现的 PRD 章节
  5. 变更记录_v4.0/YYYY-MM-DD/ → 登记变更摘要

步骤 6 - 任务验收
  对照 Checklist 中本任务的验收标准，逐项确认：
  □ 功能已实现并测试通过
  □ 契约文档与代码一致
  □ 12 项质量门禁全部通过
  □ 文档连锁更新完成

【禁止事项】
- ❌ 不允许在契约文档更新前直接写代码（违反契约先行原则）
- ❌ 不允许 Agent 直接操作数据库（必须通过 Function API）
- ❌ 不允许 Agent 间直接调用（必须通过 Event Bus）
- ❌ 不允许直接实例化 LLM 客户端（必须通过 LLM Hub）
- ❌ 不允许数据库查询不带 tenant_id 过滤
- ❌ 不允许六层 Prompt 缺失任何一层
- ❌ 不允许状态转换不经过状态机校验
- ❌ 不允许质量门禁未通过就提交代码

【输出格式】
请按以下结构输出执行报告：

=== 任务执行报告 ===
任务: {task_id} - {task_name}
状态: [进行中 / 已完成 / 阻塞]

--- 上下文摘要 ---
（本任务的技术方案摘要，200字以内）

--- 契约冻结 ---
（如涉及契约变更，列出冻结的契约项）

--- 实现摘要 ---
（新建/修改的文件列表，关键设计决策）

--- 测试覆盖 ---
（测试文件列表，覆盖率数据）

--- 质量门禁 ---
（12 项逐项结果：通过 / 未通过 / 不适用）

--- 文档更新 ---
（已更新的文档清单及变更摘要）

--- 阻塞项（如有） ---
（需要人工决策的问题，附带建议方案）
```

1. 读取 `07-现有代码基线改造设计.md` §3.3 P0-5

3. 读取 `工程纪律_v4.0.md` 相关红线
4. 按步骤 1~6 执行并输出报告
