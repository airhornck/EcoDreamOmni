# 提示词模板：新增或变更需求（含文档连锁更新）

> **使用场景**：产品提出新需求、架构调整、紧急 Bug 修复需要变更设计时调用
> **输入**：变更类型（新增/修改/删除）+ 变更范围 + 变更描述
> **输出**：影响面分析 + 文档连锁更新清单 + 变更标记
> **核心原则**：任何变更必须先更新文档真源，再更新代码；文档与代码必须同步标记变更状态

---

## 提示词正文（复制使用）

```
你正在处理 EcoDream Omni v4.0 的新增或变更需求。

【变更上下文】
- 变更类型: {change_type}（新增功能 / 需求修改 / 设计变更 / Bug 修复 / 技术债务）
- 变更范围: {change_scope}（API / 数据库 Schema / 状态机 / Agent / Skill / UI / 配置）
- 变更描述: {change_description}
- 提出者: {requester}
- 优先级: {priority}（P0-紧急 / P1-高 / P2-中 / P3-低）
- 提出日期: {request_date}
- 期望完成: {target_date}

【变更影响分析流程】

步骤 1 - 需求对齐（必须执行）
  读取 PRD 真源: EcoDream_Omni_PRD_v4_AI_Native_Architecture.md
  确认以下问题：
  □ 本变更是否在 PRD v4.0 已有定义？（是 → 定位章节；否 → 标记为 PRD 偏差）
  □ 本变更是否与 8 项架构评审结论冲突？
  □ 本变更是否属于改造范围边界内？（参考改造设计文档 §1.3）
  
  如果与 PRD 冲突或属于 PRD 未定义的新需求：
  → 在 PRD偏差报告_v4.0.md 中登记为【新增偏差】
  → 如果影响架构设计，需先走架构评审流程

步骤 2 - 影响面分析（必须输出）
  分析本变更对以下文档的影响（逐项勾选）：

  【契约与数据层】（如涉及则必须更新）
  □ 01-API接口契约.md — 影响项：错误码 / 枚举 / 响应格式 / 限流规则
  □ 02-数据库ER图.md — 影响项：新增表 / 修改字段 / 调整索引 / 外键关系
  □ 03-核心业务状态流转.md — 影响项：新增状态 / 修改转换条件 / 新增状态机

  【数据词典层】（自动同步）
  □ 01-数据库模型层.md — ORM 类定义变更
  □ 02-后端API路由层.md — API 端点变更
  □ 03-后端Service层.md — Service 分类变更
  □ 04-前端Store与路由.md — Store/API 映射变更
  □ 05-模块调用关系矩阵.md — 依赖关系变更
  □ 06-Agent与Skill映射.md — Agent/Skill 绑定变更

  【架构设计层】（如涉及则必须更新）
  □ 00-架构设计总纲.md — 架构概念变更
  □ 01-六层Prompt拼接策略.md — Prompt 层变更
  □ 02-PlatformContentTypeStyle设计.md — 风格层变更
  □ 03-ContentTemplate设计.md — 模板层变更
  □ 04-多模态LLM路由设计.md — 路由策略变更
  □ 05-EventBus设计.md — 事件定义变更
  □ 06-MCPGateway设计.md — MCP 协议变更
  □ 07-现有代码基线改造设计.md — Phase 任务/工作量调整

  【前端设计层】（如涉及则必须更新）
  □ 00-前端设计总纲.md — 页面/路由变更
  □ 01-实验室设计.md — 实验室 功能变更
  □ 02-AICopilot面板设计.md — Copilot 交互变更
  □ 03-页面架构拆解.md — UI 区域/字段变更

  【工程纪律层】（如涉及则必须更新）
  □ 工程纪律_v4.0.md — 新增红线 / 修改门禁

  【项目管理层】（必须更新）
  □ v4.0_开发Checklist.md — 新增/修改检查项
  □ 文档总纲_v4.0.md — 新增文档登记 / 状态变更
  □ PRD偏差报告_v4.0.md — 登记偏差（如适用）

步骤 3 - 文档连锁更新（按依赖顺序执行）

  更新顺序（必须遵守，上游文档优先）：
  
  第一轮：设计层真源（人工维护，不可自动生成）
    1. 契约与数据/ 对应文档 ← 更新设计约束
    2. 架构设计/ 对应文档 ← 更新架构决策
    3. 前端设计/ 对应文档 ← 更新 UI 规范
    4. 工程纪律_v4.0.md ← 更新红线（如适用）

  第二轮：实现层映射（可脚本辅助生成）
    5. 数据词典_v4.0/ 对应文档 ← 更新代码映射

  第三轮：索引与清单（必须人工确认）
    6. 文档总纲_v4.0.md ← 更新 §4 分类清单状态
    7. v4.0_开发Checklist.md ← 新增/修改检查项
    8. PRD偏差报告_v4.0.md ← 登记偏差（如适用）

  第四轮：变更记录（必须执行）
    9. 变更记录_v4.0/YYYY-MM-DD/ ← 创建变更记录文件

步骤 4 - 变更标记规范（必须在所有受影响文档中执行）

  在文档顶部添加变更标记：
  ```markdown
  > **变更状态**: 🔄 变更中 / ✅ 已同步 / ⏸️ 待决策
  > **变更类型**: 新增 / 修改 / 删除
  > **变更日期**: YYYY-MM-DD
  > **变更来源**: 需求变更 / 架构调整 / Bug 修复 / 技术优化
  > **影响文档**: 列出所有受影响的文档路径
  > **PRD 偏差**: 是 / 否（如是，登记编号）
  ```

  在文档内容变更处添加行内标记：
  ```markdown
  <!-- [CHANGED:2026-06-02] 原内容: xxx → 新内容: yyy -->
  新内容...
  <!-- [/CHANGED] -->
  
  <!-- [ADDED:2026-06-02] 新增内容 -->
  新增内容...
  <!-- [/ADDED] -->
  
  <!-- [DEPRECATED:2026-06-02] 将在 Phase X 删除 -->
  旧内容...
  <!-- [/DEPRECATED] -->
  ```

  在文档末尾追加变更记录表：
  ```markdown
  ## 变更记录

  | 日期 | 版本 | 变更内容 | 变更人 | 状态 |
  |------|------|---------|--------|------|
  | 2026-06-02 | v1.1 | 新增 XXX 字段 | {name} | 🔄 变更中 |
  ```

步骤 5 - 代码同步检查清单
  文档更新完成后，检查代码是否同步：
  □ 契约文档中的错误码 ↔ 代码 src/core/errors.py 一致
  □ 契约文档中的表结构 ↔ Alembic 迁移脚本一致
  □ 契约文档中的状态机 ↔ 代码状态转换逻辑一致
  □ 契约文档中的枚举值 ↔ 前后端类型定义一致
  □ 架构设计文档中的决策 ↔ 代码实现一致

【输出格式】
请按以下结构输出变更分析报告：

=== 需求变更分析报告 ===
变更: {change_type} - {change_scope}
优先级: {priority}
状态: [分析中 / 文档已更新 / 代码已同步 / 已完成]

--- PRD 对齐结果 ---
（是否在 PRD 内 / 偏差登记编号 / 是否需要架构评审）

--- 影响面矩阵 ---
（以表格形式列出所有受影响文档及影响程度：高/中/低/无）

--- 文档连锁更新清单 ---
（已更新的文档清单，每行格式：文档路径 → 变更摘要 → 标记状态）

--- 代码同步检查 ---
（5 项检查结果：一致 / 不一致 / 待验证）

--- 风险与建议 ---
（变更引入的风险、回滚方案、建议的 Phase 插入点）

--- 下一步行动 ---
（明确的待办事项列表，责任人，截止日期）
```

## 使用示例

### 示例 1：新增 API 端点

```
【变更上下文】
- 变更类型: 新增功能
- 变更范围: API
- 变更描述: 实验室 页面需要新增 POST /api/v2/lab/parse 接口，用于解析爆款链接
- 提出者: 前端开发者
- 优先级: P1
- 提出日期: 2026-06-03
- 期望完成: 2026-06-05
```

按上述提示词调用后，AI 将：
1. 检查 PRD §5.4.1 是否已有定义 → 确认有，无需登记偏差
2. 分析影响面 → 01-API接口契约.md（新增错误码/枚举）、02-后端API路由层.md、07-改造设计.md（Phase 4 工作量）
3. 按顺序更新所有受影响文档
4. 在文档中添加变更标记和变更记录
5. 输出完整的变更分析报告

### 示例 2：数据库 Schema 变更

```
【变更上下文】
- 变更类型: 设计变更
- 变更范围: 数据库 Schema
- 变更描述: ContentTemplate 表需要新增 source_platform_id 字段，用于记录爆款来源平台
- 提出者: 后端开发者
- 优先级: P0
- 提出日期: 2026-06-03
- 期望完成: 2026-06-04
```

按上述提示词调用后，AI 将：
1. 先更新 `02-数据库ER图.md`（契约先行）
2. 再写 Alembic 迁移脚本
3. 同步更新 `01-数据库模型层.md`
4. 标记所有变更并输出报告

## 变更标记速查表

| 标记 | 含义 | 使用场景 |
|------|------|---------|
| `🔄 变更中` | 文档已更新，代码开发中 | 文档更新完成，代码尚未提交 |
| `✅ 已同步` | 文档与代码已对齐 | 代码已通过 CI，文档已最终确认 |
| `⏸️ 待决策` | 变更方案待评审 | 架构评审中或需求待确认 |
| `📌 已冻结` | 设计已锁定，不再变更 | Phase 验收通过后的基线状态 |
| `⚠️ 已知偏差` | 与 PRD 存在已知偏差 | 已登记在 PRD 偏差报告中 |
