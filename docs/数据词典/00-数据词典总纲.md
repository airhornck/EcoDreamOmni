## 一、总览

本文档目录是 EcoDreamOmni 项目的**数据词典（Data Dictionary）真源**，覆盖从数据库模型到前端 Store 的完整调用链路，用于：

- 🔍 快速定位某个 API 对应的 Service 和 Store
- 🏗️ 评估修改影响范围（变更一个模块会影响哪些上下游）
- 🧪 编写集成测试时参考端点清单
- 📋 新成员 onboarding 时作为架构速查手册

## 四、关键约定

### 4.1 API 路径约定

- 前端请求统一以 `/api/` 开头（由 nginx rewrite 去掉 `/api/` 后转发到后端）
- 后端 Router **不**加 `/api` prefix，仅定义业务路径（如 `/dashboard/overview`）
- **例外**：`agent_orchestra.py` 的 router 未设 prefix，端点为 `/agents`、`/workflows`、`/pipelines`

### 4.2 Service 分层红线

| 层级 | 文件特征 | DB 访问权限 | 典型文件 |
|------|----------|-------------|----------|
| **Function 层** | `*_function.py` | ✅ 直接 `AsyncSession` | `auth_function.py`, `task_function.py` |
| **Agent/Skill 层** | `*.py` (非 function) | ❌ `db: Any` 透传 | `task_hub.py`, `llm_hub.py` |
| **Celery 层** | `celery_tasks*.py` | ✅ `AsyncSessionLocal` | `celery_tasks_function.py` |

### 4.3 前端 Store 命名

- 每个业务模块一个 Store 文件：`{模块名}Store.ts`
- 统一使用 `create<State>()` (Zustand) 定义
- API 路径与后端 Router prefix 对齐（如 `/api/dashboard/*` → `dashboardStore.ts`）

## 六、维护指南

### 新增模块时

1. 在 `02-后端API路由层.md` 中新增 Router 小节
2. 在 `03-后端Service层.md` 中新增 Service 文件小节
3. 在 `01-数据库模型层.md` 中新增 ORM 表（如有）
4. 在 `04-前端Store与路由.md` 中新增 Store / 页面
5. 在 `05-模块调用关系矩阵.md` 中更新依赖关系
6. 在 **本文件** 的「快速查找索引」中补充一行

### 修改端点时

1. 更新 `02-后端API路由层.md` 对应文件小节
2. 检查 `05-模块调用关系矩阵.md` 中的调用链是否失效
3. 检查前端 Store 中的 `fetch()` 路径是否需要同步更新
