## 四、关键架构决策（Architecture Decision Records）

| # | 决策 | 选项 | 选择 | 理由 |
|---|------|------|------|------|
| ADR-001 | DataAnalyst 归属 | A. 保留为 Agent / B. 降级为 Function+Skill | **B** | 数据回流是确定性 ETL，不需要 ReAct 循环 |
| ADR-002 | PlatformContentTypeStyle 关系 | A. 合并入 PlatformFormat / B. 独立概念 | **B** | Format=硬约束（低变更频率），Style=风格指导（高变更频率） |
| ADR-003 | 六层 Prompt 拼接 | A. 单一大 Prompt / B. 六层结构化拼接 | **B** | 便于单独维护、A/B 测试、效果归因 |
| ADR-004 | Event Bus 技术选型 | A. Redis Streams / B. RabbitMQ / C. Kafka | **A (MVP)** | 团队已熟悉 Redis，零新增运维负担；Scale 期迁移 Kafka |
| ADR-005 | MCP Gateway 接入 | A. 直接接入 / B. 内部 Tool Registry / C. 渐进接入 | **C** | MVP 保留现有 Tool Registry，Phase 2 演进为 MCP Gateway |
| ADR-006 | LLM 路由策略 | A. 全局统一模型 / B. 按 Skill 模态路由 / C. 按成本路由 | **B** | Skill 声明 modality_support，LLM Hub 自动匹配支持该模态的模型 |
| ADR-007 | 实验室 定位 | A. 独立页面 / B. 嵌入 Generate 向导 / C. 两者并存 | **A** | 实验室 面向有经验的运营人员，Generate 向导面向新手，互不替代 |

## 五、与 v2.7.x 架构的差异

| 维度 | v2.7.x | v4.0 |
|------|--------|------|
| **Agent 数量** | 11 个常驻（含 DataAnalyst） | 10 个常驻 + Meta-Orchestrator |
| **Skill 体系** | 9 个 L1 Skill，无内容生产级 Skill 清单 | 13 内容生产级 + 9 系统级 = 22 个 |
| **Pipeline 步骤** | 9 步 | 10 步（新增 keyword_inject） |
| **Prompt 策略** | 单一大 Prompt | 六层结构化拼接 |
| **LLM 路由** | 按供应商路由 | 按模态路由（国内首选/海外兜底） |
| **前端页面** | 9 页 | 10 页（新增 实验室） |
| **Event Bus** | MsgHub + Blackboard 双层 | Event Bus 统一承载 + Blackboard 状态管理 |
| **MCP** | 未接入 | Phase 2 评估接入 |
| **编排模式** | 子 Agent 委派 | 四种模式：Pipeline/Swarm/Dynamic/Classifier-Router |
