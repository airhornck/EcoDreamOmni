## 一、改造总体策略

### 1.1 核心决策

**基于现有代码基线 `D:\project\contxt\EcoDreamOmni` 进行改造**，而非从零开发。

| 指标 | 数据 |
|------|------|
| 现有后端核心代码 | ~37,000 行（API 12,587 + Service 19,416 + Model 2,529 + Core/Harness 2,403） |
| 现有前端代码 | ~26,200 行（125 TS/TSX 文件） |
| 可直接复用代码 | ~59,600 行 |
| 需新增/修改代码 | ~25,000 行 |
| 预估总 Prompt Token | ~230 万（改造）vs ~645 万（从零） |

### 1.2 改造原则

```
┌─────────────────────────────────────────────────────────────────┐
│                     改造五原则                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 【渐进式改造】                                               │
│     不一次性重构整个系统，按 Phase 逐个模块改造，每 Phase          │
│     完成后通过 CI 门禁，确保系统始终可运行。                      │
│                                                                 │
│  2. 【Adapter 隔离】                                             │
│     新旧代码通过 Adapter 层隔离，旧模块逐步迁移，不破坏现有        │
│     API 契约。新增模块独立开发，通过 Gateway 接入。               │
│                                                                 │
│  3. 【测试先行】                                                 │
│     每个改造任务遵循 Red-Green TDD：先写测试（验证新行为/         │
│     回归旧行为），再改代码。覆盖率门禁 ≥80%。                    │
│                                                                 │
│  4. 【数据兼容】                                                 │
│     数据库 Schema 变更通过 Alembic 迁移脚本完成，每步迁移          │
│     必须可回滚。不删除旧字段，先标记 deprecated，Phase 2 清理。   │
│                                                                 │
│  5. 【文档连锁】                                                 │
│     每处代码变更必须同步更新：数据词典 + 架构设计文档 +            │
│     PRD 偏差报告。变更完成后在 `变更记录_v4.0/` 登记。            │
│                                                                 │
│  6. 【契约先行】                                                 │
│     任何 API 变更、Schema 变更、状态变更，必须先更新              │
│     `docs/契约与数据/` 中对应的契约文档，再写代码。                │
│     Alembic 迁移脚本必须与 `02-数据库ER图.md` 一致。              │
│     OpenAPI Schema 必须与 `01-API接口契约.md` 一致。              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 1.3 改造范围边界

**在改造范围内**：
- PRD v4.0 中标注的 8 项架构评审变更
- 与 v4.0 新增概念直接相关的模块（六层 Prompt、PlatformContentTypeStyle、ContentTemplate、实验室、多模态路由、keyword_inject、Meta-Orchestrator、Event Bus）

**不在改造范围内**（保留现有实现）：
- 认证/RBAC（90% 完成，直接复用）
- 基础 ORM 模型（AccountPool、PersonaPool、BrandKnowledge 等，直接复用）
- 基础内容生产流程（content_forge、task_hub 骨架，直接复用）
- Docker/CI/CD 基础设施（直接复用）
- 前端非 AI 相关页面（账号管理、素材库、设置等，直接复用）

## 十一、风险与回滚策略

| 风险 | 影响 | 回滚策略 |
|------|------|---------|
| **Alembic 迁移失败** | 数据库 Schema 不一致 | 每个迁移脚本包含 `downgrade()`，失败时自动回滚 |
| **API 契约破坏** | 前端调用失败 | 保留旧 API 路径 1 个 Sprint，新增 `/v2/` 前缀的新 API |
| **Agent Harness 升级失败** | 所有 Agent 不可用 | 保留 `src/harness_v2/` 备份，出问题 5 分钟内切回 |
| **前端布局改造失败** | 页面渲染异常 | 使用 Feature Flag，出问题即时关闭新布局 |
| **LLM 路由异常** | 内容生成失败 | 配置兜底模型链（GPT-4o → 4o-mini → DeepSeek），异常自动降级 |
