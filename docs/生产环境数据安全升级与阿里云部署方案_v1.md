## 一、核心结论

用户的核心需求——**"在生成环节部署后，系统升级更新代码时不会影响已存入数据库的生产数据"**——当前项目**尚未满足**。

实现该需求必须建立**三层防御体系**：

| 防御层级 | 作用 | 核心手段 |
|---------|------|---------|
| **物理隔离** | 数据与容器生命周期解耦 | Docker Volume 持久化 + 阿里云 OSS/RDS 托管 + 定期备份 |
| **逻辑隔离** | Schema 变更与代码变更解耦 | Alembic 向后兼容迁移 + Expand-Contract 模式 |
| **部署隔离** | 部署过程与数据操作解耦 | 独立迁移 Job + 蓝绿部署 + 自动备份门禁 |

**一句话总结**：将 Schema 变更的绝对控制权完全交还给 Alembic，消灭 `main.py` 中的内联 DDL；通过 "独立迁移 Job + 自动备份 + 向后兼容迁移" 的三层保护，确保每次代码升级都不会触碰已存入数据库的生产数据。

## 三、三层防御体系

### 3.1 第一层：物理隔离 — 数据与容器解耦

#### 3.1.1 现状问题

当前数据存于 Docker Volume 和容器内 `/app/uploads`，容器重建即丢数据。阿里云部署必须将核心数据托管到云原生服务，彻底消除"容器销毁 = 数据丢失"的风险。

#### 3.1.2 阿里云托管方案

| 数据类型 | 当前存储 | 阿里云方案 | 优势 |
|---------|---------|-----------|------|
| **结构化数据** | PostgreSQL Docker Volume | **RDS PostgreSQL 16** | 自动备份、高可用、监控告警、无需自行维护 |
| **上传文件** | 本地 `uploads/` 目录 | **OSS 标准存储** | 无限容量、CDN 加速、跨区域复制、版本控制 |
| **缓存/队列** | Redis Docker Volume | **云数据库 Redis/Tair** | AOF 持久化、主备切换、性能监控 |
| **数据库备份** | 无 | **RDS 自动备份 + OSS 归档** | 7天自动保留、PITR 时间点恢复 |

#### 3.1.4 后端 OSS 集成代码

```python
# src/core/storage.py — 阿里云 OSS 存储适配器
import oss2
from src.core.config import settings

class AliyunOSSStorage:
    """阿里云 OSS 对象存储封装，替代本地文件系统。"""
    
    def __init__(self):
        self.auth = oss2.Auth(
            settings.OSS_ACCESS_KEY_ID,
            settings.OSS_ACCESS_KEY_SECRET
        )
        self.bucket = oss2.Bucket(
            self.auth,
            settings.OSS_ENDPOINT,          # e.g. oss-cn-hangzhou.aliyuncs.com
            settings.OSS_BUCKET_NAME
        )
        self.cdn_domain = settings.OSS_CDN_DOMAIN  # 可选 CDN 加速域名
    
    def upload_file(self, local_path: str, object_key: str) -> str:
        """上传文件到 OSS，返回可访问的 URL。"""
        self.bucket.put_object_from_file(object_key, local_path)
        return f"https://{self.cdn_domain or self.bucket.bucket_name}.{self.bucket.endpoint}/{object_key}"
    
    def get_url(self, object_key: str, expires: int = 3600) -> str:
        """获取带签名的临时访问 URL（私有 Bucket）。"""
        return self.bucket.sign_url("GET", object_key, expires)
    
    def delete(self, object_key: str) -> None:
        """删除 OSS 对象。"""
        self.bucket.delete_object(object_key)


# src/core/config.py — 新增配置项
class Settings(BaseSettings):
    # ... 现有配置 ...
    
    # 阿里云 OSS
    OSS_ACCESS_KEY_ID: str = ""
    OSS_ACCESS_KEY_SECRET: str = ""
    OSS_ENDPOINT: str = "oss-cn-hangzhou.aliyuncs.com"
    OSS_BUCKET_NAME: str = ""
    OSS_CDN_DOMAIN: str = ""          # e.g. cdn.ecodreamomni.com
    OSS_PUBLIC_READ: bool = True      # 公开读 / 私有 + 签名 URL
```

### 3.2 第二层：逻辑隔离 — Schema 变更安全策略

#### 3.2.1 现状问题：`main.py` 存在两套冲突的 Schema 管理机制

`apps/backend/src/main.py:48-77` 中，`lifespan` 函数同时执行：

1. **15+ 个 `__table__.create(checkfirst=True)`** — 绕过 Alembic 版本控制
2. **内联 `ALTER TABLE timeline_events ADD COLUMN cron_job_id`** — 无版本记录、不可回滚
3. **`CREATE EXTENSION IF NOT EXISTS vector`** — 应在迁移脚本中管理

这导致：新环境部署后，Alembic 认为需要执行迁移，但表已存在，导致 `alembic upgrade head` 失败（`Relation already exists`）。

#### 3.2.2 立即行动：消灭"双轨制"

**步骤 1：重构 `main.py` lifespan，移除所有 DDL**

```python
# apps/backend/src/main.py — 重构后
@asynccontextmanager
async def lifespan(app: FastAPI):
    # ── Startup: 仅保留 DML（数据操作），所有 DDL 交给 Alembic ──
    
    # 1. Seed default data for LLM models
    from sqlalchemy import select
    from sqlalchemy.ext.asyncio import AsyncSession
    from sqlalchemy.orm import sessionmaker
    
    AsyncSessionLocal = sessionmaker(engine, class_=AsyncSession, expire_on_commit=False)
    async with AsyncSessionLocal() as db:
        from src.services import llm_hub as lhs
        from src.models.llm_hub_orm import LLMModelORM
        result = await db.execute(select(LLMModelORM))
        if not result.scalars().all():
            default_models = [
                {"provider": "deepseek", "model_name": "deepseek-chat", ...},
                # ...
            ]
            for m in default_models:
                await lhs.register_model(db=db, ...)
    
    # 2. Load workflow engine presets
    from src.services import workflow_engine as we
    we.load_presets()
    
    # 3. Warm TaskHub cache
    # ...
    
    yield
    # Shutdown: nothing special
```

> **注意**：所有 `__table__.create()`、`ALTER TABLE`、`CREATE EXTENSION` 必须从 `lifespan` 中彻底移除。

**步骤 2：将内联变更提取为 Alembic 迁移**

```python
# alembic/versions/xxxx_add_cron_job_id_to_timeline_events.py
"""add cron_job_id to timeline_events

Revision ID: xxxx
Revises: a6135eeaae42
"""
from typing import Sequence, Union
from alembic import op
import sqlalchemy as sa

revision: str = 'xxxx'
down_revision: Union[str, Sequence[str], None] = 'a6135eeaae42'

def upgrade() -> None:
    op.add_column('timeline_events', sa.Column('cron_job_id', sa.String(64), nullable=True))

def downgrade() -> None:
    op.drop_column('timeline_events', 'cron_job_id')
```

**步骤 3：修复 `alembic.ini` 和 `env.py`**

```python
# alembic/env.py — 重构后
import os
import asyncio
from logging.config import fileConfig

from sqlalchemy import pool, text
from sqlalchemy.ext.asyncio import create_async_engine

from alembic import context
from src.core.database import Base
from src.core.config import settings
# ... 保留所有模型导入 ...

config = context.config
if config.config_file_name is not None:
    fileConfig(config.config_file_name)

target_metadata = Base.metadata


def get_database_url() -> str:
    """从应用配置读取 DATABASE_URL，覆盖 alembic.ini 的硬编码值。"""
    url = settings.DATABASE_URL if hasattr(settings, 'DATABASE_URL') else os.getenv("DATABASE_URL")
    # Alembic 需要同步驱动（psycopg2）
    return url.replace("postgresql+asyncpg", "postgresql+psycopg2")


def run_migrations_offline() -> None:
    url = get_database_url()
    context.configure(
        url=url,
        target_metadata=target_metadata,
        literal_binds=True,
        dialect_opts={"paramstyle": "named"},
        compare_type=True,
    )
    with context.begin_transaction():
        context.run_migrations()


def do_run_migrations(connection) -> None:
    # 生产安全参数：防止锁表过久
    connection.execute(text("SET statement_timeout = '120s'"))
    connection.execute(text("SET lock_timeout = '60s'"))
    
    context.configure(
        connection=connection,
        target_metadata=target_metadata,
        compare_type=True,
        transaction_per_migration=True,  # 每个迁移独立事务
    )
    with context.begin_transaction():
        context.run_migrations()


async def run_migrations_online() -> None:
    url = get_database_url()
    connectable = create_async_engine(url, poolclass=pool.NullPool)
    
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    
    await connectable.dispose()


if context.is_offline_mode():
    run_migrations_offline()
else:
    asyncio.run(run_migrations_online())
```

#### 3.2.3 Schema 变更铁律 — Expand-Contract 模式

```
阶段 1 (Deploy New Code):  添加新列/新表（nullable=True，旧代码忽略，新代码可选使用）
阶段 2 (Backfill Data):    回填新列数据（保证新旧代码都能正确读取）
阶段 3 (Switch Reads):     新代码开始读取新列
阶段 4 (Deploy Clean-up):  删除旧列/旧表（确认无代码依赖后再执行）
```

| 操作 | 安全流程 |
|------|---------|
| 删除列 | 先部署代码停止写入旧列 → 等待一个发布周期 → 再执行迁移删除列 |
| 删除表 | 重命名表为 `_deprecated_xxx` → 观察 N 天 → 再真正删除 |
| 重命名列/表 | **禁止直接 rename**，采用"添加新列 + 复制数据 + 删除旧列"三步策略 |
| 修改列类型 | 添加新列 → 双写新旧列 → 回填历史数据 → 切换读取 → 删除旧列 |

### 3.3 第三层：部署隔离 — 零停机部署架构

#### 3.3.1 核心原则：迁移与代码部署解耦

| 维度 | 独立 Job | 启动时自动执行 |
|-----|---------|--------------|
| 可观测性 | ✅ 迁移日志独立，失败容易定位 | ❌ 与业务日志混杂 |
| 失败处理 | ✅ 失败即终止部署，不污染生产 | ❌ 容器反复 Crash |
| 竞争安全 | ✅ 显式控制执行时机，无并发风险 | ⚠️ 多实例启动需分布式锁 |
| 回滚能力 | ✅ 迁移失败不启动新代码，原子性部署 | ❌ 代码已更新，迁移失败导致不一致 |

**本项目结论**：使用 Docker Compose Profile 的独立 `migration` Job，部署脚本中显式执行，失败即终止部署。

### 4.2 VPC 网络架构设计

```
VPC: 172.16.0.0/16 (ecodreamomni-vpc)
│
├── 交换机 A: 172.16.1.0/24 (cn-hangzhou-a)
│   ├── ECS-Blue:     172.16.1.10
│   ├── ECS-Green:    172.16.1.11
│   └── ECS-Mgmt:     172.16.1.20 (运维跳板机)
│
└── 交换机 B: 172.16.2.0/24 (cn-hangzhou-b)
    ├── RDS PostgreSQL: 172.16.2.10 (内网地址)
    └── Redis:          172.16.2.20 (内网地址)
```

**网络隔离原则**：
- 数据库和缓存仅分配**内网地址**，不分配公网 IP
- ECS 实例通过安全组白名单访问 RDS/Redis（172.16.2.0/24）
- 仅 ECS 和 SLB 暴露公网端口（80/443）
- 运维通过 ECS-Mgmt 跳板机 SSH 登录，再内网访问其他资源

### 4.3 安全组策略（最小权限原则）

| 安全组 | 入站规则 | 说明 |
|--------|---------|------|
| **sg-ecs-frontend** | 0.0.0.0/0 → TCP 80, 443 | 仅 SLB 和公网用户访问 |
| **sg-ecs-ssh** | <办公IP>/32 → TCP 22 | SSH 仅允许办公网络 |
| **sg-rds-postgres** | 172.16.1.0/24 → TCP 5432 | 仅 ECS 应用服务器访问 |

```bash
#!/bin/bash
# scripts/deploy-aliyun.sh
# EcoDreamOmni 阿里云生产环境蓝绿部署脚本
# =============================================================================

set -euo pipefail

VERSION=${1:-latest}
ENV_FILE="/opt/ecodreamomni/.env.production"

RED='\033[0;31m'
GREEN='\033[0;32m'
YELLOW='\033[1;33m'
NC='\033[0m'

log_info()  { echo -e "${GREEN}[INFO]${NC} $1"; }
log_warn()  { echo -e "${YELLOW}[WARN]${NC} $1"; }
log_error() { echo -e "${RED}[ERROR]${NC} $1"; }

# ---------------------------------------------------------------------------
# Step 0: 加载环境变量
# ---------------------------------------------------------------------------
if [[ ! -f "$ENV_FILE" ]]; then
    log_error "环境文件不存在: $ENV_FILE"
    exit 1
fi

source "$ENV_FILE"
set +a

export VERSION
export REGISTRY

log_info "=== EcoDreamOmni 阿里云生产部署 ==="
log_info "版本: $VERSION"
log_info "时间: $(date)"

# ---------------------------------------------------------------------------
# Step 1: 部署前数据保护 — 自动备份
# ---------------------------------------------------------------------------
log_info ">>> Step 1: 执行部署前数据库备份..."
mkdir -p "$BACKUP_DIR"

# 使用 RDS 自动备份（推荐），或手动逻辑备份
# 若使用自托管 PostgreSQL（非推荐），执行：
# docker exec ecodream-postgres pg_dump -U ecodream -d ecodream --clean --if-exists | gzip > "${BACKUP_DIR}/pre_deploy_${TIMESTAMP}.sql.gz"

# 记录当前 Alembic 版本

CURRENT_ALEMBIC=$(alembic current 2>/dev/null || echo "unknown")
echo "$CURRENT_ALEMBIC" > "${BACKUP_DIR}/alembic_version_${TIMESTAMP}.txt"
log_info "当前 Alembic 版本: $CURRENT_ALEMBIC"

# ---------------------------------------------------------------------------
# Step 2: 执行数据库迁移（向后兼容的扩展性变更）
# ---------------------------------------------------------------------------
log_info ">>> Step 2: 执行数据库迁移..."

# 拉取最新镜像
docker compose -f "$COMPOSE_FILE" pull backend

# 运行独立迁移 Job
docker compose -f "$COMPOSE_FILE" --profile migrate up migration --exit-code-from migration

if [[ $? -ne 0 ]]; then
    log_error "❌ 数据库迁移失败！终止部署，生产数据未受影响。"
    log_error "请检查迁移日志，修复后重新部署。"
    exit 1
fi

log_info "✅ 数据库迁移成功"

# ---------------------------------------------------------------------------
# Step 3: 滚动重启（单节点下的最佳实践）
# ---------------------------------------------------------------------------
# 注：阿里云 ECS 单节点无法做真正的蓝绿部署（需要 2x ECS）
# 单节点采用：先停旧容器 → 启动新容器 → 健康检查
# 若需要蓝绿，需在 SLB 后挂载两台 ECS

log_info ">>> Step 3: 滚动更新服务..."

# 先更新后端和 Worker（会短暂停止服务，约 5-10 秒）
docker compose -f "$COMPOSE_FILE" up -d backend celery-worker celery-beat

# 等待后端健康
log_info ">>> Step 4: 等待后端健康检查..."
for i in {1..12}; do
    if curl -sf http://localhost:8000/health > /dev/null 2>&1; then
        log_info "✅ 后端服务健康"
        break
    fi
    log_warn "等待后端就绪... ($i/12)"
    sleep 5
done

if [[ $i -eq 12 ]]; then
    log_error "❌ 后端健康检查失败！执行回滚..."
    # 回滚到上一版本镜像
    # docker compose -f "$COMPOSE_FILE" up -d backend=${PREVIOUS_VERSION}
    exit 1
fi

# ---------------------------------------------------------------------------
# Step 5: 更新 Nginx（前端静态资源）
# ---------------------------------------------------------------------------
log_info ">>> Step 5: 更新 Nginx..."
docker compose -f "$COMPOSE_FILE" up -d nginx

# ---------------------------------------------------------------------------
# Step 6: 清理旧镜像
# ---------------------------------------------------------------------------
log_info ">>> Step 6: 清理旧镜像..."
docker image prune -af --filter "until=168h" || true

# ---------------------------------------------------------------------------
# Step 7: 部署后验证
# ---------------------------------------------------------------------------
log_info ">>> Step 7: 部署后验证..."
API_STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://api.ecodreamomni.com/health || echo "000")
if [[ "$API_STATUS" == "200" ]]; then
    log_info "✅ API 健康检查通过 (HTTP 200)"
else

fi

log_info "=== 部署完成 ==="
log_info "版本: $VERSION"
log_info "时间: $(date)"
log_info "备份目录: $BACKUP_DIR"
```

### 4.8 阿里云 RDS 迁移策略

**从本地 Docker PostgreSQL 迁移到阿里云 RDS**：

```bash
#!/bin/bash
# scripts/migrate-to-rds.sh
# 将本地/开发环境 PostgreSQL 数据迁移到阿里云 RDS

set -euo pipefail

SOURCE_DB="postgresql://ecodream:ecodream_dev@localhost:5432/ecodream"
TARGET_DB="postgresql+psycopg2://ecodream_prod:<PASSWORD>@rm-xxx.mysql.rds.aliyuncs.com:5432/ecodream"

# Step 1: 在 RDS 上创建空数据库（如果还没有）
psql "${TARGET_DB}" -c "SELECT 1" > /dev/null 2>&1 || \
  psql "${TARGET_DB/postgresql+psycopg2/postgresql}" -c "CREATE DATABASE ecodream;"

# Step 2: 逻辑备份 + 恢复
pg_dump "$SOURCE_DB" --clean --if-exists --no-owner --no-privileges | \
  psql "$TARGET_DB"

# Step 3: 验证数据量
echo "源数据库表数量:"
psql "$SOURCE_DB" -c "SELECT COUNT(*) FROM information_schema.tables WHERE table_schema='public';"

echo "目标数据库表数量:"
psql "$TARGET_DB" -c "SELECT COUNT(*) FROM information_schema.tables WHERE table_schema='public';"

echo "迁移完成。请运行应用验证。"
```

```python
@app.get("/health")
async def health_check(db: AsyncSession = Depends(get_db)):
    status = {"status": "ok", "version": settings.APP_VERSION, "checks": {}}
    
    # 1. 数据库连接检查
    try:
        from sqlalchemy import text
        result = await db.execute(text("SELECT 1"))
        status["checks"]["database"] = "ok"
    except Exception as e:
        status["checks"]["database"] = f"error: {str(e)}"
        status["status"] = "degraded"
    
    # 2. Redis 连接检查
    try:
        r = redis.from_url(settings.REDIS_URL)
        await r.ping()
        await r.close()
        status["checks"]["redis"] = "ok"
    except Exception as e:
        status["checks"]["redis"] = f"error: {str(e)}"
        status["status"] = "degraded"
    
    # 3. OSS 连接检查（如果配置了）
    if settings.OSS_ACCESS_KEY_ID:
        try:
            from src.core.storage import AliyunOSSStorage
            storage = AliyunOSSStorage()
            # 轻量检查：获取 Bucket 信息
            storage.bucket.get_bucket_info()
            status["checks"]["oss"] = "ok"
        except Exception as e:
            status["checks"]["oss"] = f"error: {str(e)}"
            status["status"] = "degraded"
    
    http_status = 200 if status["status"] == "ok" else 503
    return JSONResponse(status_code=http_status, content=status)
```
