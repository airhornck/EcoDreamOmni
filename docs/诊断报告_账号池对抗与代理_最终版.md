## 一、账号隔离机制诊断

### 1.1 Cookie 加密隔离

| 检查项 | 状态 | 说明 |
|--------|------|------|
| Cookie 是否加密存储 | ✅ 是 | 使用 AES-256-GCM，密钥来自 `COOKIE_VAULT_KEY` |
| Cookie 是否明文落日志 | ✅ 否 | `_decrypt_cookie` 仅在运行时内存解密，无明文日志 |
| 加密密钥是否租户隔离 | ❌ 否 | 单全局密钥 `_VAULT_KEY_32`，无租户维度隔离 |

```python
# platform_account.py
_VAULT_KEY = os.environ.get("COOKIE_VAULT_KEY", "ecodream-omni-cookie-vault-key-32b!").encode()
_VAULT_KEY_32 = _VAULT_KEY[:32].ljust(32, b"\0")   # ← 全局单密钥
```

**结论**: Cookie 加密存储已落实，但密钥为全局硬编码回退值，生产环境必须替换为 KMS/环境变量注入。

---

### 1.2 HTTP 客户端隔离

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 每账号独立 XhsClient | ⚠️ 部分 | 按 `cache_key = cookie[:32] + user_agent[:32]` 缓存 |
| Session/Cookie 隔离 | ✅ 是 | 不同 cookie → 不同 XhsClient → 独立 `requests.Session` |
| Connection Pool 隔离 | ❌ 否 | 底层仍共享 `requests` 全局连接池 |
| 客户端缓存永不清除 | ❌ 否 | `_XHS_CLIENTS` 字典只增不减，内存泄漏风险 |

```python
# xhs_publisher.py
_XHS_CLIENTS: Dict[str, object] = {}              # ← 模块级全局缓存，永不清理

def _get_xhs_client(cookie: str, user_agent: str = "", proxies: Optional[Dict] = None):
    cache_key = f"{cookie[:32]}:{user_agent[:32]}"  # ← 前缀碰撞风险极低但存在
    if cache_key not in _XHS_CLIENTS:
        _XHS_CLIENTS[cache_key] = XhsClient(...)
```

**结论**: 不同账号的 Cookie 不会串号，但连接层未彻底隔离（共享 TCP 连接池）。

### 1.3 网络层隔离（IP / 代理）

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 每账号独立出口 IP | ❌ 否 | 未配置代理，所有账号共享宿主 IP |
| 代理配置存在 | ❌ 否 | `proxy_config = None`，Proxy Store 为空 |
| IP 轮换机制 | ❌ 否 | 无代理 = 无轮换 |
| 账号-代理绑定 | ❌ 否 | 代码支持，但无数据 |

```python
# xhs_publisher.py 第 274-284 行
if account.proxy_config and account.proxy_config.proxy_id:
    proxy_id = account.proxy_config.proxy_id
    ...
else:
    logger.warning("Proxy %s not found or inactive for account %s", ...)
    # → 实际走向：proxies = None
```

**结论**: 网络层零隔离。小红书服务器看到的所有发布请求来自**同一公网 IP**，这是多账号运营的最大风控隐患。

### 1.4 浏览器指纹隔离

| 检查项 | 状态 | 说明 |
|--------|------|------|
| User-Agent 差异化 | ✅ 是 | 每账号随机生成不同 UA |
| UA 是否传入请求 | ✅ 是 | `XhsClient` 将 UA 写入 `session.headers["user-agent"]` |
| Viewport 差异化 | ⚠️ 无效 | 生成但从未传入 `XhsClient` |
| Locale 差异化 | ❌ 否 | 生成但从未使用 |
| Timezone 差异化 | ❌ 否 | 生成但从未使用 |
| Canvas 噪声 | ❌ 否 | 仅布尔标志，`requests` 无 Canvas API |
| WebGL 噪声 | ❌ 否 | 仅布尔标志，`requests` 无 WebGL API |

```python
# fingerprint_engine.py 生成但仅用 UA
return {
    "user_agent": random.choice(ua_pool),    # ✅ 被使用
    "viewport": viewport,                     # ❌ 未使用
    "locale": random.choice(_LOCALES),        # ❌ 未使用
    "timezone": random.choice(_TIMEZONES),    # ❌ 未使用
    "canvas_noise": random.choice([True, False]),   # ❌ 死代码
    "webgl_noise": random.choice([True, False]),    # ❌ 死代码
}
```

**结论**: 只有 User-Agent 真正发挥了隔离作用，其余指纹维度均为「装饰性代码」。

### 1.5 数据持久化隔离

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 账号池持久化存储 | ❌ 否 | 纯内存字典 `_account_pool_db`，重启即清空 |
| Cookie 持久化 | ❌ 否 | 随账号池一起丢失 |
| 发布历史持久化 | ❌ 否 | 无 `published_url`/`platform_post_id` 字段 |
| 账号间数据隔离 | ✅ 是 | 内存中按 `entry_id` 键隔离，无交叉访问 |

**结论**: 隔离在运行时有效，但容器重启后所有账号、Cookie、发布历史全部丢失。

## 三、发布真实性判定

### 3.1 代码逻辑矛盾

系统启动时硬编码 seed：
```python
# main.py
aps.create_account(
    platform="xhs", ...,
    cookie="demo_cookie",   # ← 占位符，非真实 Cookie
)
```

而 `publish_to_xhs` 的 Cookie 优先级：
```
account.cookie (demo_cookie)  >  settings.REDNOTE_COOKIE (真实 Cookie)
```

**矛盾**: 若账号存在于池中，必然使用无效的 `demo_cookie`；只有当 `get_pool_entry()` 返回 `None` 时才会回退到真实 Cookie。

### 3.2 任务状态 ≠ 发布成功

```python
# task_hub.py simulate_node_output(publisher)
try:
    result = publish_to_xhs(...)
except Exception as e:
    return {"publish_result": {"published": False, ...}, ...}   # ← 捕获异常，不抛错

# workflow 继续执行 → status = completed
```

数据库中 `status = 'completed'` **仅表示工作流走完**，不代表小红书 API 返回了 `note_id`。

### 3.3 无持久化审计

`tasks` 表缺失字段：
- `published_url`
- `platform_post_id`
- `published_at`
- `publish_error`

**结论**: 从后端数据无法验证这三篇文章是否真的发布到了小红书。您的观察（在小红书 App 中看到帖子）是唯一的真实性证据。

## 五、修复建议（按优先级排序）

### P0 — 阻断性风险

1. **补全发布结果持久化**
   - `tasks` 表新增：`published_url`, `platform_post_id`, `published_at`, `publish_error`
   - Publisher 节点执行后将 `publish_result` 写入 `task.prompt_variables` 并同步持久化到 DB
   - **目的**: 唯一手段验证发布真伪，支撑后续风控分析

2. **修复 Cookie 来源优先级**
   - 修改 `publish_to_xhs`：当 `account.cookie` 为占位符（`demo_cookie` 或空）时，自动回退到 `settings.REDNOTE_COOKIE`
   - 或在启动 seed 时直接将 `settings.REDNOTE_COOKIE` 写入账号池
   - **目的**: 消除「有账号 = 用假 Cookie」的逻辑陷阱

3. **账号池持久化迁移**
   - 将 `_account_pool_db` 内存字典迁移到 PostgreSQL 表 `account_pool_entries`
   - Cookie 加密字段直接复用现有 `cookie_encrypted`
   - **目的**: 消除容器重启导致的数据丢失

### P1 — 高风险

4. **配置代理并绑定账号**
   - 接入住宅代理（Bright Data / Oxylabs）
   - 每个小红书账号绑定独立代理出口 IP
   - 在 `proxy_config` 中配置 `proxy_id` 并激活
   - **目的**: 避免多账号共享同一 IP 被小红书风控关联

5. **修复健康检查 API**
   - `/platforms/xhs/account-status` 存在 `NameError: name 'settings' is not defined`
   - 补充 import 并验证 Cookie 有效性
   - **目的**: 发布前前置校验，避免无效发布

### P2 — 中风险

6. **清理 XhsClient 缓存泄漏**
   - `_XHS_CLIENTS` 添加 LRU 淘汰或定时清理
   - 账号删除/更新 Cookie 时同步清除对应缓存
   - **目的**: 防止内存无限增长及旧 Cookie 复用

7. **完善指纹对抗生效链路**
   - 将 `viewport`, `locale`, `timezone` 实际传入 `XhsClient`（需 xhs 库支持或换用 Playwright）
   - 若继续使用 `requests`，移除 `canvas_noise` / `webgl_noise` 死代码，避免误导
   - **目的**: 代码与行为一致，减少维护歧义

### P3 — 低风险

8. **Cookie Vault 密钥租户隔离**
   - `_VAULT_KEY` 改为按 `tenant_id` 派生
   - 生产环境接入 KMS（AWS KMS / HashiCorp Vault）
   - **目的**: 多租户场景下的密钥隔离合规

---

## 六、最终结论

> **当前系统处于「半真实发布」状态**：代码骨架具备真实发布的全部接口，但关键数据层（账号持久化、发布审计）和风控层（代理 IP、完整指纹）均未落地。
>
> 三篇样本任务的 `completed` 状态**不能作为发布成功的技术证据**；若您确实在小红书 App 中看到了对应帖子，最可能的解释是发布过程中触发了「账号未找到 → 回退到全局真实 Cookie」的分支，而非账号池机制正常运作的结果。
>
> **在 P0/P1 修复完成前，不建议进行大规模真实发布测试。**
