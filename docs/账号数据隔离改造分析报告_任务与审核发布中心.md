# EcoDreamOmni — 账号数据隔离改造分析报告

> **分析目标**：评估当前系统在「公共数据 + 按创建者隔离任务与审核」需求下的改造可行性，给出专家级修改建议。
> 
> **需求定义（用户调整后）**：
> - 【账号与人设】【基础功能】【系统治理】【系统设置】为**公共数据**，所有登录账号共享。
> - 【内容生产→任务中心】：每个账号**只能看到自己发布的任务**。
> - 【风控与发布→审核发布中心】：每个账号**只能看到自己的任务审核**。
> 
> **分析范围**：前后端全链路（模型 → 服务 → API → 前端）。
> **分析日期**：2026-05-31

---

## 一、执行摘要

| 维度 | 结论 |
|------|------|
| **当前是否满足需求** | ❌ **完全不满足**。当前系统任何登录用户均可查看、操作、审核、发布全部任务。 |
| **改造复杂度** | 🟡 **中等**。基础设施（JWT、`get_current_user`、ORM 字段）已存在，但全链路未接通。 |
| **核心矛盾** | 1. API 层**未引入认证依赖**；2. `created_by` 字段**未被查询逻辑消费**；3. 前端 `operator` 被**硬编码**；4. 部分关联表（`PublishTaskORM`、`ContentDraftORM`）**先天缺失 `created_by`**。 |
| **建议策略** | **后端优先，前端配合**。先在后端 API 层强制注入 `get_current_user` 并做查询过滤与权限校验；再补全缺失的 ORM 字段；最后前端将硬编码 `operator` 改为当前登录用户。 |

## 二、后端深度分析

### 2.1 数据模型层（ORM）

#### 2.1.1 任务真源表 `TaskORM`（`task_orm.py`）

```python
created_by = Column(String(64), nullable=False, default="", comment="创建者")
tenant_id  = Column(String(64), nullable=True,  index=True, comment="租户ID")
```

| 评估项 | 状态 | 说明 |
|--------|------|------|
| `created_by` 字段 | ✅ 存在 | `String(64)`，但 `default=""`，允许空字符串，导致无法区分"匿名创建"与"未填写"。 |
| `tenant_id` 字段 | ⚠️ 存在但未使用 | `nullable=True`，所有查询逻辑均未引用。 |
| 索引 | ✅ 有 `ix_tasks_tenant_status` | 但查询未使用。 |

**问题**：`created_by` 是**字符串存储用户ID**，非外键约束。创建任务时由请求体传入，可被任意伪造。

| `tenant_id` | ❌ **缺失** | 同上。 |

### 2.4 认证与上下文层

#### 2.4.1 JWT Payload 结构

```python
# auth_service.py 实际写入
{"sub": user.id, "role": user.role, "exp": <timestamp>}
```

| 评估项 | 状态 | 说明 |
|--------|------|------|
| `sub` (user_id) | ✅ 有 | 可用于标识当前用户。 |
| `role` | ✅ 有 | 当前仅 `"admin"` / `"operator"` 两种。 |
| `tenant_id` | ❌ **缺失** | 中间件尝试从 JWT 读 `tenant_id`，但生成时未写入，导致始终为空。 |
| `username` / `email` | ❌ **缺失** | 非必须，但可减少后端 DB 查询次数。 |

#### 2.4.2 `get_current_user` 返回结构

```python
class User:
    id: str           # UUID
    email: str
    username: str
    role: str         # "operator" | "admin"
    is_active: bool
    tenant_id: Optional[str]   # 从 UserORM.tenant_id 读出
```

**结论**：`get_current_user` 已经能提供完整的用户身份信息（`id`, `role`, `tenant_id`），足以支撑"按创建者隔离"的需求。问题出在**API 层没有引入该依赖**。

## 三、前端深度分析

### 3.1 状态管理与请求层

| Store | 获取任务方法 | 是否传入用户过滤参数 |
|-------|-------------|---------------------|
| `taskHubStore.ts` | `fetchTasks()` | ❌ 否，`GET /api/task-hub/tasks` |
| `reviewPublishStore.ts` | `fetchConclusions(status_filter?)` | ❌ 否，`GET /api/review-publish-center/conclusions?status_filter=...` |
| `publisherStore.ts` | `fetchPublishTasks()` | ❌ 否，`GET /api/publish-tasks` |

### 3.2 `operator` 硬编码问题

前端在多个关键操作中将 `operator` 硬编码为字符串 `'operator'`，导致后端无法追溯真实操作人：

```typescript
// reviewPublishStore.ts
await confirmPublish(taskId, {
  operator: 'operator',          // ← 硬编码
  publish_mode: 'immediate',
})

// taskHubStore.ts
body: JSON.stringify({ decision, operator: 'operator', feedback })

// TaskHubCreatePage.tsx
created_by: 'operator',          // ← 硬编码
```

**影响**：即使后端增加了 `created_by` 过滤，这些接口传入的 `operator` 仍然是固定值 `'operator'`，导致：
- 审核记录中 reviewer = `'operator'`，无法区分真实审核人。
- 如果后端改为自动从 JWT 注入 `created_by`，则前端硬编码值会被忽略，需要前后端协同改造。

### 3.3 前端过滤逻辑

前端所有列表过滤均为**纯 UI 层面**（搜索关键词、Tab 状态切换），没有任何按当前用户过滤：

```tsx
// TaskHubPage.tsx
const filteredTasks = tasks.filter(
  (t) =>
    t.name.toLowerCase().includes(filter.toLowerCase()) ||
    t.status.toLowerCase().includes(filter.toLowerCase())
)

// ReviewPublishCenterPage.tsx
const filtered = conclusions.filter((c) => {
  if (activeTab === 'all') return true
  if (activeTab === 'approved') return c.review_decision === 'APPROVE'
  // ... 只有状态过滤
})
```

**结论**：前端是**透明管道**，数据隔离的 100% 防线在后端。前端改造量较小，主要是：
1. 将硬编码 `operator` / `created_by` 改为 `authStore.user.id` 或 `username`；
2. 如果后端需要前端传 `created_by`（不推荐），则前端需要传入真实值；
3. 更优方案是后端从 JWT 自动注入，前端完全不需要传这些字段。

## 四、安全风险与影响评估

### 4.1 当前风险矩阵

| 风险场景 | 严重等级 | 说明 |
|----------|---------|------|
| **数据越权查看** | 🔴 高 | 用户 A 登录后可查看用户 B 创建的任务、草稿、审核记录。 |
| **数据越权操作** | 🔴 高 | 用户 A 可修改、删除、发布用户 B 的任务。 |
| **审核权限泄露** | 🔴 高 | 任何用户可对任意任务执行通过/拒绝/打回操作，无审核权限校验。 |
| **operator 伪造** | 🟡 中 | 前端硬编码 + 后端接受请求体传入 `created_by`，导致操作记录不可信。 |
| **发布任务冒用** | 🟡 中 | 无 `created_by` 的 `PublishTaskORM` 无法追溯是谁发起的发布。 |
| **缓存数据泄露** | 🟡 中 | `task_hub.py` 的内存缓存 `_task_db` 存储全部任务，绕过 DB 查询的隔离逻辑。 |

### 4.2 对业务的影响

在当前矩阵运营场景下，如果多个运营人员（素人号操盘手）共用系统：
- **一人误操作可导致全矩阵内容异常**：误删、误发布、误审核他人任务。
- **审核流程失效**：审核发布中心形同虚设，因为任何人都可以"自己审自己"。
- **合规风险**：如果平台要求内容发布需经特定人员审核，当前系统无法证明审核人身份。

## 五、修改建议

### 5.1 总体策略

> **原则：最小侵入、后端优先、前端配合、测试护航。**

1. **后端 API 层**：强制引入 `get_current_user`，作为数据隔离的唯一真源。
2. **后端 Service 层**：查询函数增加 `created_by` 过滤参数；权限校验增加所有权检查。
3. **后端 ORM 层**：为缺失 `created_by` 的表添加字段 + Alembic 迁移。
4. **前端**：移除硬编码 `operator`/`created_by`；依赖后端从 JWT 自动注入。
5. **缓存层**：改造内存缓存，使其支持按用户隔离，或弃用全量缓存改为按需查询。

### 5.2 改造 Phase 1：最小可行方案（MVP）

**目标**：在不改动 ORM 表结构的前提下，先让 `TaskORM` 相关的任务列表和审核列表实现按创建者隔离。

#### 5.2.1 后端改造点

| # | 文件 | 改造内容 | 优先级 |
|---|------|---------|--------|
| 1 | `api/task_hub.py` | **所有接口**增加 `user: User = Depends(get_current_user)`。创建任务时自动将 `user.id` 写入 `created_by`（覆盖请求体传入值）。列表查询增加 `created_by=user.id` 过滤。 | P0 |
| 2 | `api/human_in_loop.py` | 所有接口引入 `get_current_user`。`GET /pending` 只返回 `created_by == user.id` 且 `status == "human_wait"` 的任务。审核操作（approve/reject/revise）增加所有权校验：仅允许操作自己创建的任务。 | P0 |
| 3 | `api/review_publish.py` | 所有接口引入 `get_current_user`。`GET /conclusions` 只遍历 `created_by == user.id` 的任务。确认发布、修改内容等操作增加所有权校验。 | P0 |
| 4 | `services/task_function.py` | `list_tasks_from_db()` 增加 `created_by: Optional[str] = None` 参数，并在有值时追加 `WHERE TaskORM.created_by == created_by`。 | P0 |
| 5 | `services/task_hub.py` | `list_tasks()` 传入 `created_by` 参数。评估内存缓存 `_task_db` 的影响——**建议 Phase 1 暂时禁用缓存**，或改为按 `created_by` 分片缓存。 | P0 |
| 6 | `services/human_in_loop.py` | `get_pending_tasks()` 增加 `created_by` 过滤。`approve_task()` / `reject_task()` / `revise_task()` 增加校验：`task.created_by != user.id` 时抛出 403。 | P0 |

#### 5.2.2 前端改造点（配合后端）

| # | 文件 | 改造内容 | 优先级 |
|---|------|---------|--------|
| 7 | `taskHubStore.ts` | 移除创建任务时传入的 `created_by` 字段（由后端自动注入）。 | P1 |
| 8 | `reviewPublishStore.ts` | 移除 `operator: 'operator'` 硬编码。如果后端仍要求传入 `operator`，改为 `authStore.user.id` 或 `username`。 | P1 |
| 9 | `TaskHubCreatePage.tsx` | 移除 `created_by: 'operator'` 硬编码。 | P1 |

### 5.3 改造 Phase 2：ORM 补全与持久化隔离

**目标**：为 `PublishTaskORM`、`ContentDraftORM`、`ReviewRecordORM` 添加 `created_by` 字段，实现全链路隔离。

#### 5.3.1 数据库迁移

```python
# Alembic 迁移脚本示例（需新增 revision）
# 为 publish_tasks 添加 created_by
op.add_column('publish_tasks', sa.Column('created_by', sa.String(64), nullable=True, index=True))
# 为 content_drafts 添加 created_by
op.add_column('content_drafts', sa.Column('created_by', sa.String(64), nullable=True, index=True))
# 为 review_records 添加 created_by（或 task_creator_id，用于追溯被审核任务的创建者）
op.add_column('review_records', sa.Column('task_created_by', sa.String(64), nullable=True, index=True))
```

#### 5.3.2 API 与服务层改造

| # | 文件 | 改造内容 |
|---|------|---------|
| 10 | `api/publisher.py` | `create_task()` 自动写入 `created_by = user.id`。`list_tasks()` 过滤 `created_by == user.id`。 |
| 11 | `api/content_draft.py`（如有） | 同上逻辑。 |
| 12 | `services/publisher_service.py` | `list_publish_tasks()` 增加 `created_by` 参数过滤。 |

### 5.4 改造 Phase 3：角色权限与审核流程强化

**目标**：区分"普通运营人员"与"审核人员"，实现真正的审核分离。

#### 5.4.1 角色设计建议

当前只有 `admin` / `operator`，建议引入：

| 角色 | 权限 |
|------|------|
| `operator` | 创建任务、查看自己的任务、查看自己的审核结论、确认发布自己的任务。 |
| `reviewer` | 查看全部待审核任务（或指定范围内的）、执行审核操作。 |
| `admin` | 全部权限，可查看/操作所有任务。 |

#### 5.4.2 审核权限逻辑

```python
# 伪代码示例
def can_review_task(user: User, task: TaskORM) -> bool:
    if user.role == "admin":
        return True
    if user.role == "reviewer":
        return True   # reviewer 可以审核所有进入 human_wait 的任务
    return False      # operator 不能审核（除非是自己的任务且允许自审）

def can_view_task(user: User, task: TaskORM) -> bool:
    if user.role == "admin":
        return True
    return task.created_by == user.id
```

> **注意**：用户当前需求是"只能看到自己的任务审核"，这意味着当前用户既是创建者也是审核者（自审模式）。如果未来需要引入独立的审核人，需要 `reviewer` 角色。

### 5.5 改造 Phase 4：缓存与性能优化

**问题**：`services/task_hub.py` 中的内存缓存 `_task_db` 存储全部任务，如果按用户隔离，缓存需要重构。

#### 5.5.1 方案对比

| 方案 | 说明 | 复杂度 |
|------|------|--------|
| **A. 禁用内存缓存** | 直接查 DB，最简单，数据量小时性能可接受。 | 低 |
| **B. 按用户分片缓存** | `_task_db: Dict[str, Dict[str, Task]]` → `user_id -> task_id -> Task`。 | 中 |
| **C. Redis 缓存 + 用户标签** | 将缓存迁移到 Redis，key 设计为 `tasks:{user_id}:{task_id}`。 | 高 |

**建议**：Phase 1~2 采用 **方案 A（禁用内存缓存）**，确保隔离逻辑正确；待数据量增大后再考虑方案 B 或 C。

## 八、结论与下一步行动

### 8.1 结论

当前系统**不满足**用户调整后的数据隔离需求。虽然 `TaskORM` 预留了 `created_by` 字段，但存在以下核心阻断点：

1. **API 层零认证**：`task_hub.py`、`human_in_loop.py`、`review_publish.py` 均未引入 `get_current_user`。
2. **查询层零过滤**：`list_tasks_from_db` 未使用 `created_by` 做过滤。
3. **关联模型先天缺失**：`PublishTaskORM`、`ContentDraftORM`、`ReviewRecordORM` 无 `created_by`。
4. **前端操作人不可信**：多处硬编码 `operator: 'operator'`，后端接受请求体伪造 `created_by`。
