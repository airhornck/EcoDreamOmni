### QA-1: Dashboard 驾驶舱页面 — 快捷入口与数据正确性

#### 2.1.1 问题描述

1. **快捷入口隐藏需求**：用户要求隐藏 `"新建互动剧本"`、`"导入实际数据"`、`"查看昨日战报"` 三个快捷操作按钮；
2. **顶部数据正确性**：用户质疑顶部 5 张统计卡的数据来源与计算逻辑是否正确。

#### 2.1.2 现状与代码证据

**快捷入口代码位置**：`apps/frontend/src/pages/DashboardPage.tsx:60-65`

```tsx
const quickActions = [
```

**顶部统计卡代码位置**：`DashboardPage.tsx:166-196`

```tsx
<StatCard label="待审核任务数" value={coreMetrics?.pendingReview ?? overview?.contentsPendingReview ?? 0} />
<StatCard label="今日已发布数" value={coreMetrics?.publishedToday ?? overview?.contentsPublished ?? 0} />
```

#### 2.1.3 根因分析

| 子问题 | 根因 | 专家结论 |
|--------|------|----------|
| 快捷入口需隐藏 | 产品需求变更，当前代码硬编码了 3 个入口 | **简单 UI 调整**，注释或移除对应数组元素即可 |

#### 2.1.4 与 PRD 真源对齐

PRD §1.3 Dashboard 要求：
> "+**互动量区间**预演 + 智能选题 + 昨日战报；禁止展示「流量池层级」为预测结论"

- **"昨日战报"**：PRD 明确列为 Dashboard 增强需求，但用户当前要求隐藏。经专家组研判，用户意图是**当前实现尚未达到可用标准**（数据为 Mock 或口径不准），故先隐藏，待 DataAnalyst 模块完善后再开放。
- **"导入实际数据"**：同样依赖 DataAnalyst 的 CSV 回流能力，当前为 MVP Mock 阶段，隐藏合理。

#### 2.1.5 修复方案与工作量

| 修复项 | 方案 | 工作量 | 是否进 Sprint |
|--------|------|--------|---------------|
| 隐藏 3 个快捷入口 | 注释/删除 `quickActions` 中对应条目 | 10 min | ✅ V2.7.2 可插入 |
| 数据正确性审计 | 检查 `dashboardStore.ts` 中 `fetchCoreMetrics` 与 `fetchDashboard` 的 API 返回值，统一统计口径 | 2-4 h | ✅ V2.7.2 可插入 |

- **低风险**：纯前端调整 + 数据口径核对，不涉及架构变更。

### QA-2: Task-Hub 创建页面 — 人设与故事 / 工作流 / 发布方式

#### 2.2.1 问题描述

1. **PersonaStory 无法选择**：【人设与故事】中 `PersonaStory` 下拉框无法选择；
2. **当前节点无法选择**：`当前节点` 下拉框无法选择；
3. **工作流模板执行正确性**：选中工作流模板后，提交任务是否能正确驱动工作流执行；
4. **执行方式缺"循环执行"**：【发布方式】中仅有"立即执行"和"定时执行"，缺少"循环执行"。

#### 2.2.2 现状与代码证据

**文件**: `apps/frontend/src/pages/TaskHubCreatePage.tsx`

**PersonaStory 选择器**（第 322-338 行）：
```tsx
<select
  value={formData.storyId}
  onChange={(e) => { updateField('storyId', e.target.value); updateField('nodeId', '') }}
  disabled={!formData.personaId || personaStories.length === 0}
>
  <option value="">选择故事线</option>
  {personaStories.map((s) => (<option key={s.id} value={s.id}>{s.name}</option>))}
</select>
```

**当前节点选择器**（第 340-353 行）：
```tsx
<select
  value={formData.nodeId}
  onChange={(e) => updateField('nodeId', e.target.value)}
  disabled={!formData.storyId || storyNodes.length === 0}
>
  <option value="">选择节点</option>
  {storyNodes.map((n) => (<option key={n.id} value={n.id}>{n.label || n.theme}</option>))}
</select>
```

**执行方式**（第 421-450 行）：仅支持 `immediate` / `scheduled` 两种 Radio 选项。

**Store API 调用**：`taskHubStore.ts`
- `fetchPersonaStories(personaId)` → `GET /api/persona-stories?persona_id=${personaId}`
- `fetchPersonaStoryNodes(storyId)` → `GET /api/persona-stories/${storyId}/nodes`
- `createTask(payload)` → `POST /api/task-hub/tasks`

#### 2.2.3 根因分析

| 子问题 | 根因 | 专家结论 |
|--------|------|----------|
| PersonaStory 无法选择 | 该选择器 `disabled` 条件为 `!formData.personaId \|\| personaStories.length === 0`。若用户未先选择 **Persona（人设）**，则 `personaId` 为空，Story 选择器被禁用。**属于交互依赖设计，非 Bug** | 交互链路正确，但需确认后端 `/api/persona-stories` 是否返回有效数据 |
| 当前节点无法选择 | 同理，依赖 `storyId`。`storyNodes` 为页面本地 state，通过 `fetchPersonaStoryNodes` 获取。**若 Story 选择器未选中，或 API 返回空数组，则节点选择器被禁用** | 同上，交互链路正确，需排查后端数据 |
| 工作流模板执行 | `createTask` 提交 `workflow_template_id` 到后端。后端 `task-hub` 服务需负责将任务插入 Celery 队列并驱动工作流引擎 | **需后端走查**：确认 `POST /api/task-hub/tasks` 是否触发 `start_workflow` 调用 |
| 缺"循环执行" | 当前设计将"循环执行"（CronJob）放在 **审核发布详情页** 和 **CronCockpitPage**，而非任务创建页 | **产品设计偏差**：用户预期在任务创建时即可配置循环执行，与当前设计不符 |

#### 2.2.4 与 PRD 真源对齐

PRD V2.7.2 Sprint 明确将 **PersonaStory 第六大基础功能** 列为 P0 任务（W14）：
> "StoryNode / PersonaStoryContext 数据模型、Story Cockpit 前端、内容生成注入链路"

- 当前代码中 PersonaStory 的选择链路已打通（前端 → Store → API），但 **V2.7.2 Sprint 状态表中该任务为 `[ ]`（未完成）**。因此后端 API 可能存在数据返回为空或接口未实现的情况。
- **"循环执行"**：PRD §Publisher 要求 "+频率阶梯 + 排版随机化"，但并未明确任务创建页必须包含 Cron 配置。当前设计将 Cron 放在发布确认阶段（ReviewPublishDetailPage），属于**设计分歧**而非技术缺陷。

#### 2.2.5 修复方案与工作量

| 修复项 | 方案 | 工作量 | 风险 |
|--------|------|--------|------|
| 排查 PersonaStory/节点数据 | 测试后端 API `/api/persona-stories` 和 `/api/persona-stories/{id}/nodes`，确认返回数据格式与前端期望一致 | 2-4 h | 若后端未实现，需扩展 Sprint 范围 |
| 增加 Story/节点空数据提示 | 当 `personaStories.length === 0` 时，显示"暂无故事线，请先前往 Story Cockpit 创建"引导文案 | 30 min | 无 |
| 工作流执行正确性审计 | 走查后端 `task-hub` 创建任务后是否调用 `workflow_engine.start_workflow()` | 4-8 h | 若未调用，涉及 Service 层调整 |
| 任务创建页增加"循环执行" | 在 Step 3（发布确认）增加 Cron 配置 UI，与审核发布详情页保持一致；或统一 Cron 配置入口至任务创建页 | 4-8 h | **产品设计决策**：需用户确认 Cron 入口统一策略 |

#### 2.2.6 待用户决策

1. **Cron 配置入口**：是否将"循环执行"从审核发布阶段前移至任务创建阶段？
2. **PersonaStory W14 任务**：是否将 Story Cockpit 后端数据填充提前至当前修复窗口？

#### 2.3.3 根因分析

| 子问题 | 根因 | 偏差等级 |

| Agent摘要数据 | `compliance_result` / `prediction_result` / `quality_score` 来自 `task.prompt_variables`。若任务创建时未注入这些字段，或工作流未产出，则显示为空对象 | 🟡 P1 |

| 再次生成 | 前端调用 `regenerateContent` → `POST /api/.../regenerate`。后端需走查该端点是否调用 `workflow_engine.start_workflow()` 并传入正确的模板参数 | 🔴 P0 |

| 正文富文本 | 当前使用 `MarkdownBody` 组件，支持 Markdown 编辑，**非富文本（WYSIWYG）** | 🟡 P1 |
| 发布确认 UI | 当前 UI 包含 scheduled + cron，与用户要求的"仅立即发布"不符 | 🟡 P1 |
| 真正发布 | 需验证后端 `confirm_publish` 是否真正调用 Publisher 服务执行发布，还是仅更新数据库状态 | 🔴 P0 |

#### 2.3.4 与 PRD 真源对齐

> "+频率阶梯 + 排版随机化；时段可接竞争系数"

> "禁止展示「流量池层级」为预测结论"

- **"互动预演"**：PRD 已废弃 L0-L5 流量池层级预测，改为**互动量区间**（宽先验）。当前 `prediction_result.engagement_interval` 的实现符合 PRD 方向，但需确认后端是否返回区间数据。
- **封面比例**：PRD 未明确要求图片比例选择器，属于**产品体验增强需求**。
- **发布确认简化**：用户要求去掉定时/循环，与当前设计冲突。专家组认为，若当前阶段发布以"人工审核后立即发布"为主流程，简化 UI 合理；但 Celery 定时发布能力已建成，完全移除会造成功能回退。建议**保留后端能力，前端按用户角色/配置动态显示**。

**后端数据服务**：`apps/backend/src/services/data_analyst_service.py`
- MVP 阶段使用 **Mock 数据生成**，`DashboardData` 各字段可能因生成逻辑缺陷返回 `None`。
- 后端 `data_analyst_service.py` 第 70 行起 `generate_data_report` 函数模拟数据时，未对所有字段做非空保证。

#### 2.4.3 根因分析

**直接根因**：前端假设后端返回的数值字段**必定为有效数字**，未做防御式编程。当后端返回 `null`、`undefined` 或字段缺失时，调用 `.toFixed()`、`.toLocaleString()`、乘法运算符会抛出 `TypeError`，导致 React 渲染崩溃。

**深层根因**：
1. 前后端类型契约不一致：前端 `DashboardData` 接口声明字段为 `number`，后端实际可能返回 `None`；
2. 后端 Mock 数据生成逻辑未做非空校验；
3. 前端缺少 `ErrorBoundary` 拦截渲染错误。

#### 2.4.4 与 PRD 真源对齐

PRD §1.2 缺失模块：
> "**DataAnalyst** | 文档2 §8.9 | Phase 1 | 视代码基线 | 🔴 P0"

PRD 明确 DataAnalyst 为 P0 模块，当前处于 MVP Mock 阶段。崩溃问题不阻碍架构方向，但**严重阻碍用户测试**。

#### 2.4.5 修复方案与工作量

| 修复项 | 方案 | 工作量 |

| 前端防御式编程 | 所有数值字段使用可选链 + 默认值：`dashboard.avgCoverage?.toFixed(1) ?? '0'` | 1-2 h |

## 三、跨问题综合影响分析

### 3.1 数据流一致性风险

QA-1（Dashboard 数据）、QA-3（Agent 摘要）、QA-4（DataAnalyst 报表）均涉及**前后端数据契约不一致**问题。核心根因是：

- 后端多个 Service（`dashboard_service`、`data_analyst_service`、`review_publish_service`）各自维护 Mock 数据生成逻辑；
- 缺少统一的 **BFF（Backend-for-Frontend）聚合层**或 **Schema 校验中间件**。

**建议**：引入轻量级运行时校验（如 zod / pydantic）在 API 边界做数据契约检查，避免脏数据穿透到前端导致崩溃或展示错误。

### 3.2 工作流驱动链路风险

QA-2（TaskHub 工作流执行）和 QA-3（再次生成）均涉及 **Workflow Engine 的触发正确性**。当前架构中：

- `task-hub` 负责任务创建；
- `workflow-engine` 负责工作流执行；
- `review-publish-center` 负责审核与发布。

**关键风险点**：`POST /api/task-hub/tasks` 创建任务后，是否**同步或异步**触发 `workflow_engine.start_workflow()`？若未触发，则整个"选题→生成→合规→发布"的核心闭环断裂。

**建议**：立即审计 `task-hub` 和 `review-publish-center` 的后端 Service 代码，确认工作流触发点。

### 3.3 发布链路端到端风险

QA-3（确认发布）和 QA-5（AccountPool 配额/状态机）共同构成**发布能力的最后 100 米**。当前状态：

- 前端有发布确认 UI；
- 后端有 `publisher_service` 和 `create_publish_task`；
- **但** 缺少账号配额拦截、状态机限流、代理健康检查。

这意味着即使前端点击"确认发布"，后端也可能因为**缺少防护机制**而导致：
- 同一账号超频发布被平台风控；
- 代理失效导致发布失败；
- 健康分低账号继续发布加剧风险。

| 问题 | 现状 | 专家结论 |
|------|------|----------|

`AGENTS.md` §5.3 需求对齐约束：
