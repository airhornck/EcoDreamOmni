## 〇、需求总览与现状差距

| 用户原始需求 | 现状基线 | 差距判定 | 建议动作 | 建议层级（Function/Agent/Skill） |
|-------------|---------|---------|---------|-------------------------------|
| 1. 封面AI生成（大字报/宠物图文/吸睛） | ImageForge仅有「配置」能力（layout选型+素材推荐），**无AI生成** | 🔴 核心能力缺失 | 扩展ImageForge：增加AIGC封面生成子系统 | **Function**: 任务表+风格预设配置；**Agent**: ImageForge AIGC链路；**Skill**: `image_generate_cover` (L2) |
| 2. 人群标签、关键词库与素材库 | AssetPool已支持素材管理（上传/标签/推荐/版权），但**无独立人群标签体系、无关键词库** | 🟡 部分缺失 | 新增TagPool（人群标签+关键词库），与AssetPool联动 | **Function**: audience_tags + keyword_pool ORM；**Agent**: 标签推荐+关键词扫描；**Skill**: `audience_tag_match` + `keyword_recommend` + `keyword_compliance_scan` (L2) |
| 3. 1:1复制笔记→生成内容模板→选取模板 | ContentForge支持Persona注入+AIPL模板生成，但**无「笔记克隆→模板化」链路** | 🟡 流程缺失 | 新增ContentTemplate模块：笔记结构克隆→模板萃取→模板商店→生成时注入 | **Function**: content_templates ORM；**Agent**: 结构萃取+模板应用生成（ContentForge扩展）；**Skill**: `content_template_extract` + `content_template_apply` + `content_template_recommend` (L2) |
| 4. 工作流中生成前可编辑prompt、生成后可调整 | Human-in-the-Loop支持生成后revise（修订），PromptRegistry支持模板版本管理，但**无「生成前prompt预览编辑」节点** | 🟡 体验缺口 | 在Workflow Engine中新增`PRE_GENERATION_EDIT`节点类型；ContentForge API增加`prompt_preview`模式 | **Function**: prompt_edit_history 审计表；**Agent**: PRE_GENERATION_EDIT节点+Prompt预览组装+Revise增强；**Skill**: `prompt_preview` + `prompt_edit_generate` (L2) |

**总体判定**：4项需求均落在现有架构可扩展范围内，**不触发架构调整**，属于「能力增强 + 流程补齐」级别变更。与V2.7.2 Sprint目标一致（PersonaStory + LLM Hub精简 + Human-in-the-Loop弹性）。

## 一、需求1：AI封面生成引擎（ImageForge增强）

### 1.1 需求定义

运营人员在内容生产流程中，需要系统为每篇笔记**自动生成符合平台调性的封面图**，覆盖以下场景：

| 场景 | 平台 | 风格要求 | 吸睛要素 |
|------|------|---------|---------|
| 大字报 | 小红书/抖音 | 高饱和色块、粗体文字、emoji点缀 | 数字冲击（"3个""90%"）、痛点词（"避坑""后悔"） |
| 宠物图文 | 小红书/视频号 | 萌宠实拍感、暖色调、手写字体 | 宠物表情特写、互动引导（"你家猫也这样吗？"） |
| 对比图 | 全平台 | Before/After分栏、箭头标注 | 视觉反差、结果可视化 |
| 清单图 | 小红书 | 条目编号、勾选框、清爽排版 | 信息密度高、可读性强 |

**合规红线**（继承PlatformRule L1/L2）：
- 禁止生成处方药包装、医疗器械、兽医处方笺图像
- 禁止生成「治愈率100%」「绝对有效」等疗效承诺文字
- 商业合作内容须保留「合作/体验」标注区域（由发布前ComplianceGuard二次校验）

### 1.2 架构方案

#### A. 扩展ImageForge数据模型（Function层）

> **层级判定**: Function层 — 新增数据库ORM模型，负责封面生成任务的基础数据管理。Agent禁止直接操作此表，须通过Function层Service接口。

```python
# 新增：image_generation_jobs 表（ORM）
class ImageGenerationJob(Base):
    __tablename__ = "image_generation_jobs"
    
    id: UUID = Column(UUID(as_uuid=True), primary_key=True, default=uuid4)
    content_draft_id: str = Column(String(64), ForeignKey("content_drafts.id"), nullable=False)
    account_id: str = Column(String(64), ForeignKey("account_pool.id"), nullable=False)
    
    # 生成参数
    generation_type: str = Column(String(32), default="cover")  # cover / body / carousel
    style_preset: str = Column(String(32), default="pet_cute")  # 见下方风格库
    platform: str = Column(String(16), default="xhs")  # xhs / douyin / wechat
    
    # 内容输入
    title_text: str = Column(Text)           # 封面主标题（大字报用）
    subtitle_text: str = Column(Text)        # 副标题/标签
    pet_subject: str = Column(String(128))   # 宠物主体描述（如"橘猫打哈欠"）
    color_theme: str = Column(String(32))    # 色板：warm / cool / neon / pastel
    
    # 素材引用（关联AssetPool）
    reference_asset_ids: list = Column(JSON, default=list)  # 参考图/素材ID列表
    
    # 生成结果
    generated_image_url: str = Column(Text)   # 生成图URL
    generated_alt_text: str = Column(Text)    # 无障碍描述（自动产出）
    
    # 状态机
    status: str = Column(String(16), default="pending")  # pending / generating / completed / failed / rejected
    
    # 审核
    compliance_checked: bool = Column(Boolean, default=False)
    compliance_result: dict = Column(JSON, default=dict)  # {passed: bool, issues: [...]}
    
    created_at: DateTime = Column(DateTime(timezone=True), default=now)
    updated_at: DateTime = Column(DateTime(timezone=True), default=now, onupdate=now)
```

#### C. 生成链路设计（Agent层 + Skill层）

> **层级判定**: 
> - **Agent层**: ImageForge Agent执行业务编排（prompt组装 → AIGC调用 → 合规初审 → 输出）
> - **Skill层**: `image_generate_cover` (L2) 封装为标准化Skill，供Workflow/Human-in-the-Loop调用
> - **Function层**: 任务状态读写、风格预设查询、结果URL存储

## 二、需求2：人群标签、关键词库与素材库

### 2.1 需求定义

建立三层内容资产体系，支撑内容生成的「精准靶向」与「素材复用」：

| 层级 | 名称 | 定义 | 使用场景 |
|------|------|------|---------|
| L1 | **人群标签（AudienceTag）** | 目标受众维度标签：人群属性 + 宠物生命周期 + 消费特征 | 内容定向、账号人设匹配、投放受众包导出 |
| L2 | **关键词库（KeywordPool）** | 平台搜索/话题关键词：高搜索量词、长尾词、竞品词、禁忌词 | 标题优化、话题标签推荐、SEO覆盖、合规拦截 |
| L3 | **素材库（AssetPool）** | 已存在的图片/视频/音频素材（含AI生成、实拍、Stock） | 封面配图、正文插图、视频素材、版权合规 |

**现状**：AssetPool（L3）已在W14完成ORM化，功能成熟。L1和L2为新增模块。

## 三、需求3：笔记克隆与内容模板系统（ContentTemplate）

### 3.1 需求定义

运营人员手动复制一篇平台爆款笔记的**结构**（非全文抄袭），系统将其抽象为可复用的内容模板，后续生成新内容时可选取模板作为结构骨架。

**核心流程**：
1. **克隆输入**：运营粘贴对标笔记URL或文本（标题+正文+标签）
2. **结构萃取**：AI分析笔记的hook结构、段落节奏、CTA位置、标签策略
3. **模板化**：生成结构化模板（可含变量占位符）
4. **模板选取**：生成新内容时，从模板库选取匹配模板注入

**合规边界**（法务红线）：
- 仅萃取「结构模式」（如"痛点开场→3点干货→互动提问"），**不复制原文案**
- 对标笔记来源须为**公开内容**，禁止爬取私密/未发布内容
- 模板须标注"结构灵感来源"（平台+作者ID），满足原创性声明要求

### 3.2 架构方案

```python
# Agent层：ContentForge扩展能力
def extract_template_from_note(note_text: str, title: str, tags: list) -> dict:
    """
    从对标笔记中提取结构模板。
    MVP阶段使用规则引擎 + LLM-as-Extractor（轻量调用）。
    """
    # 步骤1: 规则预分析
    structure_signals = {
        "has_numbers_in_title": bool(re.search(r'\d+', title)),
        "has_question_in_hook": note_text.strip().startswith(("?", "？", "你", "有没有", "为什么")),
        "list_pattern": detect_list_pattern(note_text),  # 数字列表 / 符号列表 / 无列表
        "cta_position": "end" if note_text.rstrip().endswith(("?", "！", "!")) else "middle",
        "emoji_density": count_emoji(note_text) / len(note_text),
        "has_disclaimer": "不构成" in note_text or "仅供参考" in note_text,
    }
    
    # 步骤2: LLM结构化萃取（单次调用，低成本模型）
    extraction_prompt = f"""
    分析以下小红书笔记的结构模式，输出JSON：
    - hook类型（痛点提问/悬念/数字冲击/情感共鸣/其他）
    - 正文段落类型列表（列表项/故事叙述/对比/引用/其他）
    - CTA类型（互动提问/关注引导/行动号召/其他）
    - 是否有免责声明
    
    标题：{title}
    正文：{note_text[:800]}...
    标签：{tags}
    """
    
    # 步骤3: 组装模板结构（规则映射）
    template = map_to_template_schema(structure_signals, llm_output)
    return template

## 四、需求4：工作流Prompt可编辑闭环（Human-in-the-Loop增强）

### 4.1 需求定义

当前工作流中，ContentForge节点的prompt由系统根据Persona/BrandKnowledge/模板自动组装，运营人员**无法在生产流程中干预prompt内容**。用户希望在两个关键节点获得控制权：

| 干预节点 | 当前状态 | 需求 | 价值 |
|---------|---------|------|------|
| **生成前（Pre-Generation）** | prompt黑盒组装后直接送LLM | 运营可预览、编辑、增强prompt后再触发生成 | 提升生成质量、注入临时创意、修正上下文偏差 |
| **生成后（Post-Generation）** | Human-in-the-Loop支持revise（修改变量） | 在revise时**可同步修改prompt**并重新生成 | 精准迭代、快速调优 |

    template_name: str
    
    # 组装后的prompt（分块展示，便于理解）
    system_prompt: str          # 系统指令（Persona voice + 方法论约束 + 合规红线）
    user_prompt: str            # 用户指令（topic + 结构模板 + 关键词约束）
    context_injection: list     # 上下文注入项（BrandKnowledge RAG结果 + PersonaStory前情回顾）
    
    # 可编辑区（运营可修改的部分）
    editable_sections: dict     # {"user_prompt": True, "system_prompt": False, ...}
    
    # 模型参数
    model_info: dict            # {provider, model_name, temperature, max_tokens}
    estimated_cost: float       # 预估成本（按token数估算）
    
    # 元数据
    assembled_at: str
    assembled_by: str           # "system" / 运营可覆盖


class PromptEditRequest(BaseModel):
    """运营对prompt的编辑请求"""
    
    # 编辑内容
    system_prompt_override: Optional[str] = None    # 覆盖系统prompt（需高级权限）
    user_prompt_override: Optional[str] = None      # 覆盖用户prompt
    additional_context: Optional[str] = None        # 追加上下文（不替换原有内容）
    temperature_override: Optional[float] = None    # 覆盖temperature
    
    # 编辑理由（审计用）
    edit_reason: str = ""
    
    # 操作
    action: str = "edit_and_generate"  # edit_and_generate / edit_and_save / discard
```

#### C. 生成后Prompt修订增强（Agent层 — Human-in-the-Loop扩展）

> **层级判定**: Agent层 — Human-in-the-Loop Agent的业务逻辑扩展。revise时调用`prompt_edit_generate` Skill（L2）执行重生成。

扩展Human-in-the-Loop的`revise_task`，支持**同步修改prompt**后重新生成：

```python
class ReviseRequestV2(BaseModel):
    operator: Optional[str] = None
    target_node_index: int = 3          # 默认回到正文生成节点
    
    # 现有：修改变量
    revised_variables: Dict[str, Any] = {}
    
    # 新增：修改prompt并重新生成
    prompt_adjustment: Optional[PromptEditRequest] = None  # 调整prompt
    regenerate_with_new_prompt: bool = False               # 是否用新prompt重生成
    
    reason: str = ""
```

**修订流程**：
```
运营在审核面板点击[修订]

```

### 7.1 架构分层原则（评审基准）

| 层级 | 定义 | 核心特征 | 示例 | 架构红线 |
|------|------|---------|------|---------|
| **Function** | 基础数据/能力层 | 操作数据库、基础CRUD、配置管理、原子查询 | AssetPool、BrandKnowledge、VetDrugDB、PlatformRule | Agent**禁止**直接操作数据库，必须通过Function层 |
| **Agent** | 业务逻辑智能单元 | 封装业务逻辑、调用LLM/ML、复杂编排、工作流节点 | ContentForge、ComplianceGuard、Publisher、PoolPredictor、TrendScout | Agent**禁止**直接操作数据库；通过Orchestrator调度 |
| **Skill** | 可复用能力单元 | 标准化run(ctx)接口、注册于SkillHub（L1-L4）、可被Agent调用 | content_generate、compliance_check、engagement_predict | Skill代码通过Tool Registry统一schema；L2-L4执行受限沙箱 |

**分层判定原则**:
1. 凡涉及**数据库ORM定义/基础CRUD/配置存储/审计日志** → **Function层**
2. 凡涉及**LLM调用/ML推理/业务编排/工作流调度/智能决策** → **Agent层**
3. 凡是**可独立复用的能力片段**，有标准化输入输出，需被多Agent共享 → **封装为Skill（注册于SkillHub）**
4. **同一能力可同时存在于Agent和Skill**：Agent实现业务逻辑，Skill封装可复用接口

| 角色 | 结论 | 建议 |
|------|------|------|

| **架构** | ✅ 全部基础数据管理落入Function层，符合红线 | AudienceTag和KeywordPool建议独立Function文件（非挤在AssetPool中） |
| **算法** | ✅ 推荐算法MVP用规则引擎足够 | trending排序可用简单加权公式（搜索量×时效系数×平台系数），Phase 2再考虑ML |

| **运营** | ⚠️ 维护成本需明确SOP | 建议：关键词库由运营每周review一次；人群标签由产品+运营共建，季度更新 |

### 7.4 需求3：笔记克隆与内容模板 — 专家评审

#### 7.4.1 各组件分层判定

| 组件 | 建议层级 | 判定理由 | 与现有系统关系 |
|------|---------|---------|-------------|
| `content_templates` 数据模型/ORM + CRUD + 版本管理 | **Function** | 模板的基础数据管理，含structure JSON存储 | 新增Function层，与PromptRegistry同级 |
| 模板效果统计数据聚合查询 | **Function** | 按template_id聚合usage_count/success_rate等 | 纯数据查询，Function层实现 |
| 结构萃取算法（规则引擎+轻量LLM） | **Agent** | 涉及NLP分析、LLM调用（萃取结构模式）、智能决策 | **ContentForge Agent的扩展能力**，非独立Agent |
| 模板应用生成（按模板结构约束LLM填充） | **Agent** | 涉及prompt组装、LLM调用、结构约束注入 | **ContentForge Agent的扩展能力** |
| 模板推荐排序（按topic+platform+persona匹配） | **Agent** | 涉及匹配算法 | ContentForge Agent的子能力或独立轻量服务 |
| `content_template_extract` 可复用能力 | **Skill (L2)** | 从笔记文本萃取结构模板，标准化接口 | SkillHub注册：input={title, body, tags}，output={structure, confidence} |
| `content_template_apply` 可复用能力 | **Skill (L2)** | 使用模板生成内容，标准化接口 | SkillHub注册：input={template_id, topic, variable_values}，output={title, body, tags} |
| `content_template_recommend` 可复用能力 | **Skill (L2)** | 推荐最适合的模板 | SkillHub注册：input={topic, platform, persona_id}，output={ranked_templates} |

#### 7.4.2 专家结论

| 角色 | 结论 | 建议 |
|------|------|------|
| **产品** | ✅ ContentTemplate与MarketingMethodology AIPL互补 | ContentTemplate管「结构模式」，Methodology管「阶段约束」，两者可叠加使用 |
| **架构** | ⚠️ **关键红线检查** | 结构萃取算法当前草案直接import ContentForge Service，**违反Agent不直接操作DB/Service红线**。必须通过API调用或Agent消息传递 |
| **算法** | ✅ MVP规则+轻量LLM足够 | 萃取prompt建议用低成本模型（deepseek-chat/gpt-4o-mini），单次成本<¥0.05 |
| **法务** | ✅ 结构萃取合规边界清晰 | 仅萃取结构模式（hook类型/段落节奏/CTA位置），不复制原文案；需标注灵感来源 |
| **运营** | ⚠️ 模板运营SOP需建立 | 建议：萃取后模板默认draft状态，需运营审核激活；月度清理成功率<30%的模板 |

**分层调用链**:
```
运营前端
    ├──► 调用 content_template_extract Skill（Skill层 L2）
    │       └──► ContentForge Agent（Agent层）执行萃取
    │               └──► 写入 content_templates（Function层，状态=draft）
    └──► 调用 content_template_apply Skill（Skill层 L2）
            └──► ContentForge Agent（Agent层）按模板结构生成
                    ├──► 查询 PromptRegistry（Function层）获取prompt模板
                    ├──► 查询 PersonaPool（Function层）获取人设
                    └──► 调用 LLM Hub（通过Agent编排）生成内容
```

**⚠️ 架构红线整改项**: 方案3.2节中的`extract_template_from_note()`函数（结构萃取）和`generate_with_persona()`调用，在Skill实现中**禁止直接import底层Service**。必须通过以下方式之一：
- **方式A（推荐）**: Skill的`run(ctx)`中通过HTTP API调用ContentForge的`/content-generate`接口
- **方式B**: 通过Agent Orchestra的消息传递机制，由Orchestrator调度ContentForge Agent执行

### 7.5 需求4：工作流Prompt可编辑闭环 — 专家评审

#### 7.5.1 各组件分层判定

| 组件 | 建议层级 | 判定理由 | 与现有系统关系 |
|------|---------|---------|-------------|
| PromptRegistry模板CRUD/版本化（已有P2） | **Function** | Prompt模板的基础数据管理 | 现有Function层，无需改动 |
| Prompt编辑历史记录存储 | **Function** | 审计数据的持久化 | 新增表 `prompt_edit_history` 或扩展PromptRegistry版本机制 |
| Prompt预览组装（读取模板+Persona+BrandKnowledge+上下文注入） | **Agent** | 涉及多数据源查询+智能组装逻辑 | WorkflowEngine / ContentForge Agent的前置逻辑 |
| 生成前Prompt编辑工作流节点（PRE_GENERATION_EDIT） | **Agent** | WorkflowEngine的节点调度逻辑 | 扩展WorkflowEngine节点类型 |
| 生成后Revise支持prompt调整 | **Agent** | Human-in-the-Loop Agent的业务逻辑扩展 | 扩展Human-in-the-Loop Agent |
| 带prompt编辑的内容生成 | **Agent** | 涉及LLM调用、参数覆盖、上下文管理 | ContentForge Agent的扩展能力 |
| `prompt_preview` 可复用能力 | **Skill (L2)** | 预组装prompt并返回结构化预览 | SkillHub注册：input={topic, platform, persona_id, template_id}，output={PromptPreview} |
| `prompt_edit_generate` 可复用能力 | **Skill (L2)** | 使用编辑后的prompt生成内容 | SkillHub注册：input={base_preview_id, prompt_edit}，output={GeneratedContent} |

#### 7.5.2 专家结论

| 角色 | 结论 | 建议 |
|------|------|------|
| **产品** | ✅ Pre-Generation Edit节点设计解决「黑盒」痛点 | 建议该节点默认timeout=24h，运营不编辑则自动跳过（不阻塞流程） |
| **架构** | ✅ PRE_GENERATION_EDIT作为独立节点类型合理 | 该节点本质上是HUMAN_APPROVAL的变体（审批对象是prompt而非内容），建议复用HUMAN_APPROVAL的底层实现，减少重复代码 |
| **算法** | ✅ PromptPreview结构分层清晰 | system_prompt/user_prompt/context_injection三分法便于运营理解；建议增加「diff对比」展示编辑前后变化 |
| **法务** | ✅ 编辑历史保存满足审计要求 | 建议保存周期≥2年，与ComplianceGuard证据链留存要求一致 |
| **工程** | ⚠️ 历史版本存储策略需确定 | 推荐方案：独立`prompt_edit_history`表（每条编辑一条记录），不按content_draft冗余存储。该表仅审计查询用，不参与业务逻辑 |

**分层调用链**:
```
WorkflowEngine（Orchestrator调度）
    ├──► 节点2: ContentForge Agent（框架生成）
    ├──► 节点3: PRE_GENERATION_EDIT（Agent层 — HUMAN_APPROVAL变体）
    │       └──► 调用 prompt_preview Skill（Skill层 L2）
    │               └──► 查询 PromptRegistry（Function层）
    │               └──► 查询 PersonaPool（Function层）
    │               └──► 查询 BrandKnowledge（Function层）
    │       └──► 等待运营编辑 → 写入 prompt_edit_history（Function层）
    ├──► 节点4: ContentForge Agent（正文生成，使用编辑后prompt）
    │       └──► 调用 prompt_edit_generate Skill（Skill层 L2）
    └──► 节点8: Human-in-the-Loop Agent（审核，支持revise+prompt调整）
            └──► 调用 prompt_edit_generate Skill（Skill层 L2）重生成
```

### 7.7 实施优先级重排（基于分层判定后）

根据专家评审，优先级调整如下：

| 轮次 | 任务 | 涉及层级 | 工期 | 优先级调整理由 |
|------|------|---------|------|--------------|
| **第1轮（W15）** | TagPool Function层：audience_tags + keyword_pool 数据模型 + API | Function | 3天 | **P0不变** — Function层是Agent和Skill的基座，必须先有数据 |
| | ContentTemplate Function层：content_templates 数据模型 + CRUD API | Function | 2天 | **P0不变** — 同上 |
| | PromptRegistry扩展：prompt_edit_history 表 | Function | 1天 | **提前至P0** — 需求4的审计基座 |
| **第2轮（W15-W16）** | ContentForge Agent扩展：模板应用生成 + 结构萃取 | Agent | 4天 | **P0不变** — 核心业务能力 |
| | SkillHub注册：首批4个Skill（image_generate_cover + audience_tag_match + keyword_recommend + content_template_apply） | Skill | 2天 | **提前至P0** — Skill是Agent调用的标准化接口，需在Agent开发前注册 |
| **第3轮（W16）** | ImageForge Agent扩展：AIGC封面生成链路 | Agent | 4天 | **P1** — 依赖Function层和Skill层就绪 |
| | WorkflowEngine扩展：PRE_GENERATION_EDIT节点 | Agent | 3天 | **P1** — 体验增强，非阻塞 |
| | Human-in-the-Loop扩展：revise支持prompt调整 | Agent | 2天 | **P1** — 体验增强 |
| | SkillHub注册：第二批4个Skill（content_template_extract + content_template_recommend + prompt_preview + prompt_edit_generate） | Skill | 1天 | **P1** — 补充完善 |
| **第4轮（W17）** | 前端集成 + E2E测试 + 验收 | 全层 | 5天 | — |

**关键调整说明**:
1. **Function层全部提前至第1轮**：所有Agent和Skill都依赖Function层的数据基座，必须先完成
2. **Skill注册提前至第2轮**：Skill是Agent调用的标准化契约，SkillHub注册应在Agent开发前完成，确保Agent调用时有稳定的Skill接口
3. **需求4（Prompt可编辑）整体后移至P1**：其核心价值是「体验增强」，不阻塞内容生成的核心链路；且依赖需求2和3的Function层就绪
