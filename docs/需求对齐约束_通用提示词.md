# 需求对齐约束 — 通用提示词

> **用途**：在处理任何 Bug 修复或功能需求前，加载本约束作为强制检查清单。
> **位置**：粘贴于每段具体需求的末尾，作为 Agent 执行的前置条件。
> **真源文档**：
> - PRD：`EcoDream_Omni_PRD_v2_对齐核心方案.md`（V2.7.2 / V3.1）
> - 文档总纲：`docs/文档总纲.md`
> - 数据词典：`docs/数据词典/00-数据词典总纲.md`
> - 工程纪律：`AGENTS.md`

---

## 一、PRD 业务真源对齐（强制）

在处理上述需求前，必须先回答以下问题：

1. **该需求是否存在于 PRD 中？**
   - 如果是 Bug 修复：Bug 描述是否与 PRD 预期行为一致？修复后是否符合 PRD 定义？
   - 如果是功能需求：该功能是否在 PRD 的某个章节中有定义？
   - 如果 PRD 中**未定义**或**与 PRD 存在偏差**：
     - ❗ **这是新增需求或偏差需求**
     - 必须在实施前，将需求描述补充到 `EcoDream_Omni_PRD_v2_对齐核心方案.md` 的对应章节
     - 同步更新 `docs/PRD偏差报告.md` — 如消除了某项偏差，更新其状态为 ✅ 已修复

2. **需求范围是否超出当前 Phase？**
   - PRD 将功能划分为 Phase 1（MVP）、Phase 1.5、Phase 2
   - 如果需求属于 Phase 2+ 但当前处于 Phase 1：标记为 🟢 P2 远期规划，**仅做预留接口，不实现完整逻辑**

## 二、技术架构对齐（强制）

### 2.1 调用关系维护原则

- ✅ **允许**：仅维护现有调用关系（修改函数体、修正参数、调整返回值）
- ✅ **允许**：在已有 Router 下新增端点（需同步更新数据词典）
- ✅ **允许**：在已有 Service 文件中新增纯业务逻辑函数（不涉及 DB）
- ✅ **允许**：在已有 Function 文件中新增 ORM CRUD 函数
- ❌ **禁止**：新增独立 Service 文件而不通过 Function 层访问 DB
- ❌ **禁止**：在 Agent/Skill 层直接 `import AsyncSession` 或 `import AsyncSessionLocal`
- ❌ **禁止**：修改 Function/Agent/Celery 三层职责边界

### 2.2 架构调整红线

如果需求涉及以下任何一项，**必须立即停止实施，请求用户严格审核**：

| 架构调整类型 | 示例 | 触发条件 |
|-------------|------|----------|
| 新增 API Router | 新增 `api/xxx.py` 文件 | 需要新增业务模块 |
| 新增 Service 文件 | 新增 `services/xxx.py` | 需要全新业务逻辑层 |
| 新增 ORM 模型 | 新增 `models/xxx_orm.py` | 需要新数据库表 |
| 修改路由前缀 | 修改 `APIRouter(prefix="...")` | 影响前端所有调用路径 |
| 修改数据库字段 | 新增/删除/修改 ORM 列 | 需要 Alembic 迁移 |
| 引入新的外部依赖 | 新的 pip/npm 包 | 影响 Docker 构建 |
| 修改 Celery 任务签名 | 修改 `@shared_task` 参数 | 影响已有 Worker |
| 修改前端路由结构 | 修改 `App.tsx` 路由 | 影响用户访问路径 |
| 修改全局状态设计 | 修改 Zustand Store 结构 | 影响所有页面 |

**暂停后的流程**：
1. 向用户说明：「该需求涉及架构调整，具体为：xxx」
2. 提供架构影响范围评估（会影响哪些文件、哪些调用链）
3. 等待用户明确批准后，再继续实施

## 三、文档连锁更新（强制）

### 3.1 所有变更必须从总纲可索引

如果修改了任何文档，必须确保 `docs/文档总纲.md` 中的索引是最新的：

| 修改类型 | 必须更新的文档 |
|----------|---------------|
| 新增 API 端点 | `数据词典/02-后端API路由层.md` + `数据词典/05-模块调用关系矩阵.md` |
| 新增 Service 函数 | `数据词典/03-后端Service层.md` + `数据词典/05-模块调用关系矩阵.md` |
| 新增 ORM 模型/字段 | `数据词典/01-数据库模型层.md` |
| 新增前端 Store | `数据词典/04-前端Store与路由.md` + `数据词典/05-模块调用关系矩阵.md` |
| 新增前端页面/路由 | `数据词典/04-前端Store与路由.md` |
| 新增/修改 PRD 章节 | `EcoDream_Omni_PRD_v2_对齐核心方案.md` + `PRD偏差报告.md` |
| 新增独立文档 | `文档总纲.md` — 在对应分类中补充一行 |
| 文档过期/归档 | `文档总纲.md` — 移至「已归档」小节 |

### 3.2 禁止文档孤岛

- ❌ 不允许创建不在 `文档总纲.md` 中登记的文档
- ❌ 不允许修改文档后不更新总纲索引
- ❌ 不允许存在与总纲描述不一致的文档内容

## 四、变更记录登记（强制）

### 4.1 变更记录格式

实施完成后，必须在 `docs/变更记录/YYYY-MM-DD/` 下创建当天的变更总结文件：

```
docs/变更记录/2026-05-27/开发变更总结_2026-05-27.md
```

如果当天已有变更记录文件，追加到该文件末尾。

### 4.2 变更记录必须包含的内容

| 章节 | 必填内容 |
|------|----------|
| **需求概述** | 本次修复的 Bug 或实现的需求的一句话描述 |
| **PRD 对齐说明** | 该需求对应 PRD 哪个章节，是否为新增/偏差需求 |
| **修改文件清单** | 精确到文件名 + 修改类型（新增/修改/删除）+ 影响行数范围 |
| **调用关系变更** | 新增/修改/删除了哪些 API→Service→Function→ORM 链路 |
| **前端映射变更** | Store→API 调用路径是否有变化 |
| **测试验证** | 运行的测试命令及结果（通过数/失败数） |
| **文档更新清单** | 更新了哪些数据词典分册 + 总纲 + PRD |
| **风险与后续** | 回归风险、待办事项 |

### 4.3 变更记录模板

```markdown
# 开发变更总结 — YYYY-MM-DD

> **需求概述**: xxx
> **PRD 章节**: §x.x / 新增需求

---

## 修改文件清单

| 文件 | 类型 | 说明 |
|------|------|------|
| `apps/backend/src/api/xxx.py` | 修改 | 新增/修改端点 |
| `apps/backend/src/services/xxx_function.py` | 修改 | 新增/修改 CRUD |
| `apps/frontend/src/stores/xxxStore.ts` | 修改 | 新增/修改 API 调用 |

## 调用关系变更

```
前端: xxxStore.ts → fetch('/api/xxx')
后端: api/xxx.py → services/xxx_function.py → ORM
```

## 测试验证

```bash

# 结果: X passed, Y skipped
```

## 文档更新

- [x] `数据词典/03-后端Service层.md`
- [x] `数据词典/05-模块调用关系矩阵.md`
- [x] `文档总纲.md`

```

## 五、工程纪律约束（强制）

本约束与 `AGENTS.md` 的红绿灯 TDD 纪律叠加执行：

1. **Red-Green-Blue 循环不可跳过**
   - 🔴 先写复现测试（必须看到失败）
   - 🟢 最小化实现（使测试通过）
   - 🔵 重构（保持测试通过）

2. **质量门禁**
   - 后端：pytest ≥ 80% 覆盖率，0 mypy 错误
   - 前端：vitest 全绿，tsc --noEmit 0 错误

3. **禁止事项**
   - ❌ 不写测试直接实现功能
   - ❌ 跳过红灯阶段
   - ❌ 一次修改多个不相关模块
   - ❌ 删除或修改他人写的测试
   - ❌ 修改后不更新数据词典和总纲

## 六、快速检查清单（每段需求执行前默念）

```
□ 该需求在 PRD 中有定义吗？没有 → 补充到 PRD + 更新偏差报告
□ 修改仅维护调用关系吗？涉及架构 → 暂停，请求用户审核
□ 涉及哪些文件？列出清单
□ 涉及哪些文档？准备连锁更新
□ 测试策略是什么？先写复现测试
□ 变更记录格式准备好了吗？按模板填写
```

如果以上任一项无法回答，**暂停实施，向用户澄清后再继续**。
