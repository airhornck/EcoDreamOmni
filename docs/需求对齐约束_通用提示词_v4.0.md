# 需求对齐约束 — 通用提示词 v4.0

> **用途**：在处理任何 Bug 修复、功能需求或设计变更前，加载本约束作为强制检查清单。
> **位置**：粘贴于每段具体需求的末尾，作为 Agent 执行的前置条件。
> **版本基线**：v4.0 AI-Native Workspace（架构评审后对齐版）
> **真源文档**：
> - PRD：`EcoDream_Omni_PRD_v4_AI_Native_Architecture.md`
> - 文档总纲：`docs/文档总纲_v4.0.md`
> - 数据词典：`docs/数据词典_v4.0/00-数据词典总纲.md`
> - 工程纪律：`AGENTS.md`
> - 架构设计：`docs/架构设计/00-架构设计总纲.md`

## 一、PRD 业务真源对齐（强制）

在处理上述需求前，必须先回答以下问题：

### 1.1 PRD 存在性检查

1. **该需求是否存在于 PRD v4.0 中？**
   - 如果是 Bug 修复：Bug 描述是否与 PRD 预期行为一致？修复后是否符合 PRD 定义？
   - 如果是功能需求：该功能是否在 PRD 的某个章节中有定义？
   - 如果 PRD 中**未定义**或**与 PRD 存在偏差**：
     - ❗ **这是新增需求或偏差需求**
     - 必须在实施前，将需求描述补充到 `EcoDream_Omni_PRD_v4_AI_Native_Architecture.md` 的对应章节
     - 同步更新 `docs/PRD偏差报告_v4.0.md` — 如消除了某项偏差，更新其状态为 ✅ 已修复

2. **需求范围是否超出当前 Phase？**
   - PRD 将功能划分为 Phase 0（基础设施）、Phase 1（主题+导航）、Phase 2（工作台+内容生产）、Phase 3（审核发布）、Phase 4（Pipeline）、Phase 5（接入层）、Phase 6（前端层 Copilot + 实验室）
   - 如果需求属于未来 Phase 但当前处于较早 Phase：标记为 🟢 远期规划，**仅做预留接口，不实现完整逻辑**

### 1.2 专家评审触发条件（强制）

以下情况**必须**先组织专家评审（架构/前端/后端/产品 4 维度），获得综合评分 ≥ 3.0 后方可实施：

| 触发条件 | 说明 |
|---------|------|
| 新增独立页面/路由 | 如新建 `/xxx` 而非复用现有页面 |
| 新增独立数据模型 | 如新建 `ViralTemplate` 而非复用 `ContentTemplate` |
| 新增独立 API Router | 如新建 `/api/viral/*` 而非扩展现有 `/api/playground/*` |
| 修改核心交互范式 | 如从按钮面板改为 Copilot Action Cards |
| 引入新的外部依赖 | 新的 pip/npm 包，影响 Docker 构建 |
| 跨 Phase 边界 | 当前 Phase 未规划的功能提前实施 |

**专家评审流程**：
1. 输出专家评审报告到 `docs/变更记录_v4.0/YYYY-MM-DD/需求名_专家评审报告.md`
2. 报告必须包含：4 维度评分 + unanimous 共识 + 实施条件
3. 综合评分 < 3.0：必须按评审意见修改后重新评审
4. 综合评分 ≥ 3.0：按评审条件实施

## 二、前端 Mode C 合规红线（强制）

### 2.1 绝对禁止的业务按钮

工作区（Workspace Canvas）内**绝对禁止**出现以下类型的按钮：

- ❌ 保存 / 提交 / 确认
- ❌ 生成 / 新建 / 创建
- ❌ 删除 / 移除 / 清空
- ❌ 导出 / 下载 / 上传（拖拽区除外）
- ❌ 批量操作按钮

**唯一合法的操作入口**：Copilot Action Cards（右侧面板）

### 2.2 允许存在的控件

- ✅ 拖拽上传区（无上传按钮，拖拽即触发）
- ✅ 表单输入控件（input / textarea / select）
- ✅ Tab 切换导航
- ✅ 搜索/过滤输入框
- ✅ 纯展示性按钮（如「展开/折叠」）

### 2.3 Copilot Action Cards 设计规范

- 每个 Action Card 必须对应一个明确的业务操作
- Action Card 必须根据状态机动态显示/隐藏
- 禁止在 Action Card 外提供相同的业务操作入口
- 状态机：empty → input_ready → processing → processed → [后续状态]

## 三、实验室架构约束（强制）

### 3.1 实验室是容器，不是功能

- 实验室 (`/lab`) 是**能力容器页面**，不是某个具体功能
- 每个能力是独立的模块，有自己的 Store、组件、API
- 能力之间禁止直接耦合，通过 labStore 进行能力切换

### 3.2 能力扩展规范

- 新增能力必须：在 `labStore.ts` 中注册 `LAB_CAPABILITIES` 配置
- 能力画布组件必须放在 `components/lab/` 目录下
- 能力级 API 保持独立（如 `/api/playground/*` 代表「爆款笔记分析」能力）
- 未来能力默认状态为 `locked`，需单独评审后方可激活

## 四、技术架构对齐（强制）

### 4.1 调用关系维护原则

- ✅ **允许**：仅维护现有调用关系（修改函数体、修正参数、调整返回值）
- ✅ **允许**：在已有 Router 下新增端点（需同步更新数据词典）
- ✅ **允许**：在已有 Service 文件中新增纯业务逻辑函数（不涉及 DB）
- ✅ **允许**：在已有 Function 文件中新增 ORM CRUD 函数
- ❌ **禁止**：新增独立 Service 文件而不通过 Function 层访问 DB
- ❌ **禁止**：在 Agent/Skill 层直接 `import AsyncSession` 或 `import AsyncSessionLocal`
- ❌ **禁止**：修改 Function/Agent/Celery 三层职责边界
- ❌ **禁止**：Agent 间直接调用，必须走 EventBus 路由

### 4.2 架构调整红线

如果需求涉及以下任何一项，**必须立即停止实施，请求用户严格审核**：

| 架构调整类型 | 示例 | 触发条件 |
|-------------|------|----------|
| 新增 API Router | 新增 `api/xxx.py` 文件 | 需要新增业务模块 |
| 新增 Service 文件 | 新增 `services/xxx.py` | 需要全新业务逻辑层 |
| 新增 ORM 模型 | 新增 `models/xxx_orm.py` | 需要新数据库表 |
| 修改路由前缀 | 修改 `APIRouter(prefix="...")` | 影响前端所有调用路径 |
| 修改数据库字段 | 新增/删除/修改 ORM 列 | 需要 Alembic 迁移 |
| 引入新的外部依赖 | 新的 pip/npm 包 | 影响 Docker 构建 |
| 修改 Celery 任务签名 | 修改 `@shared_task` 参数 | 影响已有 Worker |
| 修改前端路由结构 | 修改 `App.tsx` 路由 | 影响用户访问路径 |
| 修改全局状态设计 | 修改 Zustand Store 结构 | 影响所有页面 |
| 修改 Mode C 范式 | 在工作区新增业务按钮 | 违反架构红线 |

**暂停后的流程**：
1. 向用户说明：「该需求涉及架构调整，具体为：xxx」
2. 提供架构影响范围评估（会影响哪些文件、哪些调用链）
3. 如涉及新增页面/模型/API，触发专家评审流程
4. 等待用户明确批准后，再继续实施

## 五、HTML 预览审核流程（强制）

### 5.1 触发条件

以下前端变更必须通过 HTML 预览审核：

- 新增/重构页面组件（Page 级别）
- 修改全局布局（WorkspaceLayout / IconNav / Header）
- 新增 Copilot Action Cards 交互
- 修改设计系统 Token（颜色 / 字体 / 间距）

### 5.2 审核流程

1. **生成 HTML 预览**：`apps/frontend/demo/page-preview/页面名.html`
2. **自检查清单**：
   - □ Mode C 合规：无业务按钮
   - □ 颜色合规：仅使用设计系统 Token（`#FFFFFF`, `#1F1F1F`, `#7C3AED`）
   - □ 布局合规：三栏布局（IconNav 52px | Canvas | Copilot 320px）
   - □ 交互合规：Copilot Action Cards 状态机正确
3. **提交审核**：将 HTML 文件路径告知用户，等待审核通过
4. **审核通过后**：方可同步到 React 代码

## 八、工程纪律约束（强制）

本约束与 `AGENTS.md` 的红绿灯 TDD 纪律叠加执行：

### 8.1 Red-Green-Blue 循环不可跳过

- 🔴 先写复现测试（必须看到失败）
- 🟢 最小化实现（使测试通过）
- 🔵 重构（保持测试通过）

### 8.2 质量门禁（Quality Gates）

| 门禁 | 阈值 | 工具 | 说明 |
|------|------|------|------|
| 测试覆盖率 | ≥ 80% | pytest / vitest | 后端用 pytest --cov，前端用 vitest |
| TypeScript 类型检查 | 0 errors | tsc --noEmit --skipLibCheck | 前端强制 |
| Python 类型检查 | 0 errors | mypy | 后端新增代码 |
| Lint | 0 errors | eslint / ruff | 前后端均强制 |
| 构建 | 必须通过 | vite build / docker build | CI 流水线 |
| HTML 预览 | 必须通过 | 人工审核 | 前端页面变更 |

### 8.3 LLM 优先，规则辅助

- 分析类需求（如爆款笔记分析）**主路径必须使用 LLM**
- 规则引擎（关键词库、结构定义）仅作为**校准层**，不得作为主分析路径
- 关键词库规模控制在 50-100 核心词（MVP），避免运营负债

### 8.4 禁止事项

- ❌ 不写测试直接实现功能
- ❌ 跳过红灯阶段
- ❌ 一次修改多个不相关模块
- ❌ 删除或修改他人写的测试
- ❌ 修改后不更新数据词典和总纲
- ❌ 在工作区新增业务按钮（违反 Mode C）
- ❌ Agent 直接访问 DB（必须通过 Function 层）
- ❌ Agent 间直接调用（必须走 EventBus）
- ❌ 绕过 HTML 预览审核直接提交前端代码
- ❌ 综合评分 < 3.0 未经修改直接实施

## 九、快速检查清单（每段需求执行前默念）

```
□ 该需求在 PRD v4.0 中有定义吗？没有 → 补充到 PRD + 更新偏差报告
□ 是否触发专家评审？（新增页面/模型/API/依赖/Mode C 变更）→ 先评审后实施
□ 修改仅维护调用关系吗？涉及架构 → 暂停，请求用户审核
□ 前端变更涉及工作区吗？→ 检查 Mode C 合规（无业务按钮）
□ 前端页面变更 → 生成 HTML 预览并等待审核
□ 涉及哪些文件？列出清单
□ 涉及哪些文档？准备连锁更新
□ 测试策略是什么？先写复现测试
□ LLM 优先路径是否已设计？规则仅作校准
□ 变更记录格式准备好了吗？按模板填写
```

如果以上任一项无法回答，**暂停实施，向用户澄清后再继续**。
